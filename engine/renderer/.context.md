# engine/renderer -- System Context

## System Purpose

The renderer subsystem draws everything you see on screen. It provides a Render Hardware Interface (RHI) that abstracts OpenGL into simple create/bind/destroy calls for buffers, textures, and shaders. On top of the RHI, it provides a sprite batching system for efficient 2D rendering with automatic texture atlas packing (shelf packing into 2048x2048 pages to minimize draw calls), a camera for view/projection math, a render queue that sorts draw commands to minimize GPU state changes, a shader library with built-in shaders, and ECS components (`Transform`, `Sprite`, `SpriteAnimation`) plus systems (`animationUpdateSystem`, `renderPrepareSystem`) that automatically animate and draw your entities. If you want to put a sprite on screen, you create an entity with a `Transform` and a `Sprite` component, and the renderer handles the rest. The texture atlas is transparent -- game code does not interact with it directly. For atlas-based animation, add a `SpriteAnimation` component and the animation system will update the sprite's UV coordinates each frame.

### Texture Atlas (`renderer/texture_atlas.h`, namespace `ffe::renderer`)

Automatic sprite atlas packing. The atlas system reduces draw calls by packing multiple small textures into shared GPU textures (atlas pages). It uses a shelf-packing algorithm and is **transparent to game code** -- the sprite batch queries the atlas internally and remaps UVs automatically. Game developers do not need to call any atlas functions directly; the system is fully backward compatible with existing texture loading and sprite workflows.

**How it works:** When a texture is loaded, the sprite batch checks whether it fits in the atlas. If it does, it is packed into a shared atlas page and future draws of that texture use the atlas page texture instead of the individual texture. This means sprites that previously caused texture-change flushes (one draw call per texture) can now share the same atlas page and be drawn in a single batch.

**Constants:**

| Name | Value | Description |
|------|-------|-------------|
| `MAX_ATLAS_PAGES` | `4` | Maximum atlas page textures (up to 64 MB total VRAM at 2048x2048 RGBA8) |
| `MAX_ATLAS_ENTRIES` | `512` | Maximum individual textures tracked in the atlas lookup table |
| `MAX_ATLAS_SPRITE_DIM` | `512` | Maximum width or height (pixels) eligible for packing. Larger textures are not packed |
| `ATLAS_PADDING` | `1` | 1-pixel border around each packed sprite to prevent texture bleeding at sub-pixel sampling |

**AtlasRegion struct:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `u0` | `f32` | `0.0f` | Top-left U in atlas space [0, 1] |
| `v0` | `f32` | `0.0f` | Top-left V in atlas space [0, 1] |
| `u1` | `f32` | `1.0f` | Bottom-right U in atlas space [0, 1] |
| `v1` | `f32` | `1.0f` | Bottom-right V in atlas space [0, 1] |
| `page` | `i32` | `-1` | Atlas page index (-1 = not in atlas) |

**Functions:**

| Function | Signature | Description |
|----------|-----------|-------------|
| `initAtlas` | `void initAtlas(i32 maxSize = 2048)` | Initialize the atlas system. Creates the first atlas page texture. `maxSize` is the width/height of each page (power-of-two, max 4096, clamped to `GL_MAX_TEXTURE_SIZE`). Must be called after RHI init. **Cold path.** |
| `shutdownAtlas` | `void shutdownAtlas()` | Destroy all atlas page textures and free internal state |
| `addToAtlas` | `AtlasRegion addToAtlas(rhi::TextureHandle tex)` | Pack a texture into the atlas. Returns the region if successful, or a region with `page == -1` if already packed, too large, or atlas full. Performs `glTexSubImage2D`. **Cold path -- call at load time, not per-frame.** |
| `getAtlasRegion` | `AtlasRegion getAtlasRegion(rhi::TextureHandle tex)` | Look up the atlas region for a previously packed texture. Returns `page == -1` if not in the atlas |
| `getAtlasPageTexture` | `rhi::TextureHandle getAtlasPageTexture(i32 page)` | Get the RHI texture handle for an atlas page. Invalid handle if page is out of range or not created |
| `isInAtlas` | `bool isInAtlas(rhi::TextureHandle tex)` | Check whether a texture has been packed into the atlas |
| `getAtlasUtilization` | `f32 getAtlasUtilization()` | Returns the fraction of atlas space used across all pages [0.0, 1.0]. Debugging/profiling only |
| `getAtlasPageSize` | `i32 getAtlasPageSize()` | Returns the atlas page size in pixels (width == height). 0 if not initialized |
| `getAtlasPageCount` | `u32 getAtlasPageCount()` | Returns the number of atlas pages currently allocated |
| `remapUVs` | `inline void remapUVs(const AtlasRegion& region, f32 origUMin, f32 origVMin, f32 origUMax, f32 origVMax, f32& outUMin, f32& outVMin, f32& outUMax, f32& outVMax)` | Remap original texture UVs to atlas-space UVs. **Hot path -- called per-sprite per-frame.** Inline, no branches, no allocations |

**Key design points:**

- **Default page size:** 2048x2048 RGBA8 (16 MB per page). This fits comfortably in LEGACY tier's 1 GB VRAM budget.
- **Shelf packing:** Textures are placed left-to-right on shelves (rows). When a shelf is full, a new shelf is started below. When a page is full, a new page is allocated (up to `MAX_ATLAS_PAGES`).
- **Backward compatible:** Textures that are too large for the atlas (either dimension > 512px) or that cannot fit (atlas full) fall back to non-atlas rendering with zero behavior change. Existing code that loads textures and assigns them to sprites works identically whether the atlas is active or not.
- **Transparent to users:** Game developers do not call `addToAtlas` or `remapUVs` directly. The sprite batch handles atlas lookup and UV remapping internally. The only user-facing function is `getAtlasUtilization()`, exposed via Lua as `ffe.getAtlasUtilization()` for profiling.

## Common Usage Patterns

### 1. Draw a colored rectangle (no texture)

```cpp
ffe::EntityId box = world.createEntity();
world.addComponent<ffe::Transform>(box, glm::vec3{100.0f, 50.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(box);
sprite.size = {120.0f, 80.0f};           // 120x80 pixels
sprite.color = {0.2f, 0.6f, 1.0f, 1.0f}; // Light blue

// That's it. renderPrepareSystem picks it up automatically.
// No texture needed -- the engine uses a 1x1 white pixel as default.
```

### 2. Draw a textured sprite

```cpp
// Load a texture
ffe::rhi::TextureDesc texDesc;
texDesc.width = imgWidth;
texDesc.height = imgHeight;
texDesc.format = ffe::rhi::TextureFormat::RGBA8;
texDesc.filter = ffe::rhi::TextureFilter::NEAREST; // Pixel art? Use NEAREST
texDesc.pixelData = myPixelData; // Pointer to RGBA8 pixel data

ffe::rhi::TextureHandle myTex = ffe::rhi::createTexture(texDesc);

// Create entity
ffe::EntityId hero = world.createEntity();
world.addComponent<ffe::Transform>(hero, glm::vec3{0.0f, 0.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(hero);
sprite.texture = myTex;
sprite.size = {32.0f, 32.0f};
```

### 3. Use a sprite sheet (UV sub-regions)

```cpp
// Suppose your sprite sheet is 256x256 and each frame is 32x32.
// To show the frame at row 1, column 2 (zero-indexed):
auto& sprite = world.getComponent<ffe::Sprite>(entity);
sprite.uvMin = {2.0f * 32.0f / 256.0f, 1.0f * 32.0f / 256.0f}; // {0.25, 0.125}
sprite.uvMax = {3.0f * 32.0f / 256.0f, 2.0f * 32.0f / 256.0f}; // {0.375, 0.25}
```

### 4. Layer-based draw ordering

```cpp
// Background layer (draws first)
auto& bgSprite = world.addComponent<ffe::Sprite>(background);
bgSprite.layer = 0;

// Player (draws on top of background)
auto& playerSprite = world.addComponent<ffe::Sprite>(player);
playerSprite.layer = 5;

// UI (draws on top of everything)
auto& uiSprite = world.addComponent<ffe::Sprite>(healthBar);
uiSprite.layer = 10;

// Within the same layer, use sortOrder for fine control:
auto& bullet1 = world.addComponent<ffe::Sprite>(b1);
bullet1.layer = 5;
bullet1.sortOrder = 1; // Draws after player (sortOrder 0)
```

### 5. Animate a sprite with a spritesheet (C++)

```cpp
// Suppose your spritesheet is 128x64 with 8 frames (4 columns x 2 rows), each 32x32.
ffe::EntityId character = world.createEntity();
world.addComponent<ffe::Transform>(character, glm::vec3{200.0f, 100.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(character);
sprite.texture = mySpritesheetTex;
sprite.size = {32.0f, 32.0f};
// Do NOT set uvMin/uvMax manually — animationUpdateSystem overwrites them each frame.

auto& anim = world.addComponent<ffe::SpriteAnimation>(character);
anim.frameCount = 8;       // 8 frames total
anim.columns    = 4;       // 4 columns in the atlas grid
anim.frameTime  = 0.1f;    // 100ms per frame
anim.looping    = true;
anim.playing    = true;     // start immediately

// That's it. animationUpdateSystem (priority 50) will advance the frame
// and update sprite.uvMin/uvMax each tick before renderPrepareSystem runs.
```

### 6. Creating a custom shader (advanced)

```cpp
ffe::rhi::ShaderDesc desc;
desc.vertexSource = myVertGLSL;
desc.fragmentSource = myFragGLSL;
desc.debugName = "custom_glow";

ffe::rhi::ShaderHandle glowShader = ffe::rhi::createShader(desc);
if (!ffe::rhi::isValid(glowShader)) {
    FFE_LOG_ERROR("Game", "Glow shader failed to compile!");
}

// Set uniforms after binding
ffe::rhi::bindShader(glowShader);
ffe::rhi::setUniformFloat(glowShader, "u_glowIntensity", 2.5f);
ffe::rhi::setUniformVec4(glowShader, "u_glowColor", {1.0f, 0.8f, 0.2f, 1.0f});

// Clean up when done
ffe::rhi::destroyShader(glowShader);
```

### 7. Create a particle emitter (C++)

```cpp
// Create an explosion emitter on an existing entity with a Transform.
auto& em = world.addComponent<ffe::ParticleEmitter>(entity);
em.emitRate     = 0.0f;     // Not using continuous emission
em.lifetimeMin  = 0.3f;
em.lifetimeMax  = 0.8f;
em.speedMin     = 50.0f;
em.speedMax     = 200.0f;
em.sizeStart    = 6.0f;
em.sizeEnd      = 1.0f;
em.gravityY     = -50.0f;
em.colorStart   = {1.0f, 0.8f, 0.2f, 1.0f};
em.colorEnd     = {1.0f, 0.1f, 0.0f, 0.0f};
em.layer        = 5;
em.sortOrder    = 10;        // Draw above other layer-5 sprites

// Fire a one-shot burst of 50 particles.
em.burstCount = 50;

// particleUpdateSystem (priority 55) handles the rest.
```

### 8. Burst particles then remove emitter (C++)

```cpp
// Spawn a burst and let particles die naturally.
auto& em = world.addComponent<ffe::ParticleEmitter>(entity);
em.burstCount  = 30;
em.lifetimeMin = 0.2f;
em.lifetimeMax = 0.5f;
em.speedMin    = 40.0f;
em.speedMax    = 120.0f;

// After the burst, burstCount resets to 0 and emitting stays false.
// Particles live out their lifetime. You can safely removeComponent
// once activeCount reaches 0.
```

### 9. Change emitter config at runtime (C++)

```cpp
// Switch an existing emitter from fire to smoke mid-game.
auto& em = world.getComponent<ffe::ParticleEmitter>(entity);
em.colorStart = {0.5f, 0.5f, 0.5f, 0.8f};
em.colorEnd   = {0.3f, 0.3f, 0.3f, 0.0f};
em.sizeStart  = 8.0f;
em.sizeEnd    = 16.0f;
em.speedMin   = 10.0f;
em.speedMax   = 30.0f;
em.gravityY   = 20.0f;  // Smoke rises

// Already-alive particles keep their birth config.
// Only newly emitted particles use the updated values.
```

## What Not To Do

### Do not forget to destroy GPU resources

```cpp
// BAD -- GPU memory leak
ffe::rhi::TextureHandle tex = ffe::rhi::createTexture(desc);
// ... game ends, tex is never destroyed

// GOOD
ffe::rhi::destroyTexture(tex);
```

Every `createBuffer`, `createTexture`, `createShader` must have a matching `destroy` call.

### Do not set uniforms without binding the shader first

```cpp
// BAD -- undefined behaviour, sets uniform on wrong (or no) shader
ffe::rhi::setUniformFloat(myShader, "u_time", 1.0f);

// GOOD
ffe::rhi::bindShader(myShader);
ffe::rhi::setUniformFloat(myShader, "u_time", 1.0f);
```

### Do not create textures or buffers every frame

```cpp
// BAD -- creates a new GPU buffer every frame, leaks memory
void mySystem(ffe::World& world, float dt) {
    auto buf = ffe::rhi::createBuffer(desc); // WRONG -- this is a hot path
}

// GOOD -- create once at startup, update with updateBuffer if needed
```

### Do not ignore invalid handles

```cpp
// BAD -- using a handle that failed to create
ffe::rhi::ShaderHandle shader = ffe::rhi::createShader(desc);
ffe::rhi::bindShader(shader); // Might be id=0 if compile failed!

// GOOD
ffe::rhi::ShaderHandle shader = ffe::rhi::createShader(desc);
if (!ffe::rhi::isValid(shader)) {
    FFE_LOG_ERROR("Game", "Shader compile failed");
    return;
}
```

### Do not manually set uvMin/uvMax on an animated sprite

If an entity has a `SpriteAnimation` component and the animation is playing, `animationUpdateSystem` overwrites `uvMin`/`uvMax` every tick. Any manual UV changes you make will be immediately overwritten. To show a specific frame, use `setAnimationFrame` from Lua or set `currentFrame` directly in C++ — the system will compute the correct UVs.

```cpp
// BAD — these UVs will be overwritten next tick by animationUpdateSystem
auto& sprite = world.getComponent<ffe::Sprite>(entity);
sprite.uvMin = {0.25f, 0.0f};  // overwritten next frame

// GOOD — set the frame index and let the system compute UVs
auto& anim = world.getComponent<ffe::SpriteAnimation>(entity);
anim.currentFrame = 3;
```

### Do not manually populate the RenderQueue for sprites

The `renderPrepareSystem` does this automatically for any entity with `Transform` + `Sprite`. You only need to touch the RenderQueue directly if you are building custom rendering beyond the sprite system.

### Do not set Sprite.size to zero

A sprite with `size = {0, 0}` produces degenerate geometry. Always set a non-zero size.

### Do not exceed 15 render layers

The sort key encodes layer in 4 bits (0-15). Layer values outside this range are clamped.

### Do not create a new emitter every frame

```cpp
// BAD -- creates a new ParticleEmitter component every frame
void mySystem(ffe::World& world, float dt) {
    world.addComponent<ffe::ParticleEmitter>(entity); // WRONG -- hot path allocation
}

// GOOD -- add the emitter once, then use emitBurst or toggle emitting
auto& em = world.getComponent<ffe::ParticleEmitter>(entity);
em.burstCount = 10; // Trigger a burst without re-creating the component
```

### Do not set lifetimeMin or lifetimeMax to zero

A zero lifetime causes division-by-zero in the alpha/size interpolation. The Lua bindings clamp lifetime to a minimum of `0.001f`. In C++, ensure you never set lifetime below that value.

### Do not expect particles to collide

Particles are purely visual. They do not interact with the collision/physics system. If you need physical debris, create normal entities with `Collider2D` components instead.

### Do not exceed MAX_PARTICLES (128) per emitter

Each emitter has a fixed inline pool of 128 particles. When the pool is full, new emissions are silently dropped. If you need more particles, use multiple emitters on separate entities. Do not increase `MAX_PARTICLES` without understanding the memory cost (each `ParticleEmitter` is already a large component).

## 3D Mesh Rendering

### System Purpose

The 3D mesh rendering subsystem loads binary glTF 2.0 (`.glb`) meshes from disk, uploads them to the GPU as indexed geometry (VAO + VBO + IBO), and renders them with a Blinn-Phong directional lighting shader before the 2D sprite pass each frame. 3D entities coexist with 2D entities in the same ECS world: the 3D pass runs first with depth testing enabled, then depth testing is disabled and the 2D sprite batch renders on top. A game that uses no 3D components is entirely unaffected — `meshRenderSystem` iterates zero entities and returns immediately.

---

### Shadow Mapping (`renderer/shadow_map.h`, namespace `ffe`)

The shadow mapping subsystem adds directional-light shadows to the 3D Blinn-Phong pass. It renders a depth-only pass from the light's point of view into a square FBO, then samples the resulting depth texture with a 3×3 PCF (percentage-closer filtering) kernel during the main mesh pass to produce soft shadow edges. Shadows are disabled by default — no GPU resources are allocated and no extra draw calls are issued until `enableShadows` is called.

**Tier support:** LEGACY (OpenGL 3.3 core). FBOs and depth textures are core in OpenGL 3.3. Headless mode is safe: `createShadowMap` returns a zero-initialized `ShadowMap` and all other functions are no-ops.

#### C++ API

**ShadowConfig struct** — per-scene configuration, stored as ECS registry context. All fields have sensible defaults.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `resolution` | `int` | `1024` | Shadow map width and height in texels |
| `bias` | `float` | `0.005` | Depth bias to reduce shadow acne |
| `areaWidth` | `float` | `20.0` | Orthographic frustum width (world units) |
| `areaHeight` | `float` | `20.0` | Orthographic frustum height (world units) |
| `nearPlane` | `float` | `0.1` | Light frustum near clip |
| `farPlane` | `float` | `50.0` | Light frustum far clip |
| `enabled` | `bool` | `false` | Master switch — no overhead when false |

**ShadowMap struct** — GPU-side resources. Zero-initialized means no GL resources allocated.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `fbo` | `GLuint` | `0` | Framebuffer object |
| `depthTexture` | `GLuint` | `0` | `GL_DEPTH_COMPONENT24` 2D texture |
| `resolution` | `int` | `0` | Width == height in texels |

**Free functions:**

| Function | Signature | Description |
|----------|-----------|-------------|
| `createShadowMap` | `ShadowMap createShadowMap(int resolution)` | Create a square depth-only FBO. Returns zero-initialized `ShadowMap` in headless mode or on GL error. |
| `destroyShadowMap` | `void destroyShadowMap(ShadowMap& sm)` | Free GPU resources and zero the struct. Safe on zero-initialized input (no-op). |
| `computeLightSpaceMatrix` | `glm::mat4 computeLightSpaceMatrix(const glm::vec3& lightDir, const ShadowConfig& cfg)` | Compute ortho light-space matrix for the shadow pass. `lightDir` is normalized internally. |

#### Lua Bindings

| Function | Signature | Returns | Description |
|----------|-----------|---------|-------------|
| `ffe.enableShadows` | `ffe.enableShadows(size?)` | `boolean` | Enable shadow mapping with the given resolution. `size` must be a power of 2 in `[256, 4096]`; defaults to `1024`. Returns `true` on success, `false` on failure. If already enabled at the same resolution, returns `true` (no-op). If resolution changed, the old shadow map is destroyed and a new one created. Cold path — call at scene init. |
| `ffe.disableShadows` | `ffe.disableShadows()` | nothing | Disable shadow mapping and free the GPU depth texture/FBO. Safe to call when already disabled (no-op). |
| `ffe.setShadowBias` | `ffe.setShadowBias(bias?)` | nothing | Set the depth bias to reduce shadow acne. Clamped to `[0.0, 0.1]`. Defaults to `0.005`. Non-finite values are rejected (warning logged). |
| `ffe.setShadowArea` | `ffe.setShadowArea(width?, height?, near?, far?)` | nothing | Configure the orthographic frustum for the light camera. All values must be positive and finite, with `near < far`. Defaults: `20, 20, 0.1, 50`. Invalid inputs are rejected (warning logged). |

#### Shadow Usage Patterns

**1. Enable shadows with default settings**

```lua
function init()
    local mesh = ffe.loadMesh("models/scene.glb")
    ffe.createEntity3D(mesh, 0, 0, 0)

    -- Enable shadows (1024x1024 by default)
    local ok = ffe.enableShadows()
    if not ok then
        -- GPU allocation failed; game still works, just without shadows
    end

    ffe.setLightDirection(1, -2, 1)
    ffe.set3DCamera(5, 5, 5,  0, 0, 0)
end
```

**2. High-resolution shadows for a close-up scene**

```lua
function init()
    -- 2048x2048 shadow map, tighter frustum for better texel density
    ffe.enableShadows(2048)
    ffe.setShadowArea(10, 10, 0.1, 30)    -- smaller area = sharper shadows
    ffe.setShadowBias(0.003)               -- tune bias for the scene
end
```

**3. Toggle shadows at runtime (settings menu)**

```lua
function onShadowToggle(enabled)
    if enabled then
        ffe.enableShadows(1024)
    else
        ffe.disableShadows()   -- frees GPU resources immediately
    end
end
```

**4. Adjust shadow area to match scene scale**

```lua
-- Large outdoor scene: wide frustum
ffe.setShadowArea(40, 40, 0.5, 100)

-- Small indoor room: narrow frustum for sharper shadows
ffe.setShadowArea(8, 8, 0.1, 20)
```

#### Shadow Anti-Patterns

**Do not call `ffe.enableShadows()` every frame.** It checks for no-op when already enabled at the same resolution, but calling it per-frame is wasteful and confusing. Call it once at scene init.

```lua
-- WRONG — calling enableShadows in update
function update(dt)
    ffe.enableShadows(1024)  -- pointless per-frame call
end

-- RIGHT — call once in init
function init()
    ffe.enableShadows(1024)
end
```

**Do not use a shadow area much larger than your visible scene.** A 100×100 world-unit frustum with a 1024×1024 shadow map gives ~10 texels per world unit — shadows will be blocky. Either increase resolution or reduce the area to match what the camera actually sees.

**Do not set bias to 0.** A zero bias will cause severe shadow acne (self-shadowing artifacts on lit surfaces). The default `0.005` works for most scenes. Increase slightly if acne appears; decrease if shadows detach from geometry ("Peter Panning").

**Do not forget to set light direction.** Shadows are cast from the directional light. If `setLightDirection` is never called, the default light direction is used, which may not match your scene. Always set light direction explicitly when using shadows.

**Do not pass non-power-of-2 sizes to `enableShadows`.** The function validates this and returns `false`, but your game will have no shadows. Valid sizes: 256, 512, 1024, 2048, 4096.

### Skybox Rendering (`renderer/mesh_renderer.h`, namespace `ffe::renderer`)

The skybox subsystem renders a cubemap environment behind all 3D geometry. It draws a unit cube with `GL_LEQUAL` depth (fragments at max depth) so the skybox appears behind every mesh. The cubemap texture is loaded from 6 face images via `loadCubemap`. Skybox rendering is disabled by default -- no GPU resources are allocated until `loadSkybox` is called from Lua.

**Tier support:** LEGACY (OpenGL 3.3 core). `GL_TEXTURE_CUBE_MAP` is core in 3.3.

#### C++ API

**SkyboxConfig struct** -- per-scene configuration, stored as ECS registry context.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `cubemapTexture` | `u32` | `0` | GL cubemap texture ID (0 = none loaded) |
| `enabled` | `bool` | `false` | Master switch -- no overhead when false |

**Free functions:**

| Function | Signature | Description |
|----------|-----------|-------------|
| `renderSkybox` | `void renderSkybox(World& world, const Camera& camera3d, const SkyboxConfig& skyboxCfg)` | Render the skybox cubemap. Must be called AFTER `meshRenderSystem` (so 3D geometry writes depth first) and BEFORE the 2D sprite pass. No-op if `enabled` is false or `cubemapTexture` is 0. Uses a lazy-initialized unit cube VAO (static lifetime, no per-frame allocation). |
| `loadCubemap` | `u32 loadCubemap(const std::array<std::string, 6>& facePaths)` | Load 6 face images into a `GL_TEXTURE_CUBE_MAP`. Face order: `+X, -X, +Y, -Y, +Z, -Z` (right, left, top, bottom, front, back). Returns a GL texture ID on success, 0 on failure. Caller must delete via `glDeleteTextures()`. |

#### Lua Bindings

| Function | Signature | Returns | Description |
|----------|-----------|---------|-------------|
| `ffe.loadSkybox` | `ffe.loadSkybox(right, left, top, bottom, front, back)` | `boolean` | Load 6 face images into a cubemap texture and enable skybox rendering. Face paths are relative to the asset root (same rules as `ffe.loadTexture`). Parameter order follows the standard OpenGL cubemap convention: `+X, -X, +Y, -Y, +Z, -Z`. Returns `true` on success, `false` on failure. If a skybox is already loaded, the previous cubemap is destroyed first. Cold path -- call at scene init. |
| `ffe.unloadSkybox` | `ffe.unloadSkybox()` | nothing | Delete the cubemap texture and disable skybox rendering. Frees GPU memory immediately. Safe to call when no skybox is loaded (no-op). |
| `ffe.setSkyboxEnabled` | `ffe.setSkyboxEnabled(enabled)` | nothing | Toggle skybox rendering without unloading the cubemap texture. `enabled` is a boolean. Useful for letting the player toggle the skybox on/off without reloading textures. No-op if no `SkyboxConfig` is in the ECS context. |

#### Skybox Usage Patterns

**1. Load a skybox at scene init**

```lua
function init()
    local mesh = ffe.loadMesh("models/terrain.glb")
    ffe.createEntity3D(mesh, 0, 0, 0)

    -- Load 6 face images (right, left, top, bottom, front, back)
    local ok = ffe.loadSkybox(
        "skybox/right.png",
        "skybox/left.png",
        "skybox/top.png",
        "skybox/bottom.png",
        "skybox/front.png",
        "skybox/back.png"
    )
    if not ok then
        ffe.log("Skybox load failed -- game continues without skybox")
    end

    ffe.set3DCameraOrbit(0, 0, 0, 10, 0, 20)
end
```

**2. Toggle skybox at runtime (settings menu)**

```lua
function onSkyboxToggle(enabled)
    ffe.setSkyboxEnabled(enabled)
    -- Cubemap stays in GPU memory; toggling is instant
end
```

**3. Switch skybox for a new scene**

```lua
function loadNewScene()
    ffe.unloadSkybox()  -- free the old cubemap

    ffe.loadSkybox(
        "skybox2/right.png", "skybox2/left.png",
        "skybox2/top.png",   "skybox2/bottom.png",
        "skybox2/front.png", "skybox2/back.png"
    )
end
```

**4. Clean up skybox in shutdown**

```lua
function shutdown()
    ffe.unloadSkybox()
    -- other cleanup ...
end
```

#### Skybox Anti-Patterns

**Do not call `ffe.loadSkybox()` every frame.** It loads 6 images from disk and uploads a cubemap to the GPU. Call it once at scene init.

```lua
-- WRONG -- loading 6 images from disk every frame
function update(entityId, dt)
    ffe.loadSkybox("skybox/right.png", "skybox/left.png", ...)
end

-- RIGHT -- load once in init
function init()
    ffe.loadSkybox("skybox/right.png", "skybox/left.png", ...)
end
```

**Do not pass the face images in the wrong order.** The parameter order is `right (+X), left (-X), top (+Y), bottom (-Y), front (+Z), back (-Z)`. Swapping faces produces a visually broken skybox with seams at the edges.

**Do not forget to call `ffe.unloadSkybox()` in shutdown.** The cubemap texture leaks GPU memory if not explicitly freed. Unlike entity components, the skybox cubemap is a scene-global GPU resource.

**Do not use `ffe.setSkyboxEnabled(true)` before loading a skybox.** If no cubemap is loaded (`cubemapTexture == 0`), `renderSkybox` is a no-op regardless of the enabled flag. Load the skybox first.

### Linear Fog (`renderer/mesh_renderer.h`, namespace `ffe::renderer`)

The fog subsystem applies distance-based linear fog to all 3D mesh fragments in the Blinn-Phong pass. Fog blends fragment colors towards a configurable fog color based on the distance from the camera. The fog formula is: `fogFactor = clamp((fogFar - dist) / (fogFar - fogNear), 0, 1)`, then `finalColor = mix(fogColor, fragColor, fogFactor)`. Fragments closer than `fogNear` are unaffected; fragments beyond `fogFar` are fully fog-colored.

Fog is NOT applied to the skybox -- the skybox always renders at full color regardless of fog settings.

**Tier support:** LEGACY (OpenGL 3.3 core). No additional textures or FBOs required. The fog computation is a single `length()` + `clamp()` + `mix()` per fragment, guarded by a uniform boolean branch (zero cost when disabled).

#### C++ API

**FogParams struct** -- per-scene fog configuration, stored as ECS registry context pointer.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `r` | `f32` | `0.7` | Fog color red component [0, 1] |
| `g` | `f32` | `0.7` | Fog color green component [0, 1] |
| `b` | `f32` | `0.8` | Fog color blue component [0, 1] |
| `nearDist` | `f32` | `10.0` | Distance from camera where fog starts (world units) |
| `farDist` | `f32` | `100.0` | Distance from camera where fog is fully opaque (world units) |
| `enabled` | `bool` | `false` | Master switch -- no overhead when false |

**meshRenderSystem** accepts a `const FogParams&` parameter (defaults to disabled `FogParams{}`). Fog uniforms are uploaded once per frame to both the static and skinned Blinn-Phong shaders.

#### Lua Bindings

| Function | Signature | Returns | Description |
|----------|-----------|---------|-------------|
| `ffe.setFog` | `ffe.setFog(r, g, b, near, far)` | nothing | Enable linear fog with the given color and distance range. Color components are clamped to `[0, 1]`. `near` and `far` must be positive and finite, with `near < far`. Defaults: `0.7, 0.7, 0.8, 10.0, 100.0`. Non-finite or invalid values are rejected (warning logged). |
| `ffe.disableFog` | `ffe.disableFog()` | nothing | Disable fog rendering. Fog params are preserved -- calling `setFog` again re-enables with the new values. |

#### Fog Usage Patterns

**1. Enable fog at scene init**

```lua
function init()
    -- Gray-blue fog starting at 20 units, fully opaque at 80 units
    ffe.setFog(0.7, 0.7, 0.8, 20, 80)
end
```

**2. Dense fog for atmosphere**

```lua
function init()
    -- White fog, very close range for a foggy forest scene
    ffe.setFog(0.9, 0.9, 0.95, 5, 30)
end
```

**3. Toggle fog at runtime**

```lua
function onFogToggle(enabled)
    if enabled then
        ffe.setFog(0.7, 0.7, 0.8, 20, 80)
    else
        ffe.disableFog()
    end
end
```

**4. Match fog color to skybox/background**

```lua
function init()
    -- Set fog color to match the background clear color
    ffe.setBackgroundColor(0.6, 0.7, 0.8)
    ffe.setFog(0.6, 0.7, 0.8, 15, 60)
end
```

#### Fog Anti-Patterns

**Do not set near >= far.** The Lua binding rejects this with a warning. The fog factor formula divides by `(far - near)`, so equal values would cause a division by zero in the shader (though the shader clamp prevents visual artifacts).

**Do not use fog as a substitute for draw distance culling.** Fog hides distant geometry visually, but the GPU still processes every vertex and fragment. If you need to reduce draw calls for distant objects, implement a draw distance check in your game logic.

**Do not forget to match fog color to the background.** If the fog color does not match the clear color / skybox, objects will fade to a visibly different color at the horizon, which looks wrong. Match `ffe.setFog` color to `ffe.setBackgroundColor` or the dominant skybox color.

### Skeletal Animation (`renderer/mesh_loader.h`, `renderer/render_system.h`, namespace `ffe` / `ffe::renderer`)

The skeletal animation subsystem drives bone-based mesh deformation for 3D entities. Animation data (clips, bones, keyframes) is loaded from `.glb` files alongside the mesh geometry. At runtime, `animationUpdateSystem3D` advances the animation timer, samples keyframes, computes per-bone matrices, and uploads them to the shader as uniforms. The shader applies bone transforms to vertices for smooth skeletal deformation.

**Tier support:** LEGACY (OpenGL 3.3 core). Bone matrices are uploaded as uniform arrays. CPU-side keyframe sampling and bone matrix computation.

**Constants** (`renderer/skeleton.h`):

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_BONES` | `64` | Maximum bones per skeleton. Fits within GL 3.3 uniform limits (1024 vertex uniform components = 64 mat4) |
| `MAX_BONE_INFLUENCES` | `4` | Maximum bone influences per vertex (standard glTF skinning) |
| `MAX_ANIMATIONS_PER_MESH` | `16` | Maximum animation clips per mesh asset |
| `MAX_KEYFRAMES_PER_CHANNEL` | `256` | Maximum keyframes per bone per animation channel (translation, rotation, or scale) |

#### C++ Components

**Skeleton struct** -- per-entity bone state. Added automatically by `playAnimation3D` if not already present.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `boneMatrices` | `glm::mat4[MAX_BONES]` | identity | Final bone transform matrices sent to the shader |
| `boneCount` | `u32` | `0` | Number of active bones for this skeleton |

**AnimationState struct** -- per-entity animation playback state (32 bytes). Added automatically by `playAnimation3D`.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `clipIndex` | `u32` | `0` | Index of the currently playing (target) animation clip (0-based) |
| `time` | `f32` | `0.0` | Current playback time in seconds (target clip) |
| `speed` | `f32` | `1.0` | Playback speed multiplier. 0 = paused. Negative values clamped to 0 |

| `playing` | `bool` | `false` | Whether animation is currently advancing |

| `blendDuration` | `f32` | `0.0` | Total crossfade duration in seconds |
| `blendElapsed` | `f32` | `0.0` | Time elapsed in the current crossfade blend |

**RootMotionDelta struct** -- per-entity root bone motion extraction (28 bytes). Added via `ffe.setRootMotion3D`.

When enabled, the animation system extracts the root bone (index 0) XZ translation each frame and stores it as a delta. The root bone's XZ translation is zeroed in the skeleton so the animation plays in-place. Gameplay code reads the delta via `ffe.getRootMotionDelta3D` and applies it to the entity's `Transform3D`.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `translationDelta` | `glm::vec3` | `{0,0,0}` | Root bone translation delta since last frame |
| `previousRootTranslation` | `glm::vec3` | `{0,0,0}` | Previous frame's root bone position (internal tracking) |
| `enabled` | `bool` | `false` | Whether root motion extraction is active |

#### Skeletal Animation Usage Patterns

**1. Play a looping walk animation**

```lua
local mesh = ffe.loadMesh("models/character.glb")
local entity = ffe.createEntity3D(mesh, 0, 0, 0)

-- Check how many animations the mesh has
local count = ffe.getAnimationCount3D(entity)
ffe.log("Mesh has " .. count .. " animations")

-- Play animation 0 (looping by default)
ffe.playAnimation3D(entity, 0, true)
```

**2. Play a one-shot death animation**

```lua
-- Play animation 2, do NOT loop
ffe.playAnimation3D(entity, 2, false)

-- Check each frame if the animation finished
function update(entityId, dt)
    if not ffe.isAnimation3DPlaying(entity) then
        ffe.destroyEntity(entity)
    end
end
```

**3. Adjust playback speed**

```lua
-- Slow-motion: half speed
ffe.setAnimationSpeed3D(entity, 0.5)

-- Fast-forward: double speed
ffe.setAnimationSpeed3D(entity, 2.0)

-- Pause (freeze current pose)
ffe.setAnimationSpeed3D(entity, 0)
```

**4. Monitor animation progress**

```lua
function update(entityId, dt)
    local progress = ffe.getAnimationProgress3D(entity)
    ffe.drawText("Anim: " .. string.format("%.0f%%", progress * 100), 10, 10)
end
```

**5. Switch between animations (hard cut)**

```lua
-- Switch from walk (clip 0) to run (clip 1) -- instant, no blending
ffe.playAnimation3D(entity, 1, true)
-- playAnimation3D resets time to 0 and starts the new clip immediately
```

**6. Crossfade between animations (smooth blend)**

```lua
-- Start walking
ffe.playAnimation3D(entity, 0, true)

-- Later, blend smoothly to run (clip 1) over 0.3 seconds
ffe.crossfadeAnimation3D(entity, 1, 0.3)
-- Bone matrices are interpolated between walk and run during the blend.
-- After 0.3s, blending completes automatically and only clip 1 plays.
```

**7. Root motion -- apply animation movement to entity position**

```lua
-- Enable root motion extraction
ffe.setRootMotion3D(entity, true)

function update(entityId, dt)
    -- Each frame, read the root bone's XZ delta
    local dx, dy, dz = ffe.getRootMotionDelta3D(entity)
    -- Apply delta to the entity's position
    local x, y, z = ffe.getPosition3D(entity)
    ffe.setPosition3D(entity, x + dx, y + dy, z + dz)
end

-- Disable when switching to physics-driven movement
ffe.setRootMotion3D(entity, false)
```

#### Skeletal Animation Anti-Patterns

**Do not call `ffe.playAnimation3D()` every frame to "keep" an animation playing.** It resets playback time to 0 on each call, so the animation will never advance past the first frame. Call it once to start playback.

```lua
-- WRONG -- resets to frame 0 every frame, animation never advances
function update(entityId, dt)
    ffe.playAnimation3D(entity, 0, true)  -- BAD: called every frame
end

-- RIGHT -- call once, then let the system advance automatically
ffe.playAnimation3D(entity, 0, true)  -- called once at init

function update(entityId, dt)
    -- animation advances automatically via animationUpdateSystem3D
end
```

**Do not use `ffe.playAnimation3D` on a mesh without skeleton data.** Not all `.glb` files contain bones and animations. If the mesh has no skeleton, `playAnimation3D` logs a warning and does nothing. Use `ffe.getAnimationCount3D` to check first.

```lua
-- DEFENSIVE -- check before playing
if ffe.getAnimationCount3D(entity) > 0 then
    ffe.playAnimation3D(entity, 0, true)
else
    ffe.log("Mesh has no animations")
end
```

**Do not pass an out-of-range `animIndex`.** The function validates the index against the mesh's clip count and logs an error if out of range. Use `ffe.getAnimationCount3D` to determine valid indices.

**Do not call `ffe.crossfadeAnimation3D` before `ffe.playAnimation3D`.** Crossfade requires an existing `AnimationState` component. `playAnimation3D` creates it; `crossfadeAnimation3D` does not. If you call crossfade on an entity with no `AnimationState`, it logs a warning and does nothing.

```lua
-- WRONG -- no AnimationState exists yet
ffe.crossfadeAnimation3D(entity, 1, 0.3)  -- BAD: logs warning, no effect

-- RIGHT -- start an animation first, then crossfade later
ffe.playAnimation3D(entity, 0, true)
-- ... later ...
ffe.crossfadeAnimation3D(entity, 1, 0.3)
```

**Do not call `ffe.crossfadeAnimation3D` every frame.** Each call restarts the blend from the current state. Call it once when you want to transition; the animation system completes the blend automatically.

**Do not forget to read and apply root motion deltas.** Enabling root motion with `ffe.setRootMotion3D(entity, true)` zeroes the root bone's XZ translation in the skeleton. If you do not read the delta with `ffe.getRootMotionDelta3D` and apply it to the entity's `Transform3D`, the character will animate in-place but never move in the world.

### Post-Processing (`renderer/post_process.h`, namespace `ffe::renderer`)

HDR scene capture, bloom (half-resolution ping-pong Gaussian blur), tone mapping (Reinhard / ACES filmic), gamma correction, and anti-aliasing (MSAA / FXAA). The pipeline renders the scene into a `GL_RGBA16F` FBO, then runs configurable post-process passes before blitting to the default framebuffer. All GPU resources (FBOs, textures, fullscreen quad VAO) are created at init/resize only. No per-frame heap allocation. No virtual functions.

**PostProcessConfig struct (ECS singleton):**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `bloomEnabled` | `bool` | `false` | Master bloom switch |
| `bloomThreshold` | `f32` | `1.0` | Luminance cutoff for bright-pass extraction |
| `bloomIntensity` | `f32` | `0.5` | Additive bloom blend strength |
| `toneMapper` | `i32` | `0` | 0 = none, 1 = Reinhard, 2 = ACES filmic |
| `gammaEnabled` | `bool` | `true` | Apply `pow(color, 1/gamma)` in the final pass |
| `gamma` | `f32` | `2.2` | Gamma exponent |
| `aaMode` | `i32` | `0` | Anti-aliasing mode: 0 = none, 1 = MSAA, 2 = FXAA |
| `msaaSamples` | `i32` | `2` | MSAA sample count (2, 4, or 8 -- clamped to nearest valid) |

The renderer activates the post-processing pipeline when a `PostProcessConfig` exists in the ECS registry context. Removing the config disables the pipeline entirely (the scene is blitted directly to the default framebuffer).

**Anti-aliasing pass order:**
1. MSAA resolve (if `aaMode == 1`): blit multisample FBO to regular HDR FBO
2. Bloom: extract bright pixels, Gaussian blur, composite
3. Tone mapping: Reinhard or ACES filmic HDR-to-LDR conversion
4. FXAA (if `aaMode == 2`): edge-detection anti-aliasing on the tone-mapped image
5. Gamma correction: linear-to-sRGB conversion

MSAA and FXAA are mutually exclusive (use one or the other). MSAA anti-aliases geometry edges; FXAA anti-aliases everything including shader aliasing. FXAA is cheaper (~0.3ms at 720p) but slightly blurs the image.

**C++ Pipeline Functions:**

| Function | Signature | Description |
|----------|-----------|-------------|
| `initPostProcessing` | `bool initPostProcessing(i32 width, i32 height)` | Create HDR scene FBO, bloom ping-pong FBOs (half-res), and fullscreen quad VAO. Call once after the GL context is available. Returns `false` on failure. |
| `resizePostProcessing` | `void resizePostProcessing(i32 width, i32 height)` | Recreate FBOs at a new resolution. Called from the framebuffer resize callback. Safe to call before `initPostProcessing` (no-op). |

| `executePostProcessing` | `void executePostProcessing(const PostProcessConfig& config)` | Run the post-process chain: bloom extract, blur, composite + tone map + gamma. Outputs to the default framebuffer (FBO 0). If no effects are enabled, blits the HDR FBO directly. |

- The pipeline is entirely driven by the presence/absence of `PostProcessConfig` in the ECS context. No config = no post-processing overhead.
- MSAA and FXAA are mutually exclusive. Setting `aaMode` to 1 uses MSAA; setting it to 2 uses FXAA. Setting it to 0 disables both.

### Screen-Space Ambient Occlusion (`renderer/ssao.h`, namespace `ffe::renderer`)

SSAO adds contact shadows in corners, creases, and crevices where ambient light would naturally be occluded. It implements John Chapman's hemisphere-sampling algorithm: a per-pixel kernel samples the depth buffer in a hemisphere around each fragment, accumulates occlusion, and outputs a single-channel AO texture that darkens the ambient term in the final composite. The pass runs at **half resolution** for performance and is blurred with a 4x4 box filter to reduce noise. All GPU resources (FBOs, textures, sample kernel, noise texture) are created at init/resize only. No per-frame heap allocation. No virtual functions.

**Tier support:** LEGACY (OpenGL 3.3 core profile). SSAO uses basic texture sampling and FBOs, all available in GL 3.3.

**SSAOConfig struct (ECS singleton):**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | `bool` | `false` | Master SSAO switch |
| `radius` | `f32` | `0.5` | Sample hemisphere radius in view space. Clamped to [0.01, 5.0]. Larger values darken broader areas; smaller values catch only tight crevices. |
| `bias` | `f32` | `0.025` | Depth bias to prevent self-occlusion (acne). Clamped to [0.0, 0.5]. Increase if you see dark halos on flat surfaces. |
| `sampleCount` | `i32` | `32` | Number of hemisphere samples per pixel: 16, 32, or 64. Higher = better quality, more GPU cost. Clamped to nearest valid value. |
| `intensity` | `f32` | `1.0` | AO darkening multiplier. Clamped to [0.0, 5.0]. Values > 1.0 exaggerate occlusion; values < 1.0 soften it. |

The renderer activates SSAO when an `SSAOConfig` with `enabled == true` exists in the ECS registry context. When disabled (or absent), no SSAO pass runs and no GPU cost is incurred.

**C++ Pipeline Functions:**

| Function | Signature | Description |
|----------|-----------|-------------|
| `initSSAO` | `bool initSSAO(i32 width, i32 height)` | Create half-resolution SSAO FBO, blur FBO, 64-sample hemisphere kernel, and 4x4 noise texture. Call once after the GL context is available. Returns `false` on failure. |
| `resizeSSAO` | `void resizeSSAO(i32 width, i32 height)` | Recreate FBOs at a new resolution (half-res internally). Called from the framebuffer resize callback. Safe to call before `initSSAO` (no-op). |
| `renderSSAOPass` | `void renderSSAOPass(const glm::mat4& view, const glm::mat4& proj, const SSAOConfig& config)` | Run the SSAO pass: sample hemisphere from depth, write raw occlusion, then 4x4 box blur. Must be called after scene render (so depth is available) and before the final post-process composite. |
| `getSSAOTexture` | `u32 getSSAOTexture()` | Returns the GL texture name of the blurred AO result. Bind this during the final composite to multiply ambient occlusion. Returns 0 if SSAO is not initialised. |
| `shutdownSSAO` | `void shutdownSSAO()` | Delete all FBOs, textures, and kernel data. Safe to call multiple times. |
| `isSSAOInitialised` | `bool isSSAOInitialised()` | Returns `true` if the SSAO pipeline has been initialised. |
| `setSSAOShaders` | `void setSSAOShaders(rhi::ShaderHandle ssao, rhi::ShaderHandle blur)` | Provide SSAO and blur shader handles from `initShaderLibrary`. |
| `setSceneDepthTexture` | `void setSceneDepthTexture(u32 depthTextureGlName)` | Notify the SSAO pipeline of the scene depth texture (from the HDR FBO). Must be called each frame before `renderSSAOPass`. |

**Validation Utilities (header-only, no GL context required -- exposed for unit testing):**

| Function | Signature | Description |
|----------|-----------|-------------|
| `clampSSAOSamples` | `i32 clampSSAOSamples(i32 requested)` | Clamp sample count to nearest valid: <=16 -> 16, 17-48 -> 32, 49+ -> 64. |
| `clampSSAORadius` | `f32 clampSSAORadius(f32 radius)` | Clamp radius to [0.01, 5.0]. |
| `clampSSAOBias` | `f32 clampSSAOBias(f32 bias)` | Clamp bias to [0.0, 0.5]. |
| `clampSSAOIntensity` | `f32 clampSSAOIntensity(f32 intensity)` | Clamp intensity to [0.0, 5.0]. |

**Lua Bindings:**

| Binding | Signature | Description |
|---------|-----------|-------------|
| `ffe.enableSSAO(radius?, bias?, samples?)` | `(number?, number?, int?) -> nothing` | Enable SSAO. All parameters are optional with defaults: radius=0.5, bias=0.025, samples=32. Creates an `SSAOConfig` in the ECS context if one does not exist. Non-finite values are rejected (warning logged). Values are clamped to valid ranges. |
| `ffe.disableSSAO()` | `() -> nothing` | Set `SSAOConfig.enabled = false`. No-op if no `SSAOConfig` exists in the ECS context. The SSAO pass stops running but GPU resources remain allocated (cheap to re-enable). |
| `ffe.setSSAOIntensity(intensity?)` | `(number?) -> nothing` | Set the AO darkening multiplier. Default 1.0. Clamped to [0.0, 5.0]. Non-finite values rejected. No-op if no `SSAOConfig` exists (call `enableSSAO` first). |

**Important behavioural notes:**
- `enableSSAO` auto-creates the `SSAOConfig` if absent (you can call it without any prior setup).
- `disableSSAO` and `setSSAOIntensity` do NOT create the config -- they are no-ops if `enableSSAO` has not been called.
- SSAO requires the post-processing pipeline to be active (HDR FBO must exist for depth access). If post-processing is not initialised, SSAO has no depth to read.
- The SSAO pass runs at **half resolution** internally for performance. The blur pass smooths the result back to full resolution quality.

### 3D Common Usage Patterns

#### 1. Load a mesh and create a 3D entity

```lua
-- In your game's init() function (not per-frame):
local meshHandle = ffe.loadMesh("models/crate.glb")
if meshHandle == 0 then
    -- loadMesh already logged the error; handle gracefully
    return
end

-- Create an entity at position (0, 0, 0)
local entity = ffe.createEntity3D(meshHandle, 0, 0, 0)
```

#### 2. Animate a 3D entity with setTransform3D

```lua
-- Rotate a mesh 45 degrees around the Y axis, scaled up 2x:
ffe.setTransform3D(entity,
    0, 0, 0,          -- position x, y, z
    0, 45, 0,         -- rotation rx, ry, rz in degrees (YXZ order)
    2, 2, 2)          -- scale x, y, z

-- Spin the mesh over time in your update() function:
local angle = 0
function update(dt)
    angle = angle + 90 * dt  -- 90 degrees per second
    ffe.setTransform3D(entity,  0, 0, 0,  0, angle, 0,  1, 1, 1)
end
```

#### 3. Configure scene lighting

```lua
-- Warm afternoon sun from the upper-right:
ffe.setLightDirection(1.0, -1.0, 0.5)      -- direction (normalised internally)
ffe.setLightColor(1.0, 0.95, 0.8)          -- warm white
ffe.setAmbientColor(0.1, 0.1, 0.15)        -- cool blue-grey fill

-- Night scene:
ffe.setLightColor(0.05, 0.05, 0.1)
ffe.setAmbientColor(0.02, 0.02, 0.05)
```

#### 4. Enable shadows on a lit scene

```lua
-- After setting up meshes and lighting:
ffe.setLightDirection(1, -2, 1)
ffe.setLightColor(1, 0.95, 0.8)
ffe.setAmbientColor(0.1, 0.1, 0.15)

-- Enable 1024x1024 shadow map (default)
ffe.enableShadows()

-- For a small scene, tighten the shadow area for sharper texels:
ffe.setShadowArea(15, 15, 0.1, 30)
```

#### 5. Position the 3D perspective camera

```lua
-- Isometric-style view:
ffe.set3DCamera(5, 5, 5,   -- camera position
                0, 0, 0)   -- look-at target (world origin)

-- Third-person follow (called every frame with updated playerX/playerZ):
ffe.set3DCamera(playerX, playerY + 3, playerZ + 6,
                playerX, playerY,     playerZ)
```

#### 6. FPS camera — first-person perspective

```lua
-- Store yaw and pitch in Lua variables (updated by input each frame).
-- NEVER hardcode angles and call set3DCameraFPS with the same values every frame —
-- doing so achieves nothing useful. Always derive angles from accumulated input.

local camX, camY, camZ = 0, 1.8, 0   -- eye position (standing height)
local yaw   = 0    -- degrees, accumulated
local pitch = 0    -- degrees, accumulated

function update(entityId, dt)
    -- Mouse-look: accumulate yaw and pitch from input (example uses placeholder values)
    local mouseSensitivity = 0.15
    -- yaw = yaw + (deltaMouseX * mouseSensitivity)   -- fill in actual delta
    -- pitch = pitch + (deltaMouseY * mouseSensitivity)

    -- WASD movement (simplified, forward based on yaw only)
    local speed = 5.0
    local rad = math.rad(yaw)
    if ffe.isKeyHeld(ffe.KEY_W) then
        camX = camX + math.sin(rad) * speed * dt
        camZ = camZ + math.cos(rad) * speed * dt
    end
    if ffe.isKeyHeld(ffe.KEY_S) then
        camX = camX - math.sin(rad) * speed * dt
        camZ = camZ - math.cos(rad) * speed * dt
    end

    ffe.set3DCameraFPS(camX, camY, camZ, yaw, pitch)
end
```

#### 7. Orbit camera — third-person / inspection view

```lua
-- Store orbit state in Lua variables, updated from input each frame.
local targetX, targetY, targetZ = 0, 0, 0   -- point to orbit around
local orbitRadius  = 5.0
local orbitYaw     = 0      -- degrees, accumulated horizontal rotation
local orbitPitch   = 20     -- degrees, start slightly above the target

function update(entityId, dt)
    -- Right-drag or arrow keys rotate the orbit (example uses arrow keys)
    local rotSpeed = 60  -- degrees per second
    if ffe.isKeyHeld(ffe.KEY_LEFT)  then orbitYaw   = orbitYaw   - rotSpeed * dt end
    if ffe.isKeyHeld(ffe.KEY_RIGHT) then orbitYaw   = orbitYaw   + rotSpeed * dt end
    if ffe.isKeyHeld(ffe.KEY_UP)    then orbitPitch  = orbitPitch  + rotSpeed * dt end
    if ffe.isKeyHeld(ffe.KEY_DOWN)  then orbitPitch  = orbitPitch  - rotSpeed * dt end

    -- pitch is clamped to [-85, 85] internally — no need to clamp in Lua
    ffe.set3DCameraOrbit(targetX, targetY, targetZ,
                         orbitRadius, orbitYaw, orbitPitch)
end
```

#### 8. Combine a 3D world with a 2D HUD

```lua
-- 3D entities: use ffe.createEntity3D / ffe.setTransform3D
local mesh = ffe.loadMesh("models/ship.glb")
local shipEntity = ffe.createEntity3D(mesh, 0, 0, 0)

-- 2D HUD: use the standard 2D API (sprites, text, drawRect)
-- The 3D pass renders before the 2D pass automatically.
-- 2D HUD entities live in the same world, just use Transform (not Transform3D).
-- No extra setup required — the render loop handles the pass ordering.

```

### 3D What NOT to Do

#### Do not call loadMesh per frame

```lua
-- BAD -- file I/O + GPU upload every frame, severe frame drops and VRAM leak
function update(dt)
    local mesh = ffe.loadMesh("models/crate.glb")  -- WRONG
end

-- GOOD -- load once at init, store the handle, reuse it every frame
local meshHandle
function init()
    meshHandle = ffe.loadMesh("models/crate.glb")
end
```

#### Do not use .gltf files — only .glb is accepted

```lua
-- BAD -- .gltf extension is rejected before any file I/O
local mesh = ffe.loadMesh("models/crate.gltf")  -- returns 0, logs error

-- GOOD -- binary glTF only
local mesh = ffe.loadMesh("models/crate.glb")
```

#### Do not load more than 100 meshes simultaneously

The mesh asset pool has a hard cap of `MAX_MESH_ASSETS = 100` slots. `loadMesh` returns 0 when the pool is full. Unload unused meshes with `ffe.unloadMesh` when switching scenes.

```lua
-- When transitioning to a new scene, free meshes from the old scene:
ffe.unloadMesh(oldMeshHandle)
oldMeshHandle = 0
```

#### Do not pass a zero-length vector to setLightDirection

`ffe.setLightDirection(0, 0, 0)` is silently rejected (the previous value is preserved and a warning is logged). Passing a zero vector would produce `NaN` in the GLSL `normalize()` call, corrupting all lit mesh rendering. Always pass a non-zero direction vector.

```lua
-- BAD
ffe.setLightDirection(0, 0, 0)  -- rejected, warning logged

-- GOOD
ffe.setLightDirection(0, -1, 0)  -- straight down
```

#### Do not omit scale in setTransform3D

All nine parameters to `setTransform3D` default to 0 via `luaL_optnumber` when omitted, including `sx`, `sy`, `sz`. A scale of `{0, 0, 0}` collapses the mesh to a point. Always provide explicit scale values.

```lua
-- BAD -- scale defaults to {0,0,0}, mesh invisible
ffe.setTransform3D(entity, 0, 0, 0, 0, 45, 0)

-- GOOD -- explicit scale
ffe.setTransform3D(entity, 0, 0, 0, 0, 45, 0, 1, 1, 1)
```

#### Do not call set3DCameraFPS or set3DCameraOrbit with hardcoded angles every frame

Both functions are thin math wrappers — they compute a target point from angles and write it to the camera. Calling them with the same constant values every frame is valid but pointless; the camera does not move. The intended use is to accumulate yaw/pitch in Lua variables, update them from input (mouse delta, joystick, key presses), and pass the accumulated values to the function each frame.

```lua
-- BAD -- camera never actually rotates; the same angles are passed every frame
function update(entityId, dt)
    ffe.set3DCameraFPS(0, 1.8, 0, 45, 10)  -- WRONG: hardcoded every frame
end

-- GOOD -- accumulate yaw/pitch in variables, update from input
local yaw, pitch = 0, 0
local camX, camY, camZ = 0, 1.8, 0

function update(entityId, dt)
    -- yaw = yaw + (deltaMouseX * sensitivity)  -- update from actual input
    -- pitch = pitch + (deltaMouseY * sensitivity)
    ffe.set3DCameraFPS(camX, camY, camZ, yaw, pitch)
end
```

The same rule applies to `set3DCameraOrbit`: store `orbitYaw` and `orbitPitch` as Lua variables and update them from input each frame.

#### 19. Toggle SSAO at runtime (settings menu)

#### Do not call ffe.enableSSAO() every frame

`enableSSAO` creates an `SSAOConfig` in the ECS context if none exists. While repeated calls are cheap when the config already exists, calling it per-frame is a code smell. Call it once at scene init.

```lua
-- BAD -- unnecessary per-frame call
function update(entityId, dt)
    ffe.enableSSAO(0.5, 0.025, 32)
end

-- GOOD -- call once at init
function init()
    ffe.enableSSAO(0.5, 0.025, 32)
end
```

#### Do not call ffe.setSSAOIntensity() before ffe.enableSSAO()

`setSSAOIntensity` does NOT create the `SSAOConfig` -- it is a no-op if `enableSSAO` has not been called. Always enable SSAO first.

```lua
-- BAD -- no-op, SSAOConfig does not exist yet
ffe.setSSAOIntensity(2.0)

-- GOOD -- enable first, then adjust
ffe.enableSSAO()
ffe.setSSAOIntensity(2.0)
```

#### Do not use high sample counts on low-end hardware without profiling

64 samples per pixel at half resolution is manageable on most LEGACY-tier GPUs, but on very old hardware, drop to 16 samples. Measure frame time before and after.

```lua
-- Conservative settings for older GPUs
ffe.enableSSAO(0.3, 0.025, 16)
```

#### Do not call ffe.enableBloom() every frame

`enableBloom` creates a `PostProcessConfig` if none exists (ECS context emplace). While subsequent calls are cheap when the config already exists, calling it per-frame is a code smell. Call it once at scene init.

```lua
-- BAD -- unnecessary per-frame call
function update(entityId, dt)
    ffe.enableBloom(0.8, 1.2)  -- wasteful per-frame call
end

-- GOOD -- call once at init
function init()
    ffe.enableBloom(0.8, 1.2)
end
```

#### Do not use tone mapping without gamma correction

Tone mapping maps HDR values into [0,1] range (linear space). Without gamma correction, the result appears washed-out on sRGB displays. The engine guards against this: `setToneMapping` with mode > 0 automatically sets `gammaEnabled = true`. Do not manually disable gamma after enabling tone mapping unless you have a specific reason (e.g., you are writing to a linear framebuffer).

```lua
-- BAD -- washed-out result on sRGB displays
ffe.setToneMapping(2)
ffe.setGammaCorrection(false)  -- defeats the purpose of tone mapping

-- GOOD -- let setToneMapping handle gamma automatically
ffe.setToneMapping(2)  -- gammaEnabled is set to true internally
```

#### Do not pass non-finite values to ffe.enableBloom

`NaN` and `Inf` values for threshold or intensity are rejected with a warning logged. The bloom state is not modified. Always pass finite, non-negative numbers.

```lua
-- BAD -- rejected, warning logged
ffe.enableBloom(0/0, 1.0)   -- NaN threshold
ffe.enableBloom(1.0, 1/0)   -- Inf intensity

-- GOOD
ffe.enableBloom(0.8, 1.2)
```

#### Do not pass an invalid mode to ffe.setToneMapping

Only modes 0, 1, and 2 are valid. Values outside this range are rejected with a warning. The tone mapper setting is not modified.

```lua
-- BAD -- rejected, warning logged
ffe.setToneMapping(5)

-- GOOD
ffe.setToneMapping(0)  -- none
ffe.setToneMapping(1)  -- Reinhard
ffe.setToneMapping(2)  -- ACES filmic
```

#### Do not pass a negative handle to setMeshNormalMap or setMeshSpecularMap

```lua
-- BAD -- negative handle is rejected, warning logged
ffe.setMeshNormalMap(entity, -1)

-- GOOD -- pass 0 to disable, or a positive handle from ffe.loadTexture
ffe.setMeshNormalMap(entity, 0)             -- disable normal mapping
ffe.setMeshNormalMap(entity, myNormalTex)   -- myNormalTex > 0 from ffe.loadTexture
```

#### Do not exceed 8 point lights

The point light array has a fixed capacity of 8 slots (indices 0-7). `addPointLight` with an out-of-range index is rejected with a warning. If you need more lights, consider baking static lights into your scene or using multiple directional lights in future releases.

```lua
-- BAD -- index 8 is out of range (valid: 0-7)
ffe.addPointLight(8, 0, 0, 0, 1, 1, 1, 10)  -- rejected, warning logged

-- GOOD -- use indices 0-7
ffe.addPointLight(7, 0, 0, 0, 1, 1, 1, 10)  -- last valid slot
```

#### Do not call addPointLight per frame

`addPointLight` sets all properties and enables the light. If you only need to update position or color per frame, use `setPointLightPosition` or `setPointLightColor` instead. Calling `addPointLight` per frame is functionally correct but wasteful — it overwrites all properties when you likely only want to change one.

```lua
-- BAD -- re-sets all properties every frame just to move the light
function update(entityId, dt)
    ffe.addPointLight(0, newX, newY, newZ, 1, 1, 1, 10)  -- wasteful
end

-- GOOD -- update only what changed
function update(entityId, dt)
    ffe.setPointLightPosition(0, newX, newY, newZ)
end
```

#### Do not pass a negative handle to setMeshTexture

```lua
-- BAD -- negative handle is rejected, warning logged, Material3D unchanged
ffe.setMeshTexture(entity, -1)

-- GOOD -- pass 0 to clear (white fallback) or a positive handle from ffe.loadTexture
ffe.setMeshTexture(entity, 0)            -- clear: reverts to untextured white
ffe.setMeshTexture(entity, myTexHandle)  -- myTexHandle > 0 from ffe.loadTexture
```

#### Do not expect transparent 3D meshes

The 3D pass uses `BlendMode::NONE` (opaque geometry only). Transparent 3D meshes are not supported in this release. The `Material3D.diffuseColor` alpha channel affects the shader output but blending is disabled, so semi-transparent meshes will clip hard against their depth. Transparency for 3D geometry is planned for a future ADR.

#### Do not manually batch meshes to avoid draw calls

GPU instancing is automatic. If 2 or more entities share the same `MeshHandle`, the renderer detects this and issues a single `glDrawElementsInstanced` call instead of N individual draw calls. You do not need to do anything — just create entities with the same mesh handle and instancing kicks in.

#### Do not assume per-entity materials in instanced batches

When multiple entities share a mesh and are drawn as an instanced batch, the `Material3D` from the **first entity in the batch** is used for the entire draw call. Per-instance material variation (different colors, textures per entity) is not yet supported. If you need different materials, use separate `MeshHandle`s or accept that the first entity's material wins.

#### Do not expect skinned meshes to be instanced

Entities with a `Skeleton` component (skeletal animation) are always drawn individually, never instanced. Bone transforms are per-entity and cannot be batched into the current instance buffer layout.

### GPU Instancing (`renderer/gpu_instancing.h`, `renderer/mesh_renderer.h`)

#### Purpose

GPU instancing reduces draw calls for repeated meshes. When 2+ entities share the same `MeshHandle`, the renderer collects their model matrices into a shared instance VBO and issues one `glDrawElementsInstanced` call per mesh group, instead of one `glDrawElements` per entity. This is fully automatic — game code does not opt in or configure anything.

**Tier support:** LEGACY (OpenGL 3.3 core). `glDrawElementsInstanced` and `glVertexAttribDivisor` are core in GL 3.1+.

#### Constants (`renderer/gpu_instancing.h`, namespace `ffe::renderer`)

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_INSTANCES_PER_BATCH` | `1024` | Maximum instances in a single instanced draw call. Buffer size: 64 KB. |
| `INSTANCE_BUFFER_SIZE` | `65536` | Total instance VBO size in bytes (1024 * 64). |
| `INSTANCE_ATTR_SLOT_BASE` | `8` | First vertex attribute slot used by instance data (slots 8-11 for mat4 columns). |
| `INSTANCE_ATTR_SLOT_COUNT` | `4` | Number of vec4 attribute slots consumed (one per mat4 column). |
| `INSTANCING_THRESHOLD` | `2` | Minimum entity count sharing a MeshHandle before instancing activates. With 1 entity, the regular single-draw path is used. |

#### Data Layout (`InstanceData` struct)

```cpp
struct InstanceData {
    glm::mat4 modelMatrix;  // 64 bytes per instance
};
```

Each instance uploads one model matrix (64 bytes). The mat4 is split across vertex attribute slots 8-11 (one vec4 column per slot) using `glVertexAttribDivisor(slot, 1)`. Slots 0-5 are used by mesh vertex data (position, normal, texcoord, tangent, joints, weights). Slots 6-7 are spare.

#### C++ API (`renderer/mesh_renderer.h`, namespace `ffe::renderer`)

| Function | Signature | Description |
|----------|-----------|-------------|
| `initInstancing` | `void initInstancing()` | Create the shared instance VBO (64 KB, `GL_STREAM_DRAW`). Call once after GL context is available. No-op in headless mode. Called by `Application::startup()`. |
| `shutdownInstancing` | `void shutdownInstancing()` | Destroy the shared instance VBO. Called by `Application::shutdown()`. |
| `getInstanceCount` | `u32 getInstanceCount(World& world, u32 meshHandleId)` | Count how many entities share the given mesh handle. Iterates the `Mesh` component view. Used by the `ffe.getInstanceCount` Lua binding for debugging/profiling. |

#### Lua Binding

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.getInstanceCount(meshHandle)` | integer | Returns how many entities in the world have the given mesh handle. Returns 0 if the handle is invalid, out of range, or no World is registered. Useful for debugging instancing — if the count is >= 2, those entities are being instanced automatically. |

#### How It Works (Internal)

1. `meshRenderSystem` iterates all entities with `Transform3D` + `Mesh`.
2. Entities are grouped by `MeshHandle`. Skinned entities (those with a `Skeleton` component) are excluded from instancing and drawn individually.
3. For each mesh group with >= 2 entities: model matrices are collected into `InstanceData` structs, uploaded to the shared instance VBO via `glBufferSubData`, and drawn with `glDrawElementsInstanced`.
4. For mesh groups with exactly 1 entity (or skinned meshes): the regular single-draw path is used (uniform-based model matrix, `glDrawElements`).
5. If a mesh group exceeds `MAX_INSTANCES_PER_BATCH` (1024), it is split into multiple instanced draw calls.

#### Usage Example (Lua)

```lua
function init()
    local treeMesh = ffe.loadMesh("models/tree.glb")

    -- Place 50 trees — instancing kicks in automatically (>= 2 entities)
    for i = 1, 50 do
        local x = math.random(-20, 20)
        local z = math.random(-20, 20)
        ffe.createEntity3D(treeMesh, x, 0, z)
    end

    -- Debug: confirm instancing is active
    local count = ffe.getInstanceCount(treeMesh)
    ffe.log("Tree instances: " .. count)  -- prints "Tree instances: 50"
    -- 50 trees = 1 instanced draw call instead of 50 individual calls
end
```

## Multi-Backend Support (RHI Backend Selection)

The renderer supports multiple graphics API backends behind the same `ffe::rhi` interface. The backend is selected at **compile time** via the CMake variable `FFE_BACKEND`. Only one backend is linked into any given build.

**`RhiBackend` enum** (`renderer/rhi_types.h`):

| Value | Description |
|-------|-------------|
| `RhiBackend::OPENGL` | OpenGL 3.3+ backend (default). Used for RETRO, LEGACY, and STANDARD tiers. |
| `RhiBackend::VULKAN` | Vulkan 1.0+ backend. Used for STANDARD and MODERN tiers. |

**Build configuration:**

```bash
# Default build -- OpenGL LEGACY (unchanged from before Vulkan support)
cmake -B build -G Ninja -DFFE_TIER=LEGACY
# Equivalent to: -DFFE_BACKEND=OPENGL (the default)

# Vulkan build -- requires STANDARD or MODERN tier
cmake -B build-vk -G Ninja -DFFE_TIER=STANDARD -DFFE_BACKEND=VULKAN
```

**Tier/backend constraints** (enforced by CMake at configure time):

| Tier | Allowed Backends |
|------|-----------------|
| RETRO | OPENGL only |
| LEGACY | OPENGL only |
| STANDARD | OPENGL or VULKAN |
| MODERN | VULKAN only |

Setting `FFE_BACKEND=VULKAN` with `FFE_TIER=LEGACY` or `FFE_TIER=RETRO` is a CMake configuration error.

**Compile-time defines:** The selected backend sets exactly one of:
- `FFE_BACKEND_OPENGL=1` -- OpenGL backend is active
- `FFE_BACKEND_VULKAN=1` -- Vulkan backend is active

**Vulkan backend status (Phase 8 — feature-complete):** M1 established core init/shutdown/frame lifecycle (clear-color rendering). M2 added VMA-backed buffer creation (`createVertexBuffer`, `createIndexBuffer`), SPIR-V shader module loading (`createShaderModules`), and graphics pipeline creation (`createGraphicsPipeline`). M3 added texture creation and destruction (`createTexture` / `destroyTexture` backed by VMA image allocations and staging-buffer uploads), uniform buffer management (`createUniformBuffers` / `destroyUniformBuffers` with persistent mapping, per-frame-in-flight), and descriptor set management (pool creation, layout creation, set allocation, UBO and texture descriptor updates). M4 completed the Vulkan RHI feature surface: a `ResourceManager` with static-capacity pools (256 buffers, 256 textures, 32 shaders) maps opaque `BufferHandle`/`TextureHandle`/`ShaderHandle` IDs to Vulkan objects with no heap allocation; `drawArrays` and `drawIndexed` are fully functional with pipeline binding, dynamic viewport/scissor, UBO upload from a `UniformStaging` ring, and per-draw texture descriptor updates; `SceneUBO` (256 bytes, 4×mat4) and `LightUBO` (64 bytes, 4×vec4) provide std140-compatible uniform packing for Blinn-Phong rendering. M5 completed the phase with depth buffering, build-time SPIR-V compilation, and a full Blinn-Phong lighting shader. Depth buffer: `VulkanContext` holds a VMA-allocated `VkImage` + `VkImageView` in `VK_FORMAT_D32_SFLOAT` (with fallback chain: D32_SFLOAT → D32_SFLOAT_S8_UINT → D24_UNORM_S8_UINT via `findDepthFormat`); depth resources are created/destroyed alongside the swap chain and integrated into the render pass (depth attachment with `VK_ATTACHMENT_LOAD_OP_CLEAR` at 1.0). `PipelineConfig` gained `depthTestEnabled` (default `true`), `depthWriteEnabled` (default `true`), and `depthCompareOp` (default `VK_COMPARE_OP_LESS`). Build-time SPIR-V: `cmake/embed_spirv.cmake` compiles `.vert`/`.frag` GLSL 450 sources to `.spv` via `glslc` (or `glslangValidator` fallback), then embeds them as `static constexpr u32` arrays in generated headers under `ffe::rhi::vk::spv`; the build defines `FFE_SPIRV_COMPILED=1` when a compiler is found, with pre-committed fallback headers for environments without a SPIR-V compiler. Three shader pairs ship: `triangle` (flat color), `textured` (sampled diffuse), and `blinn_phong` (directional light with diffuse texture, Blinn-Phong specular). Headless mode works identically for both backends (returns incrementing handle IDs, no GPU operations). All `ffe::rhi` functions now have working implementations on both OpenGL and Vulkan backends. **Phase 8 is feature-complete for its planned scope.**

**For game developers:** You do not need to change any game code to switch backends. The `ffe::rhi` API is identical regardless of backend. Switch backends by rebuilding with `-DFFE_BACKEND=VULKAN`. The default build (`-DFFE_TIER=LEGACY`) remains pure OpenGL 3.3 and is completely unaffected by the Vulkan addition.

**Internal Vulkan API:** The `engine/renderer/vulkan/` directory contains backend-internal headers (`vk_init.h`, `vk_buffer.h`, `vk_shader.h`, `vk_pipeline.h`, `vk_texture.h`, `vk_uniform.h`, `vk_descriptor.h`, `vk_resource_manager.h`) and implementation files. These are NOT part of the public API and must not be included by game code. They are conditionally compiled only when `FFE_BACKEND_VULKAN=1`.

## API Review Notes

This section documents naming inconsistencies, discoverability issues, and interface concerns found during the API review. These are recommendations for future improvement, not bugs.

### 1. Inconsistent function naming style for ShaderLibrary

The shader library uses `getShader()` as a free function, but the other "get" operations in the codebase are member functions (e.g., `World::getComponent`). Meanwhile, `initShaderLibrary` and `shutdownShaderLibrary` follow the free-function style of the RHI. This is internally consistent within the renderer, but the name `getShader` is generic and could collide with user code. Consider `getBuiltinShader` for clarity.

### 2. Transform and Sprite are defined in render_system.h, not in core

`Transform` is a fundamental component that physics, scripting, and gameplay systems will all need. It is currently defined in `renderer/render_system.h` inside namespace `ffe`, with a comment acknowledging it belongs in core. This means any system that needs `Transform` must include a renderer header, creating an unwanted dependency direction. Recommend moving `Transform` to `engine/core/components.h`.

### 3. Sprite.size is uninitialized by default

Every other field in `Sprite` has a sensible default. `size` does not, which means forgetting to set it produces garbage geometry. Recommend defaulting to `{1.0f, 1.0f}` or `{32.0f, 32.0f}`.

### 4. pushDrawCommand silently drops commands when full

When the render queue is full, `pushDrawCommand` silently does nothing. For a 16-year-old debugging why their sprites disappeared, this is invisible. Recommend logging a warning on the first dropped command per frame (not every one, to avoid log spam).

### 5. endFrame takes a raw GLFWwindow pointer

This leaks the windowing backend into the RHI interface. The RHI should ideally store the window reference internally (set during `init`) rather than requiring the caller to pass it every frame. **Note (Phase 8 M1):** The Vulkan backend now receives the window via `RhiConfig::window` at init time and stores it internally. The `endFrame(GLFWwindow*)` parameter is still used by the OpenGL backend for `glfwSwapBuffers`. The Vulkan backend ignores the `endFrame` window parameter (it uses the stored pointer). A future cleanup could remove the `endFrame` parameter for both backends.

### 6. Mixed namespace organization for components

`Transform` and `Sprite` are in namespace `ffe` (not `ffe::renderer`), while all other renderer types are in `ffe::renderer` or `ffe::rhi`. This is intentional (they are ECS components, not renderer internals) but may confuse newcomers who see them defined in a renderer header. Moving them to core would fix both the namespace and the dependency issue.

### 7. Sort key bit layout is undocumented for users

`makeSortKey` is powerful but its bit layout (4 bits layer, 8 bits shader, 12 bits texture, 24 bits depth, 16 bits sub-order) is only visible by reading the implementation. Users who need custom sort keys have no documentation to guide them. The sort key constants should be named and documented.

### 8. No way to query whether RHI is initialized

There is no `rhi::isInitialized()` function. If user code tries to create a texture before `Application::run()` starts the renderer, they get undefined behaviour instead of a clear error.

### 9. setTransform3D scale parameters default to 0, not 1

`ffe.setTransform3D` uses `luaL_optnumber` with a default of `0.0` for the scale arguments (`sx`, `sy`, `sz`). A caller who omits the scale arguments gets `{0, 0, 0}` scale, which collapses the mesh to a point and renders nothing. The default should be `1.0` for scale. This is a MINOR usability issue — the "What NOT to Do" section documents the workaround. Recommend changing the `luaL_optnumber` defaults for `sx`, `sy`, `sz` to `1.0` in a future patch.

### 10. ffe.unloadMesh is present in the Lua bindings but absent from ADR-007 Section 11.1

The ADR's explicit Lua binding table in Section 11.1 does not list `ffe.unloadMesh`, though the C++ API (Section 10) specifies `unloadMesh` and the implementation correctly binds it as `ffe.unloadMesh`. The binding is correct and complete — this is an ADR documentation gap only. The omission from the task brief has been corrected in this `.context.md` by documenting `ffe.unloadMesh` in the Lua bindings table above.

## Texture Loading (`renderer/texture_loader.h`, namespace `ffe::renderer`)

Loads image files from disk and uploads them to the GPU as textures. This is a load-time API, not a per-frame API. All functions perform file I/O and/or GPU uploads.

### System Purpose

`texture_loader` bridges the filesystem and the RHI. It uses stb_image to decode PNG, JPG, BMP, and TGA files from disk (detected by magic bytes, not extension), always to RGBA8, then calls `rhi::createTexture()` to upload the pixel data to GPU memory. The result is a `TextureHandle` you can assign directly to a `Sprite` component or use with `rhi::bindTexture`.

Path traversal attacks are rejected before any syscall touches the filesystem. The asset root is set once at startup and is immutable -- it cannot be redirected by script or user input at runtime.

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_TEXTURE_DIMENSION` | `8192` | Maximum width or height in pixels. Enforced before and after decoding |
| `MAX_TEXTURE_BYTES` | `256 MiB` (268,435,456) | Maximum decoded pixel data size. Prevents pathological images that pass per-dimension checks |
| `MAX_TEXTURE_FILE_BYTES` | `64 MiB` (67,108,864) | Maximum raw file size accepted before decoding begins |

### TextureLoadParams struct

Optional parameters for the `loadTexture` overloads that accept explicit control over sampling. All fields have defaults that match the original zero-parameter overloads.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `filter` | `ffe::rhi::TextureFilter` | `LINEAR` | Sampling filter. Use `NEAREST` for pixel art (hard pixels, no blurring). Use `LINEAR` for photographs or smooth art. |
| `wrap` | `ffe::rhi::TextureWrap` | `CLAMP_TO_EDGE` | Wrap mode. Use `CLAMP_TO_EDGE` for sprites (no edge repetition). Use `REPEAT` for tiling textures. |

### Public API

| Function | Signature | Description |
|----------|-----------|-------------|
| `setAssetRoot` | `bool setAssetRoot(const char* absolutePath)` | Set the root directory for all texture loads. Must be called once before `loadTexture`. `absolutePath` must be an absolute, existing directory path. **Write-once**: once set to a valid path, further calls are silently ignored and return false (this is intentional -- prevents runtime redirection by untrusted code). Returns true on success, false on failure with a logged error. |
| `getAssetRoot` | `const char* getAssetRoot()` | Query the current asset root. Returns an empty string if `setAssetRoot()` has not been called successfully. Useful for diagnostics and test assertions. |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path)` | Load an image file relative to the global asset root. Returns a valid `TextureHandle` (id != 0) on success. Returns `TextureHandle{0}` on failure and logs the specific reason. The caller owns the handle and must call `unloadTexture()` when done. Uses `LINEAR` filter and `CLAMP_TO_EDGE` wrap. |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path, const char* assetRoot)` | Load from a specific asset root without changing the global root. **Path rules for `loadTexture`:** The `path` parameter must be a relative path with no traversal sequences. The following are rejected:
- Paths starting with `/` or `\` (absolute Unix/Windows)
- Paths containing drive letters (`C:`)
- Paths containing `../`, `..\`, `/..`, `\..` (directory traversal)
- Paths at or beyond `PATH_MAX` characters

Rejection is logged via `FFE_LOG_ERROR` and returns `TextureHandle{0}`.

**Supported image formats:** PNG, JPG/JPEG, BMP, TGA. Format is detected from file content (magic bytes), not extension. All images are decoded to RGBA8 (4 channels).

### Common Usage Patterns

#### Load a texture at game startup and assign it to a sprite

```cpp
#include "renderer/texture_loader.h"
#include "renderer/render_system.h" // for Sprite component

// In Application::startup() -- once, before the game loop:
ffe::renderer::setAssetRoot("/home/user/mygame/assets/textures");

// Load a texture. Path is relative to the asset root.
ffe::rhi::TextureHandle heroTex = ffe::renderer::loadTexture("hero.png");

if (!ffe::rhi::isValid(heroTex)) {
    FFE_LOG_ERROR("Game", "Failed to load hero.png -- check asset root and path");
    return false; // or fall back to a placeholder
}

// Assign to a Sprite component.
ffe::EntityId hero = world.createEntity();
world.addComponent<ffe::Transform>(hero, glm::vec3{0.0f, 0.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(hero);
sprite.texture = heroTex;
sprite.size = {32.0f, 32.0f};

// At shutdown:
ffe::renderer::unloadTexture(heroTex);
```

#### Load a pixel-art texture (no blurring)

```cpp
// Pixel art must use NEAREST filter -- LINEAR blurs hard pixels.
ffe::renderer::TextureLoadParams pixelArtParams;
pixelArtParams.filter = ffe::rhi::TextureFilter::NEAREST;
pixelArtParams.wrap   = ffe::rhi::TextureWrap::CLAMP_TO_EDGE;

ffe::rhi::TextureHandle heroTex = ffe::renderer::loadTexture("hero.png", pixelArtParams);

// Or use brace initialisation (filter, wrap order matches struct declaration):
ffe::rhi::TextureHandle tileTex = ffe::renderer::loadTexture(
    "terrain/grass.png",
    {ffe::rhi::TextureFilter::NEAREST, ffe::rhi::TextureWrap::REPEAT}
);
```

#### Check for load failure and use a fallback

```cpp
// Try to load a custom skin; fall back to the default if it is missing.
ffe::rhi::TextureHandle skinTex = ffe::renderer::loadTexture("skins/custom.png");

if (!ffe::rhi::isValid(skinTex)) {
    // loadTexture already logged the error.
    // Fall back to the default skin loaded at startup.
    FFE_LOG_WARN("Game", "Custom skin not found -- using default");
    skinTex = defaultSkinHandle; // previously loaded at startup
}

auto& sprite = world.addComponent<ffe::Sprite>(entity);
sprite.texture = skinTex;
sprite.size = {32.0f, 48.0f};
```

### What NOT to Do

#### Do not call loadTexture per-frame

`loadTexture` calls `stat()`, `realpath()`, `fopen()`, `stbi_load()` (CPU decode), and `rhi::createTexture()` (GPU upload). Calling it inside a system update loop will cause severe frame drops and GPU memory leaks (each call allocates a new texture that you must separately destroy).

```cpp
// BAD -- creates a new GPU texture every frame
void mySystem(ffe::World& world, float dt) {
    auto tex = ffe::renderer::loadTexture("player.png");  // WRONG: file I/O + GPU upload per frame
    // also leaked -- unloadTexture never called
}

// GOOD -- load once at startup, store the handle, reuse it every frame
```

#### Do not forget to call setAssetRoot() before loadTexture

If `loadTexture(path)` (single-argument form) is called before `setAssetRoot()`, it returns `TextureHandle{0}` and logs an error. There is no default current-working-directory fallback. This is intentional -- silently loading from cwd creates unpredictable behaviour when the working directory changes.

```cpp
// BAD
ffe::rhi::TextureHandle tex = ffe::renderer::loadTexture("hero.png");
// Logs: "loadTexture: called before setAssetRoot() -- no asset root configured"
// Returns TextureHandle{0}.

// GOOD
ffe::renderer::setAssetRoot("/path/to/assets/textures");
ffe::rhi::TextureHandle tex = ffe::renderer::loadTexture("hero.png");
```

#### Do not use absolute paths or ../ in the path argument

Absolute paths and directory traversal sequences are rejected for security. Always provide a path relative to the asset root.

```cpp
// BAD -- all of these are rejected
ffe::renderer::loadTexture("/etc/passwd");          // absolute path
ffe::renderer::loadTexture("../../../etc/shadow");  // traversal
ffe::renderer::loadTexture("C:\\Windows\\win.ini"); // drive letter

// GOOD
ffe::renderer::loadTexture("enemies/goblin.png");   // relative, no traversal
ffe::renderer::loadTexture("ui/button.png");
```

#### Do not ignore the return value

`loadTexture` returns `TextureHandle{0}` on failure. Passing an invalid handle to `rhi::bindTexture` or assigning it to a `Sprite` is not a crash (the engine handles invalid handles gracefully), but your sprite will render with no texture (white rectangle from the default 1x1 white pixel). Always check `ffe::rhi::isValid()` after loading.

```cpp
// BAD -- ignores failure
auto tex = ffe::renderer::loadTexture("missing.png");
sprite.texture = tex; // tex.id may be 0 -- sprite renders white

// GOOD
auto tex = ffe::renderer::loadTexture("missing.png");
if (!ffe::rhi::isValid(tex)) {
    FFE_LOG_ERROR("Game", "Texture load failed");
    // Use fallback or abort
}
sprite.texture = tex;
```

### Security Note

`loadTexture` implements defence-in-depth path traversal protection:

1. **String-level check** (`isPathSafe`): Rejects absolute paths, drive letters, and `../` sequences before any syscall.
2. **Length check**: Rejects paths that would overflow the `PATH_MAX` concatenation buffer.
3. **Canonicalisation** (`realpath`): Resolves symlinks and encoded sequences, then verifies the canonical path still begins with the asset root. A symlink in the assets directory that points outside it is caught here.
4. **Prefix verification**: The canonical path must have the asset root as a prefix, with the root's final character followed by `/` or end-of-string.

This means you cannot load textures from outside the asset root directory, regardless of what path string you provide. This property is guaranteed even if the game loads texture paths from a level file or network packet.

The asset root is **write-once**: `setAssetRoot()` cannot be called a second time with a different path. This prevents a compromised script or malicious level file from redirecting texture loads to a different directory.

### Tier Support

| Tier | Supported |
|------|-----------|
| RETRO | Yes (stb_image decode is CPU-side and tier-independent; GPU upload delegates to `rhi::createTexture`) |
| LEGACY | Yes (primary target) |
| STANDARD | Yes |
| MODERN | Yes |

All loaded textures use RGBA8 format, which is supported on all tiers. The `NEAREST` vs `LINEAR` filter choice determines sampling quality. Pass a `TextureLoadParams` struct to `loadTexture` to select the filter and wrap mode — pixel art should use `NEAREST`, smooth art and photographs should use `LINEAR`.

## Terrain System (`renderer/terrain.h`, namespace `ffe::renderer`)

### System Purpose

Chunked heightmap terrain rendering for outdoor 3D scenes. Loads a grayscale heightmap (raw float data or PNG image), generates a grid of independent mesh chunks (each with its own VAO/VBO/IBO), and renders them with either a single diffuse texture (`MESH_BLINN_PHONG` shader) or a 4-layer splat-map material (`TERRAIN` shader with RGBA blend weights and optional triplanar projection for steep surfaces). Supports distance-based LOD (3 levels: full, half, quarter resolution), view-frustum culling, and optional background chunk streaming (M4) for large open-world terrains. Supports CPU-side bilinear height queries so game code can place characters, objects, and cameras on the terrain surface. This is a load-time API -- terrain generation involves heap allocation and GPU uploads. Do not call loading or material setup functions per frame.

**Thread safety:** Loading and material setup functions must be called from the render thread. When streaming is enabled (`setTerrainStreamingRadius` with radius > 0), chunk uploads are managed by a background worker thread; the Lua-facing API remains single-threaded and safe to call from game code.

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_TERRAIN_ASSETS` | `4` | Maximum terrains loaded simultaneously. Each terrain at default resolution uses ~56 MB GPU. 4 terrains = ~224 MB, safe for LEGACY tier (1 GB VRAM). |
| `MAX_CHUNK_RESOLUTION` | `128` | Maximum vertices per chunk side. 128x128 = 16,384 vertices per chunk. |
| `MAX_CHUNKS_TOTAL` | `256` | Maximum total chunks per terrain (e.g. 16x16). |
| `MAX_HEIGHTMAP_DIMENSION` | `2048` | Maximum heightmap width or height in pixels. Pre-parse guard. |
| `MAX_HEIGHTMAP_FILE_BYTES` | `16 MB` | Maximum heightmap file size. Pre-parse guard. |

| `ffe.unloadTerrain()` | nil | Unload all terrains and free GPU resources. |
| `ffe.setTerrainTexture(textureHandle)` | nil | Set diffuse texture on the first active terrain (M1 single-texture path). Pass a handle from `ffe.loadTexture`. |
| `ffe.setTerrainSplatMap(textureId)` | nil | Set the RGBA splat map texture on the first active terrain. Pass an RGBA texture handle from `ffe.loadTexture`. Enables the TERRAIN shader with 4-layer blending. |
| `ffe.setTerrainLayer(layerIndex, textureId, uvScale)` | nil | Set a terrain texture layer (0-3) on the first active terrain. `layerIndex`: 0-3 (maps to splat map R/G/B/A). `textureId`: texture handle. `uvScale`: tiling multiplier (default 16.0, must be > 0). |
| `ffe.setTerrainTriplanar(enabled, threshold)` | nil | Enable/disable triplanar projection on the first active terrain. `enabled`: boolean. `threshold`: number in (0, 1] (default 0.7). Fragments with steep normals (abs(normal.y) < threshold) use triplanar sampling to reduce stretching. |
| `ffe.setTerrainLodDistances(lod1Distance, lod2Distance)` | nil | Set LOD transition distances on the first active terrain. `lod1Distance`: distance beyond which chunks render at half resolution. `lod2Distance`: distance beyond which chunks render at quarter resolution. Both must be positive numbers. Defaults: 100, 200. |
| `ffe.setTerrainStreamingRadius(radius)` | nil | Enable background chunk streaming on the first active terrain. `radius`: integer Chebyshev distance in chunk-grid coordinates (0 = disabled; all chunks remain eagerly loaded). Valid range: 0-32. Call once after `ffe.loadTerrain`. Default is 0 (streaming off) -- omitting this call is correct and keeps full eager loading, which is compatible with all existing demos. |
| `ffe.getTerrainChunkCount()` | integer | Return the number of terrain chunks currently resident on the GPU on the first active terrain. Returns 0 if no terrain is loaded. Useful for HUD debug display or verifying streaming behaviour in tests. |

### Common Usage Patterns

#### Load terrain from a PNG heightmap

```lua
function init()
    -- Load heightmap (grayscale PNG, up to 2048x2048)
    -- Args: path, worldWidth, worldDepth, heightScale
    local terrain = ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)
    if terrain == 0 then
        ffe.log("Failed to load terrain heightmap")
        return
    end

    -- Apply a grass texture
    local grassTex = ffe.loadTexture("terrain/grass.png")
    ffe.setTerrainTexture(grassTex)

    -- Set up camera and lighting
    ffe.setLightDirection(1, -2, 1)
    ffe.setLightColor(1, 0.95, 0.8)
    ffe.set3DCameraOrbit(128, 25, 128, 60, 30, 15)
end

function shutdown()
    ffe.unloadTerrain()
end
```

#### Place a player on the terrain surface

```lua
function init()
    ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)

    local playerMesh = ffe.loadMesh("models/character.glb")
    playerId = ffe.createEntity3D(playerMesh, 128, 0, 128)

    -- Snap player to terrain height
    local y = ffe.getTerrainHeight(128, 128)
    ffe.setTransform3D(playerId, 128, y, 128, 0, 0, 0, 1, 1, 1)
end

function update(entityId, dt)
    -- Keep player glued to terrain as they move
    local x, y, z = ffe.getTransform3D(playerId)
    local terrainY = ffe.getTerrainHeight(x, z)
    ffe.setTransform3D(playerId, x, terrainY, z)
end
```

#### Apply a different texture to the terrain (M1 single-texture)

```lua
function init()
    ffe.loadTerrain("terrain/mountain.png", 512, 512, 80)

    -- Switch between terrain textures based on biome
    local snowTex = ffe.loadTexture("terrain/snow.png")
    ffe.setTerrainTexture(snowTex)
end
```

#### Splat-map multi-texture terrain (M2)

```lua
function init()
    ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)

    -- Load the RGBA splat map (R=grass, G=dirt, B=rock, A=snow)
    local splatMap = ffe.loadTexture("terrain/splatmap.png")
    ffe.setTerrainSplatMap(splatMap)

    -- Assign a texture to each layer (0-3 = R/G/B/A channels)
    local grass = ffe.loadTexture("terrain/grass.png")
    local dirt  = ffe.loadTexture("terrain/dirt.png")
    local rock  = ffe.loadTexture("terrain/rock.png")
    local snow  = ffe.loadTexture("terrain/snow.png")

    ffe.setTerrainLayer(0, grass, 32)   -- grass tiles 32x across terrain
    ffe.setTerrainLayer(1, dirt,  24)   -- dirt tiles 24x
    ffe.setTerrainLayer(2, rock,  16)   -- rock tiles 16x (default)
    ffe.setTerrainLayer(3, snow,  20)   -- snow tiles 20x

    ffe.setLightDirection(1, -2, 1)
    ffe.set3DCameraOrbit(128, 25, 128, 60, 30, 15)
end

function shutdown()
    ffe.unloadTerrain()
end
```

#### Enable triplanar projection for cliffs

```lua
function init()
    ffe.loadTerrain("terrain/cliffs.png", 256, 256, 80)

    local splatMap = ffe.loadTexture("terrain/splatmap.png")
    ffe.setTerrainSplatMap(splatMap)

    ffe.setTerrainLayer(0, ffe.loadTexture("terrain/grass.png"), 32)
    ffe.setTerrainLayer(1, ffe.loadTexture("terrain/dirt.png"),  24)
    ffe.setTerrainLayer(2, ffe.loadTexture("terrain/rock.png"),  16)
    ffe.setTerrainLayer(3, ffe.loadTexture("terrain/snow.png"),  20)

    -- Enable triplanar on steep surfaces (normal.y < 0.5)
    -- This prevents texture stretching on cliff faces
    ffe.setTerrainTriplanar(true, 0.5)
end
```

#### Configure terrain LOD distances (M3)

```lua
function init()
    ffe.loadTerrain("terrain/heightmap.png", 512, 512, 60)

    -- Terrain chunks beyond 150 units from camera render at half resolution.
    -- Chunks beyond 300 units render at quarter resolution.
    -- Chunks inside the view frustum are automatically culled if fully outside.
    ffe.setTerrainLodDistances(150, 300)

    -- Combine with splat-map texturing
    local splatMap = ffe.loadTexture("terrain/splatmap.png")
    ffe.setTerrainSplatMap(splatMap)
    ffe.setTerrainLayer(0, ffe.loadTexture("terrain/grass.png"), 32)
    ffe.setTerrainLayer(1, ffe.loadTexture("terrain/rock.png"),  16)

    ffe.set3DCameraFPS(256, 30, 256, 0, -15)
end
```

#### Enable background chunk streaming for a large open-world terrain (M4)

By default, streaming is **disabled** and all chunks are loaded eagerly at `loadTerrain` time. This is the right choice for most scenes. For very large terrains where keeping all chunks in GPU memory simultaneously is not feasible, enable streaming after loading.

```lua
local chunkCountLabel  -- UI entity for debug display

function init()
    -- Load a large terrain (16x16 = 256 chunks at default TerrainConfig)
    local terrain = ffe.loadTerrain("terrain/world.png", 512, 512, 60)
    if terrain == 0 then
        ffe.log("Failed to load terrain")
        return
    end

    -- Enable streaming: keep chunks within 3 chunk-grid steps of the camera loaded.
    -- Chunks further away are evicted from GPU memory automatically.
    -- radius=0 (default) would disable streaming; 3 is a good starting value for
    -- a 64-vertex chunk resolution (each chunk ~ 32 world units wide on a 512-unit terrain).
    ffe.setTerrainStreamingRadius(3)

    -- Splat-map texturing works unchanged with streaming enabled
    local splatMap = ffe.loadTexture("terrain/splatmap.png")
    ffe.setTerrainSplatMap(splatMap)
    ffe.setTerrainLayer(0, ffe.loadTexture("terrain/grass.png"), 32)
    ffe.setTerrainLayer(1, ffe.loadTexture("terrain/rock.png"),  16)
    ffe.setTerrainLodDistances(150, 300)

    ffe.set3DCameraFPS(256, 30, 256, 0, -15)
end

function update(entityId, dt)
    -- Streaming ticks automatically each frame -- no per-frame call needed.
    -- You can query how many chunks are resident for debug display:
    local loaded = ffe.getTerrainChunkCount()
    -- e.g. update a HUD label: "Chunks: 49 / 256"
end

function shutdown()
    ffe.unloadTerrain()
end
```

**Key points:**
- `setTerrainStreamingRadius` is a cold-path call. Make it once in `init()`, not per frame.
- Streaming ticks automatically during the render pass each frame. No `update()` call is needed to drive it.
- `getTerrainChunkCount()` is O(chunkCount) -- call it at debug display frequency, not every frame in a tight loop.
- Omitting `setTerrainStreamingRadius` entirely (or passing 0) keeps all chunks eagerly loaded. This is the default and is backward-compatible with all existing demos.

### What NOT to Do

#### Do not call ffe.loadTerrain per frame

`loadTerrain` decodes a PNG, generates mesh geometry for all chunks, and uploads VAOs/VBOs/IBOs to the GPU. Calling it per frame will cause severe frame drops and GPU memory exhaustion.

```lua
-- BAD -- file I/O + mesh generation + GPU upload every frame
function update(entityId, dt)
    ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)  -- NEVER do this
end

-- GOOD -- load once at init
function init()
    ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)
end
```

#### Do not exceed MAX_TERRAIN_ASSETS (4)

Loading a 5th terrain when 4 are already loaded will fail and return 0. Unload unused terrains first with `ffe.unloadTerrain()`.

#### Do not use heightmaps larger than 2048x2048

Heightmaps exceeding `MAX_HEIGHTMAP_DIMENSION` (2048) in width or height are rejected. Files exceeding `MAX_HEIGHTMAP_FILE_BYTES` (16 MB) are also rejected before decoding begins. Use a smaller heightmap or split large worlds into multiple terrain tiles.

```lua
-- BAD -- 4096x4096 heightmap is rejected (exceeds 2048 limit)
ffe.loadTerrain("terrain/huge_4k.png", 256, 256, 50)  -- returns 0, error logged

-- GOOD -- use 1024x1024 or 2048x2048
ffe.loadTerrain("terrain/world_1024.png", 256, 256, 50)
```

#### Do not assume height outside terrain bounds

`getTerrainHeight` returns 0.0 for positions outside the terrain's world extents. If your game world extends beyond the terrain, check bounds or provide fallback logic.

```lua
-- Position (9999, 9999) is likely outside a 256x256 terrain
local y = ffe.getTerrainHeight(9999, 9999)  -- returns 0.0
```

#### Do not set splat-map layers without a splat map texture

Setting layers without a splat map has no visual effect. The terrain falls back to `MESH_BLINN_PHONG` (single texture or default color) unless a valid splat map is assigned.

```lua
-- BAD -- layers are set but no splat map, so they are ignored
ffe.setTerrainLayer(0, grassTex, 32)
ffe.setTerrainLayer(1, dirtTex, 24)

-- GOOD -- set the splat map first, then layers
ffe.setTerrainSplatMap(splatMap)
ffe.setTerrainLayer(0, grassTex, 32)
ffe.setTerrainLayer(1, dirtTex, 24)
```

#### Do not use layer indices outside 0-3

`setTerrainLayer` only accepts indices 0-3 (mapping to RGBA channels of the splat map). Values outside this range log an error and are ignored.

```lua
-- BAD -- index 4 is out of range
ffe.setTerrainLayer(4, rockTex, 16)  -- error logged, no effect

-- GOOD
ffe.setTerrainLayer(2, rockTex, 16)
```

#### Do not set LOD distances where lod1 > lod2

`setTerrainLodDistances` expects `lod2Distance >= lod1Distance`. If lod1 is larger, chunks will jump from full resolution directly to quarter, skipping the half-resolution LOD level entirely.

```lua
-- BAD -- lod1 (300) > lod2 (100), so LOD 1 is never used
ffe.setTerrainLodDistances(300, 100)

-- GOOD -- distances increase: full -> half at 150, half -> quarter at 300
ffe.setTerrainLodDistances(150, 300)
```

#### Do not call setTerrainSplatMap, setTerrainLayer, or setTerrainLodDistances per frame

These are cold-path functions that update material or LOD state. Call them once during `init()` or at scene-change boundaries, not every frame.

#### Do not call setTerrainStreamingRadius per frame

`setTerrainStreamingRadius` starts or reconfigures a background worker thread. Calling it every frame will restart the worker continuously, causing stalls and defeating the purpose of streaming.

```lua
-- BAD -- restarts the streaming worker every frame
function update(entityId, dt)
    ffe.setTerrainStreamingRadius(3)  -- NEVER do this
end

-- GOOD -- call once in init after loadTerrain
function init()
    ffe.loadTerrain("terrain/world.png", 512, 512, 60)
    ffe.setTerrainStreamingRadius(3)
end
```

#### Do not call setTerrainStreamingRadius before loadTerrain

`setTerrainStreamingRadius` operates on the first active terrain. If no terrain is loaded yet, the call logs an error and is silently ignored.

```lua
-- BAD -- no terrain is loaded yet
ffe.setTerrainStreamingRadius(3)
ffe.loadTerrain("terrain/world.png", 512, 512, 60)

-- GOOD
ffe.loadTerrain("terrain/world.png", 512, 512, 60)
ffe.setTerrainStreamingRadius(3)
```

#### Do not use getTerrainChunkCount in a tight per-frame loop for performance decisions

`getTerrainChunkCount` is O(chunkCount) and is intended for debug display and tests. Avoid calling it every frame in performance-critical code. A 256-chunk terrain means 256 state checks per call.

```lua
-- BAD -- O(256) check every frame just to gate logic
function update(entityId, dt)
    if ffe.getTerrainChunkCount() == 256 then
        doSomething()  -- avoid this pattern
    end
end

-- GOOD -- query at low frequency for debug HUD only
local debugTimer = 0
function update(entityId, dt)
    debugTimer = debugTimer + dt
    if debugTimer > 0.5 then
        debugTimer = 0
        local n = ffe.getTerrainChunkCount()
        -- update HUD label
    end
end
```

### Tier Support

| Tier | Supported |
|------|-----------|
| RETRO | No (requires OpenGL 3.3 core) |
| LEGACY | Yes (primary target -- OpenGL 3.3 core, MeshVertex layout, Blinn-Phong shader) |
| STANDARD | Yes |
| MODERN | Yes |

Each chunk is an independent VAO/VBO/IBO using the standard `MeshVertex` (32 bytes) layout. Terrain rendering uses the `TERRAIN` shader (splat-map mode with 4-layer blending and optional triplanar projection) when a splat map is assigned, or falls back to `MESH_BLINN_PHONG` for the M1 single-texture path. Each chunk has 3 LOD levels (full, half, quarter resolution) with distance-based selection, and chunks fully outside the camera frustum are culled before draw calls are issued. Background chunk streaming (M4) uses a single background worker thread for GPU upload scheduling; the OpenGL calls themselves still happen on the render thread (OpenGL context requirement). Streaming is compatible with all tiers from LEGACY upward.

| Dependency | What For |
|------------|----------|
| **RHI** (`renderer/rhi.h`) | GPU buffer and texture creation/destruction (VAO, VBO, IBO, texture units) |

## Vegetation System (`renderer/vegetation.h`, namespace `ffe::renderer`)

### System Purpose

GPU-instanced billboard grass and simple procedural tree system that populates a terrain surface with vegetation without per-frame heap allocation or CPU skinning.

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_VEGETATION_PATCHES` | `32` | Maximum simultaneous grass patches. Each patch allocates 1 VAO + 2 VBOs at `addPatch` time. |
| `MAX_GRASS_INSTANCES` | `256` | Maximum grass billboard instances per patch. `VegetationConfig::density` is clamped to `[1, 256]`. |
| `MAX_TREES` | `512` | Maximum trees in the scene across both trunk and crown instance buffers. |

### VegetationHandle

Opaque handle to a grass patch. Value 0 is always invalid. Same pattern as `TerrainHandle` and `MeshHandle`.

```cpp
struct VegetationHandle { u32 id = 0; };
bool isValid(VegetationHandle h);  // true if h.id != 0
```

### VegetationConfig struct

Per-patch grass configuration. All fields have defaults — construct and override only the fields you need.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `density` | `u32` | `64` | Instance count. Clamped to `[1, 256]` by `addPatch`. Passing `0` or a value > 256 is accepted — it will be silently clamped. |
| `grassHeight` | `float` | `0.5f` | World-space height of each billboard quad. |
| `grassWidth` | `float` | `0.3f` | World-space width of each billboard quad. |
| `fadeStartDist` | `float` | `20.0f` | Camera distance at which alpha fade begins. |
| `fadeEndDist` | `float` | `40.0f` | Camera distance at which grass is fully discarded (alpha-test). |
| `textureHandle` | `rhi::TextureHandle` | `{0}` | Grass diffuse+alpha texture. Pass `{0}` to use the built-in 4×4 solid-green fallback texture. |

### VegetationSystem Public API (C++)

The `VegetationSystem` is owned by `MeshRenderer` — game code does not construct or destroy it directly.

| Method | Description |
|--------|-------------|
| `void init()` | Allocate GPU buffers, create unit-cube tree meshes, generate fallback texture. Call once after OpenGL context creation. No heap allocation after this point. |
| `void shutdown()` | Free all GPU resources. Safe to call even if `init()` was never called. |
| `VegetationHandle addPatch(TerrainHandle terrain, const VegetationConfig& cfg)` | Scatter `cfg.density` grass instances across the terrain surface (Y positions queried via `getTerrainHeight`). Returns `VegetationHandle{0}` if `MAX_VEGETATION_PATCHES` is reached or terrain is invalid. **Cold path only — do not call per frame.** |
| `void removePatch(VegetationHandle handle)` | Remove a grass patch and free its GPU resources. No-op if handle is invalid. |
| `bool addTree(float x, float y, float z, float scale)` | Add a tree at world position `(x, y, z)`. Returns `false` silently if `MAX_TREES` is reached. Rebuilds GPU instance buffers (`glBufferData`). **Cold path only.** |
| `void clearTrees()` | Remove all trees and reset GPU instance buffers. Does not affect grass patches. |
| `void renderGrass(view, proj, cameraPos, fadeStartDist, fadeEndDist)` | Render all active patches. One `glDrawArraysInstanced` call per active patch. Alpha-test only, no blending, no sorting required. Called by `MeshRenderer` each frame — do not call directly. |
| `void renderTrees(view, proj, lightDir, lightColor, ambientColor)` | Render all trees as two instanced draw calls (trunk then crown). Uses `MESH_BLINN_PHONG_INSTANCED`. No-op if `treeCount == 0`. Called by `MeshRenderer` each frame — do not call directly. |
| `void setShaderHandles(vegetationShader, meshInstancedShader)` | Inject shader handles from `ShaderLibrary`. Must be called after `init()`. Called automatically by `MeshRenderer::init()`. |

### Lua Bindings

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.addVegetationPatch(terrainHandle, density, textureHandle)` | integer (patch handle) | Scatter grass across a terrain surface. `terrainHandle`: integer handle from `ffe.loadTerrain` (must be > 0). `density`: integer instance count; clamped to `[1, 256]` — passing 0 or a negative value is coerced to 1 silently. `textureHandle`: integer handle from `ffe.loadTexture`; pass `0` for the built-in solid-green fallback. Returns the patch handle (> 0) on success; returns `0` if `MAX_VEGETATION_PATCHES` (32) is reached or `terrainHandle` is invalid. |
| `ffe.removeVegetationPatch(handle)` | nil | Remove a grass patch and free its GPU resources. `handle`: integer returned by `addVegetationPatch`. No-op on invalid handle. |
| `ffe.addTree(x, y, z, scale)` | nil | Add a tree at world position `(x, y, z)` with uniform `scale`. The caller is responsible for vertical placement — call `ffe.getTerrainHeight(x, z)` first if placing on terrain. Negative or zero `scale` is clamped to `1.0`. Silently no-ops if `MAX_TREES` (512) is reached — there is no return value to detect overflow. |
| `ffe.clearTrees()` | nil | Remove all trees and reset GPU instance buffers. Does not affect grass patches. |

### Usage Example (Lua)

```lua
function init()
    -- Load terrain
    local terrain = ffe.loadTerrain("terrain/heightmap.png", 256, 256, 50)

    -- Scatter 128 grass instances using a loaded texture (or pass 0 for fallback)
    local grassTex = ffe.loadTexture("terrain/grass_blade.png")
    local grassPatch = ffe.addVegetationPatch(terrain, 128, grassTex)

    -- Place trees on the terrain surface
    for i = 1, 20 do
        local x = math.random(-100, 100)
        local z = math.random(-100, 100)
        local y = ffe.getTerrainHeight(x, z)
        ffe.addTree(x, y, z, 1.5)
    end
end

function shutdown()
    ffe.removeVegetationPatch(grassPatch)
    ffe.clearTrees()
    ffe.unloadTerrain()
end
```

### What NOT to Do

#### Do not call addVegetationPatch every frame

```lua
-- WRONG: allocates GPU buffers and scatters instances every frame
function update(entityId, dt)
    ffe.addVegetationPatch(terrain, 64, 0)  -- leaks patch slots
end

-- CORRECT: call once in init()
function init()
    grassPatch = ffe.addVegetationPatch(terrain, 64, 0)
end
```

#### Do not pass density > 256

```lua
-- WRONG: density 1000 is silently clamped to 256 — no error, no warning
local patch = ffe.addVegetationPatch(terrain, 1000, 0)

-- CORRECT: stay within the documented [1, 256] range
local patch = ffe.addVegetationPatch(terrain, 256, 0)
```

#### Do not call addTree per frame

```lua
-- WRONG: rebuilds GPU instance buffers every frame via glBufferData
function update(entityId, dt)
    ffe.addTree(x, y, z, 1.0)  -- accumulates until MAX_TREES silently drops calls
end

-- CORRECT: place all trees in init()
function init()
    ffe.addTree(10, ffe.getTerrainHeight(10, 10), 10, 1.0)
end
```

#### Do not assume addTree always succeeds

```lua
-- PROBLEMATIC: no return value indicates whether MAX_TREES (512) was hit
-- If you need to know the tree count, track it yourself:
local treeCount = 0
local MAX_TREES = 512

function addTreeSafe(x, y, z, scale)
    if treeCount >= MAX_TREES then
        ffe.log("Warning: tree limit reached")
        return
    end
    ffe.addTree(x, y, z, scale)
    treeCount = treeCount + 1
end
```

| Tier | Supported |
|------|-----------|
| RETRO | No (requires OpenGL 3.3 core) |
| LEGACY | Yes — primary target. GLSL 330 core, Y-axis cylindrical billboarding, alpha-test only (no blending required). |
| STANDARD | Yes |

## Frustum Utility (`renderer/frustum.h`, namespace `ffe::renderer`)

### System Purpose

Pure CPU math utility for view frustum extraction and axis-aligned bounding box (AABB) visibility testing. No GPU calls, no allocations. Currently used by the terrain renderer for per-chunk frustum culling, but the API is general-purpose and reusable by any system that needs to cull objects against a camera frustum (e.g., mesh renderers, particle systems, scene graphs).

**Thread safety:** Stateless functions operating on value types. Safe to call from any thread.

### Frustum struct

Six frustum planes extracted from a view-projection matrix. Each plane is stored as `vec4(a, b, c, d)` where `ax + by + cz + d = 0`. Normals point inward (into the frustum).

```cpp
struct Frustum {
    glm::vec4 planes[6]; // left, right, bottom, top, near, far
};
```

### Public API (C++)

| Function | Signature | Description |
|----------|-----------|-------------|
| `extractFrustum` | `Frustum extractFrustum(const glm::mat4& viewProjection)` | Extract 6 normalized frustum planes from a combined view-projection matrix. Uses the Griess-Hartmann method (row extraction from the VP matrix). Planes are normalized so the normal `(a, b, c)` has unit length, enabling correct distance calculations. |
| `isAABBVisible` | `bool isAABBVisible(const Frustum& frustum, const glm::vec3& aabbMin, const glm::vec3& aabbMax)` | Test if an AABB is at least partially inside the frustum. Returns `true` if visible (partially or fully inside), `false` if fully outside any single plane. Uses the "positive vertex" (p-vertex) method: for each plane, the AABB vertex most aligned with the plane normal is tested. If that vertex is behind the plane, the entire AABB is outside. |

### Common Usage Patterns

#### Cull objects against the camera frustum

```cpp
// Extract frustum from the current camera's VP matrix
const glm::mat4 vp = projectionMatrix * viewMatrix;
const ffe::renderer::Frustum frustum = ffe::renderer::extractFrustum(vp);

// Test each object's bounding box
for (const auto& object : objects) {
    if (ffe::renderer::isAABBVisible(frustum, object.aabbMin, object.aabbMax)) {
        // Object is visible -- submit draw call
        drawObject(object);
    }
    // Otherwise skip -- fully outside the frustum
}
```

### What NOT to Do

#### Do not skip plane normalization

`extractFrustum` normalizes planes internally. If you construct a `Frustum` manually without normalizing the plane normals to unit length, `isAABBVisible` distance tests will produce incorrect results.

#### Do not assume isAABBVisible is exact

The p-vertex test is conservative: it may return `true` for AABBs that are technically outside the frustum near the corners (false positives). It will never return `false` for an AABB that is actually visible (no false negatives). This is the standard trade-off for fast frustum culling.
