# engine/renderer -- System Context

## System Purpose

The renderer subsystem draws everything you see on screen. It provides a Render Hardware Interface (RHI) that abstracts OpenGL into simple create/bind/destroy calls for buffers, textures, and shaders. On top of the RHI, it provides a sprite batching system for efficient 2D rendering, a camera for view/projection math, a render queue that sorts draw commands to minimize GPU state changes, a shader library with built-in shaders, and ECS components (`Transform`, `Sprite`) plus a system (`renderPrepareSystem`) that automatically turns your entities into draw commands. If you want to put a sprite on screen, you create an entity with a `Transform` and a `Sprite` component, and the renderer handles the rest.

## Public API

### RHI Types (`renderer/rhi_types.h`, namespace `ffe::rhi`)

**Resource Handles (opaque IDs -- value 0 means invalid/null):**

| Type | Description |
|------|-------------|
| `BufferHandle` | Handle to a GPU buffer (vertex, index, or uniform). Has a `u32 id` field |
| `TextureHandle` | Handle to a GPU texture. Has a `u32 id` field |
| `ShaderHandle` | Handle to a compiled shader program. Has a `u32 id` field |

**Validity checks:**

```cpp
bool isValid(BufferHandle h);   // true if h.id != 0
bool isValid(TextureHandle h);  // true if h.id != 0
bool isValid(ShaderHandle h);   // true if h.id != 0
```

**RhiResult enum:**

| Value | Meaning |
|-------|---------|
| `OK` | Success |
| `ERROR_OUT_OF_MEMORY` | GPU or system memory exhausted |
| `ERROR_INVALID_HANDLE` | The handle does not refer to a valid resource |
| `ERROR_SHADER_COMPILE_FAILED` | Vertex or fragment shader failed to compile |
| `ERROR_SHADER_LINK_FAILED` | Shader program failed to link |
| `ERROR_TEXTURE_TOO_LARGE` | Texture dimensions exceed GPU limits |
| `ERROR_UNSUPPORTED` | Feature not available on this tier |
| `ERROR_CONTEXT_LOST` | OpenGL context was lost (rare) |

### RenderQueue (`renderer/render_queue.h`, namespace `ffe::renderer`)

A flat, pre-allocated array of draw commands sorted by a 64-bit key to minimize GPU state changes.

**DrawCommand struct (exactly 64 bytes -- one cache line):**

|-------|------|-------------|

| `shader` | `rhi::ShaderHandle` | Which shader to use |

| `vertexBuffer` | `rhi::BufferHandle` | Vertex data |
| `vertexOffset` | `u32` | Offset into vertex buffer |

## Common Usage Patterns

### 1. Draw a colored rectangle (no texture)

```cpp
ffe::EntityId box = world.createEntity();
world.addComponent<ffe::Transform>(box, glm::vec3{100.0f, 50.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(box);
sprite.size = {120.0f, 80.0f};           // 120x80 pixels
sprite.color = {0.2f, 0.6f, 1.0f, 1.0f}; // Light blue

// That's it. renderPrepareSystem picks it up automatically.
// No texture needed -- the engine uses a 1x1 white pixel as default.
```

### 2. Draw a textured sprite

```cpp
// Load a texture
ffe::rhi::TextureDesc texDesc;
texDesc.width = imgWidth;
texDesc.height = imgHeight;
texDesc.format = ffe::rhi::TextureFormat::RGBA8;
texDesc.filter = ffe::rhi::TextureFilter::NEAREST; // Pixel art? Use NEAREST
texDesc.pixelData = myPixelData; // Pointer to RGBA8 pixel data

ffe::rhi::TextureHandle myTex = ffe::rhi::createTexture(texDesc);

// Create entity
ffe::EntityId hero = world.createEntity();
world.addComponent<ffe::Transform>(hero, glm::vec3{0.0f, 0.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(hero);
sprite.texture = myTex;
sprite.size = {32.0f, 32.0f};
```

### 3. Use a sprite sheet (UV sub-regions)

```cpp
// Suppose your sprite sheet is 256x256 and each frame is 32x32.
// To show the frame at row 1, column 2 (zero-indexed):
auto& sprite = world.getComponent<ffe::Sprite>(entity);
sprite.uvMin = {2.0f * 32.0f / 256.0f, 1.0f * 32.0f / 256.0f}; // {0.25, 0.125}
sprite.uvMax = {3.0f * 32.0f / 256.0f, 2.0f * 32.0f / 256.0f}; // {0.375, 0.25}
```

### 4. Layer-based draw ordering

```cpp
// Background layer (draws first)
auto& bgSprite = world.addComponent<ffe::Sprite>(background);
bgSprite.layer = 0;

// Player (draws on top of background)
auto& playerSprite = world.addComponent<ffe::Sprite>(player);
playerSprite.layer = 5;

// UI (draws on top of everything)
auto& uiSprite = world.addComponent<ffe::Sprite>(healthBar);
uiSprite.layer = 10;

// Within the same layer, use sortOrder for fine control:
auto& bullet1 = world.addComponent<ffe::Sprite>(b1);
bullet1.layer = 5;
bullet1.sortOrder = 1; // Draws after player (sortOrder 0)
```

### 5. Creating a custom shader (advanced)

```cpp
ffe::rhi::ShaderDesc desc;
desc.vertexSource = myVertGLSL;
desc.fragmentSource = myFragGLSL;
desc.debugName = "custom_glow";

ffe::rhi::ShaderHandle glowShader = ffe::rhi::createShader(desc);
if (!ffe::rhi::isValid(glowShader)) {
    FFE_LOG_ERROR("Game", "Glow shader failed to compile!");
}

// Set uniforms after binding
ffe::rhi::bindShader(glowShader);
ffe::rhi::setUniformFloat(glowShader, "u_glowIntensity", 2.5f);
ffe::rhi::setUniformVec4(glowShader, "u_glowColor", {1.0f, 0.8f, 0.2f, 1.0f});

// Clean up when done
ffe::rhi::destroyShader(glowShader);
```

## What Not To Do

### Do not forget to destroy GPU resources

```cpp
// BAD -- GPU memory leak
ffe::rhi::TextureHandle tex = ffe::rhi::createTexture(desc);
// ... game ends, tex is never destroyed

// GOOD
ffe::rhi::destroyTexture(tex);
```

Every `createBuffer`, `createTexture`, `createShader` must have a matching `destroy` call.

### Do not set uniforms without binding the shader first

```cpp
// BAD -- undefined behaviour, sets uniform on wrong (or no) shader
ffe::rhi::setUniformFloat(myShader, "u_time", 1.0f);

// GOOD
ffe::rhi::bindShader(myShader);
ffe::rhi::setUniformFloat(myShader, "u_time", 1.0f);
```

### Do not create textures or buffers every frame

```cpp
// BAD -- creates a new GPU buffer every frame, leaks memory
void mySystem(ffe::World& world, float dt) {
    auto buf = ffe::rhi::createBuffer(desc); // WRONG -- this is a hot path
}

// GOOD -- create once at startup, update with updateBuffer if needed
```

### Do not ignore invalid handles

```cpp
// BAD -- using a handle that failed to create
ffe::rhi::ShaderHandle shader = ffe::rhi::createShader(desc);
ffe::rhi::bindShader(shader); // Might be id=0 if compile failed!

// GOOD
ffe::rhi::ShaderHandle shader = ffe::rhi::createShader(desc);
if (!ffe::rhi::isValid(shader)) {
    FFE_LOG_ERROR("Game", "Shader compile failed");
    return;
}
```

### Do not manually populate the RenderQueue for sprites

The `renderPrepareSystem` does this automatically for any entity with `Transform` + `Sprite`. You only need to touch the RenderQueue directly if you are building custom rendering beyond the sprite system.

### Do not set Sprite.size to zero

A sprite with `size = {0, 0}` produces degenerate geometry. Always set a non-zero size.

### Do not exceed 15 render layers

The sort key encodes layer in 4 bits (0-15). Layer values outside this range are clamped.

## Tier Support

The renderer currently supports **LEGACY** tier and above.

| Feature | RETRO | LEGACY | STANDARD | MODERN |
|---------|-------|--------|----------|--------|
| OpenGL backend | Not supported | OpenGL 3.3 core | OpenGL 4.5 | Vulkan (planned) |
| Sprite batching | N/A | 2,048 sprites/batch | 2,048 sprites/batch | 2,048 sprites/batch |
| Max draw commands | N/A | 8,192 | 32,768 | 131,072 |
| Shader version | N/A | GLSL 330 core | GLSL 450 core | SPIR-V (planned) |
| Texture formats | N/A | RGBA8, RGB8, R8, RGBA16F | All | All |
| Debug GL output | N/A | Optional (debug builds) | Optional | N/A |

All built-in shaders use `#version 330 core`, which is the LEGACY tier minimum. RETRO tier (OpenGL 2.1) is not supported by the current renderer.

| **Tracy** | Profiling zones in render queue sort/submit and render prepare system |

The renderer does NOT depend on: audio, physics, scripting, or any game-specific code.

## API Review Notes

This section documents naming inconsistencies, discoverability issues, and interface concerns found during the API review. These are recommendations for future improvement, not bugs.

### 1. Inconsistent function naming style for ShaderLibrary

The shader library uses `getShader()` as a free function, but the other "get" operations in the codebase are member functions (e.g., `World::getComponent`). Meanwhile, `initShaderLibrary` and `shutdownShaderLibrary` follow the free-function style of the RHI. This is internally consistent within the renderer, but the name `getShader` is generic and could collide with user code. Consider `getBuiltinShader` for clarity.

### 2. Transform and Sprite are defined in render_system.h, not in core

`Transform` is a fundamental component that physics, scripting, and gameplay systems will all need. It is currently defined in `renderer/render_system.h` inside namespace `ffe`, with a comment acknowledging it belongs in core. This means any system that needs `Transform` must include a renderer header, creating an unwanted dependency direction. Recommend moving `Transform` to `engine/core/components.h`.

### 3. Sprite.size is uninitialized by default

Every other field in `Sprite` has a sensible default. `size` does not, which means forgetting to set it produces garbage geometry. Recommend defaulting to `{1.0f, 1.0f}` or `{32.0f, 32.0f}`.

### 4. pushDrawCommand silently drops commands when full

When the render queue is full, `pushDrawCommand` silently does nothing. For a 16-year-old debugging why their sprites disappeared, this is invisible. Recommend logging a warning on the first dropped command per frame (not every one, to avoid log spam).

### 5. endFrame takes a raw GLFWwindow pointer

This leaks the windowing backend into the RHI interface. The RHI should ideally store the window reference internally (set during `init`) rather than requiring the caller to pass it every frame. This also makes it harder to add Vulkan or other backends later.

### 6. Mixed namespace organization for components

`Transform` and `Sprite` are in namespace `ffe` (not `ffe::renderer`), while all other renderer types are in `ffe::renderer` or `ffe::rhi`. This is intentional (they are ECS components, not renderer internals) but may confuse newcomers who see them defined in a renderer header. Moving them to core would fix both the namespace and the dependency issue.

### 7. Sort key bit layout is undocumented for users

`makeSortKey` is powerful but its bit layout (4 bits layer, 8 bits shader, 12 bits texture, 24 bits depth, 16 bits sub-order) is only visible by reading the implementation. Users who need custom sort keys have no documentation to guide them. The sort key constants should be named and documented.

### 8. No way to query whether RHI is initialized

There is no `rhi::isInitialized()` function. If user code tries to create a texture before `Application::run()` starts the renderer, they get undefined behaviour instead of a clear error.

## Texture Loading (`renderer/texture_loader.h`, namespace `ffe::renderer`)

Loads image files from disk and uploads them to the GPU as textures. This is a load-time API, not a per-frame API. All functions perform file I/O and/or GPU uploads.

### System Purpose

`texture_loader` bridges the filesystem and the RHI. It uses stb_image to decode PNG, JPG, BMP, and TGA files from disk (detected by magic bytes, not extension), always to RGBA8, then calls `rhi::createTexture()` to upload the pixel data to GPU memory. The result is a `TextureHandle` you can assign directly to a `Sprite` component or use with `rhi::bindTexture`.

Path traversal attacks are rejected before any syscall touches the filesystem. The asset root is set once at startup and is immutable -- it cannot be redirected by script or user input at runtime.

### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_TEXTURE_DIMENSION` | `8192` | Maximum width or height in pixels. Enforced before and after decoding |
| `MAX_TEXTURE_BYTES` | `256 MiB` (268,435,456) | Maximum decoded pixel data size. Prevents pathological images that pass per-dimension checks |
| `MAX_TEXTURE_FILE_BYTES` | `64 MiB` (67,108,864) | Maximum raw file size accepted before decoding begins |

### TextureLoadParams struct

Optional parameters for the `loadTexture` overloads that accept explicit control over sampling. All fields have defaults that match the original zero-parameter overloads.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `filter` | `ffe::rhi::TextureFilter` | `LINEAR` | Sampling filter. Use `NEAREST` for pixel art (hard pixels, no blurring). Use `LINEAR` for photographs or smooth art. |
| `wrap` | `ffe::rhi::TextureWrap` | `CLAMP_TO_EDGE` | Wrap mode. Use `CLAMP_TO_EDGE` for sprites (no edge repetition). Use `REPEAT` for tiling textures. |

### Public API

| Function | Signature | Description |
|----------|-----------|-------------|
| `setAssetRoot` | `bool setAssetRoot(const char* absolutePath)` | Set the root directory for all texture loads. Must be called once before `loadTexture`. `absolutePath` must be an absolute, existing directory path. **Write-once**: once set to a valid path, further calls are silently ignored and return false (this is intentional -- prevents runtime redirection by untrusted code). Returns true on success, false on failure with a logged error. |
| `getAssetRoot` | `const char* getAssetRoot()` | Query the current asset root. Returns an empty string if `setAssetRoot()` has not been called successfully. Useful for diagnostics and test assertions. |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path)` | Load an image file relative to the global asset root. Returns a valid `TextureHandle` (id != 0) on success. Returns `TextureHandle{0}` on failure and logs the specific reason. The caller owns the handle and must call `unloadTexture()` when done. Uses `LINEAR` filter and `CLAMP_TO_EDGE` wrap. |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path, const char* assetRoot)` | Load from a specific asset root without changing the global root. Useful when loading from multiple asset directories (e.g., engine assets vs game assets). `assetRoot` must be an absolute path to an existing directory. All path safety rules apply. Uses `LINEAR` filter and `CLAMP_TO_EDGE` wrap. |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path, const TextureLoadParams& params)` | Load relative to the global asset root with explicit filter and wrap control. Use this for pixel art (`NEAREST` filter) or tiling textures (`REPEAT` wrap). |
| `loadTexture` | `ffe::rhi::TextureHandle loadTexture(const char* path, const char* assetRoot, const TextureLoadParams& params)` | Load from a specific asset root with explicit filter and wrap control. |
| `unloadTexture` | `void unloadTexture(ffe::rhi::TextureHandle handle)` | Destroy a GPU texture created by `loadTexture`. Safe to call with an invalid handle (`handle.id == 0`) -- no-op. After this call, do not use the handle again. |

**Path rules for `loadTexture`:** The `path` parameter must be a relative path with no traversal sequences. The following are rejected:
- Paths starting with `/` or `\` (absolute Unix/Windows)
- Paths containing drive letters (`C:`)
- Paths containing `../`, `..\`, `/..`, `\..` (directory traversal)
- Paths at or beyond `PATH_MAX` characters

Rejection is logged via `FFE_LOG_ERROR` and returns `TextureHandle{0}`.

**Supported image formats:** PNG, JPG/JPEG, BMP, TGA. Format is detected from file content (magic bytes), not extension. All images are decoded to RGBA8 (4 channels).

### Common Usage Patterns

#### Load a texture at game startup and assign it to a sprite

```cpp
#include "renderer/texture_loader.h"
#include "renderer/render_system.h" // for Sprite component

// In Application::startup() -- once, before the game loop:
ffe::renderer::setAssetRoot("/home/user/mygame/assets/textures");

// Load a texture. Path is relative to the asset root.
ffe::rhi::TextureHandle heroTex = ffe::renderer::loadTexture("hero.png");

if (!ffe::rhi::isValid(heroTex)) {
    FFE_LOG_ERROR("Game", "Failed to load hero.png -- check asset root and path");
    return false; // or fall back to a placeholder
}

// Assign to a Sprite component.
ffe::EntityId hero = world.createEntity();
world.addComponent<ffe::Transform>(hero, glm::vec3{0.0f, 0.0f, 0.0f});

auto& sprite = world.addComponent<ffe::Sprite>(hero);
sprite.texture = heroTex;
sprite.size = {32.0f, 32.0f};

// At shutdown:
ffe::renderer::unloadTexture(heroTex);
```

#### Load a pixel-art texture (no blurring)

```cpp
// Pixel art must use NEAREST filter -- LINEAR blurs hard pixels.
ffe::renderer::TextureLoadParams pixelArtParams;
pixelArtParams.filter = ffe::rhi::TextureFilter::NEAREST;
pixelArtParams.wrap   = ffe::rhi::TextureWrap::CLAMP_TO_EDGE;

ffe::rhi::TextureHandle heroTex = ffe::renderer::loadTexture("hero.png", pixelArtParams);

// Or use brace initialisation (filter, wrap order matches struct declaration):
ffe::rhi::TextureHandle tileTex = ffe::renderer::loadTexture(
    "terrain/grass.png",
    {ffe::rhi::TextureFilter::NEAREST, ffe::rhi::TextureWrap::REPEAT}
);
```

#### Check for load failure and use a fallback

```cpp
// Try to load a custom skin; fall back to the default if it is missing.
ffe::rhi::TextureHandle skinTex = ffe::renderer::loadTexture("skins/custom.png");

if (!ffe::rhi::isValid(skinTex)) {
    // loadTexture already logged the error.
    // Fall back to the default skin loaded at startup.
    FFE_LOG_WARN("Game", "Custom skin not found -- using default");
    skinTex = defaultSkinHandle; // previously loaded at startup
}

auto& sprite = world.addComponent<ffe::Sprite>(entity);
sprite.texture = skinTex;
sprite.size = {32.0f, 48.0f};
```

### What NOT to Do

#### Do not call loadTexture per-frame

`loadTexture` calls `stat()`, `realpath()`, `fopen()`, `stbi_load()` (CPU decode), and `rhi::createTexture()` (GPU upload). Calling it inside a system update loop will cause severe frame drops and GPU memory leaks (each call allocates a new texture that you must separately destroy).

```cpp
// BAD -- creates a new GPU texture every frame
void mySystem(ffe::World& world, float dt) {
    auto tex = ffe::renderer::loadTexture("player.png");  // WRONG: file I/O + GPU upload per frame
    // also leaked -- unloadTexture never called
}

// GOOD -- load once at startup, store the handle, reuse it every frame
```

#### Do not forget to call setAssetRoot() before loadTexture

If `loadTexture(path)` (single-argument form) is called before `setAssetRoot()`, it returns `TextureHandle{0}` and logs an error. There is no default current-working-directory fallback. This is intentional -- silently loading from cwd creates unpredictable behaviour when the working directory changes.

```cpp
// BAD
ffe::rhi::TextureHandle tex = ffe::renderer::loadTexture("hero.png");
// Logs: "loadTexture: called before setAssetRoot() -- no asset root configured"
// Returns TextureHandle{0}.

// GOOD
ffe::renderer::setAssetRoot("/path/to/assets/textures");
ffe::rhi::TextureHandle tex = ffe::renderer::loadTexture("hero.png");
```

#### Do not use absolute paths or ../ in the path argument

Absolute paths and directory traversal sequences are rejected for security. Always provide a path relative to the asset root.

```cpp
// BAD -- all of these are rejected
ffe::renderer::loadTexture("/etc/passwd");          // absolute path
ffe::renderer::loadTexture("../../../etc/shadow");  // traversal
ffe::renderer::loadTexture("C:\\Windows\\win.ini"); // drive letter

// GOOD
ffe::renderer::loadTexture("enemies/goblin.png");   // relative, no traversal
ffe::renderer::loadTexture("ui/button.png");
```

#### Do not ignore the return value

`loadTexture` returns `TextureHandle{0}` on failure. Passing an invalid handle to `rhi::bindTexture` or assigning it to a `Sprite` is not a crash (the engine handles invalid handles gracefully), but your sprite will render with no texture (white rectangle from the default 1x1 white pixel). Always check `ffe::rhi::isValid()` after loading.

```cpp
// BAD -- ignores failure
auto tex = ffe::renderer::loadTexture("missing.png");
sprite.texture = tex; // tex.id may be 0 -- sprite renders white

// GOOD
auto tex = ffe::renderer::loadTexture("missing.png");
if (!ffe::rhi::isValid(tex)) {
    FFE_LOG_ERROR("Game", "Texture load failed");
    // Use fallback or abort
}
sprite.texture = tex;
```

### Security Note

`loadTexture` implements defence-in-depth path traversal protection:

1. **String-level check** (`isPathSafe`): Rejects absolute paths, drive letters, and `../` sequences before any syscall.
2. **Length check**: Rejects paths that would overflow the `PATH_MAX` concatenation buffer.
3. **Canonicalisation** (`realpath`): Resolves symlinks and encoded sequences, then verifies the canonical path still begins with the asset root. A symlink in the assets directory that points outside it is caught here.
4. **Prefix verification**: The canonical path must have the asset root as a prefix, with the root's final character followed by `/` or end-of-string.

This means you cannot load textures from outside the asset root directory, regardless of what path string you provide. This property is guaranteed even if the game loads texture paths from a level file or network packet.

The asset root is **write-once**: `setAssetRoot()` cannot be called a second time with a different path. This prevents a compromised script or malicious level file from redirecting texture loads to a different directory.

### Tier Support

| Tier | Supported |
|------|-----------|
| RETRO | Yes (stb_image decode is CPU-side and tier-independent; GPU upload delegates to `rhi::createTexture`) |
| LEGACY | Yes (primary target) |
| STANDARD | Yes |
| MODERN | Yes |

All loaded textures use RGBA8 format, which is supported on all tiers. The `NEAREST` vs `LINEAR` filter choice determines sampling quality. Pass a `TextureLoadParams` struct to `loadTexture` to select the filter and wrap mode — pixel art should use `NEAREST`, smooth art and photographs should use `LINEAR`.
