# engine/editor ‚Äî System Context

## System Purpose

The editor overlay is a debug-only ImGui interface compiled behind `#ifdef FFE_EDITOR` (Debug builds). It provides a real-time performance panel (FPS, frame time, entity count, audio voice count) and an entity inspector that lets developers select entities and edit their Transform components at runtime. This is an internal development tool, not a user-facing API.

---

## Public API

### EditorOverlay (`editor/editor.h`, namespace `ffe::editor`)

All code is guarded by `#ifdef FFE_EDITOR`. The class does not exist in Release builds.

| Method | Signature | Description |
|--------|-----------|-------------|
| `init` | `void init(GLFWwindow* window)` | Initialize ImGui context, install GLFW+OpenGL3 backends. Safe to call multiple times (no-ops after first). Must be called after GLFW window creation |
| `shutdown` | `void shutdown()` | Tear down ImGui backends and destroy the context. Safe to call if not initialised |
| `beginFrame` | `void beginFrame()` | Start a new ImGui frame. Call once per frame before `render()`. No-ops if not initialised |
| `render` | `void render(World& world)` | Draw all overlay panels (if visible) and submit ImGui draw data to OpenGL. Call once per frame after `beginFrame()` and after all game rendering |
| `toggle` | `void toggle()` | Toggle overlay visibility on/off. Typically bound to F1 |
| `isVisible` | `bool isVisible() const` | Returns `true` if the overlay is currently shown |
| `wantsMouse` | `bool wantsMouse() const` | Returns `true` if ImGui wants to capture mouse input (e.g., cursor is over an ImGui window). Use this to suppress game input when the overlay is active |
| `wantsKeyboard` | `bool wantsKeyboard() const` | Returns `true` if ImGui wants to capture keyboard input (e.g., a text field is focused). Use this to suppress game input when the overlay is active |
| `setShowHud` | `void setShowHud(bool show)` | Enable or disable HUD text rendering programmatically. When disabled, the HUD text window is not drawn even if `HudTextBuffer` contains text. Enabled by default. This is independent of the F1 inspector toggle (`m_visible`) |

### HUD Text Rendering

The editor overlay renders on-screen HUD text from the `HudTextBuffer` stored in the ECS registry context (see `engine/core/.context.md`). Key behaviours:

- **Position:** Top-center of the screen, 10 pixels from the top edge, horizontally centred.
- **Background:** Semi-transparent (60% opacity), auto-sized to fit the text content.
- **Always visible:** HUD rendering is **not** gated by `m_visible` (the F1 inspector toggle). The HUD is drawn even when the performance panel and entity inspector are hidden. This allows game scripts to display status text without requiring the full editor overlay.
- **Controlled by `m_showHud`:** The `setShowHud(bool)` method provides a programmatic way to disable HUD rendering entirely. When `m_showHud` is `false`, the HUD window is not drawn.
- **Data source:** Reads `HudTextBuffer` from `world.registry().ctx()` each frame. If the buffer is not present or `text[0]` is `'\0'`, nothing is drawn.
- **Written by Lua:** Game scripts set HUD text via `ffe.setHudText(text)`. See `engine/scripting/.context.md` for the Lua API.

### Panels (private, for reference)

| Panel | Description |
|-------|-------------|
| Performance | FPS (1-second rolling average), frame time in ms, live entity count, active audio voice count |
| Entity Inspector | Two-column layout: scrollable entity list (left) with editable Transform fields and read-only Sprite info (right) |
| Console | Scrollable log viewer showing recent engine log messages, color-coded by level. Reads from `LogRingBuffer` (see `engine/core/logging.h`). Auto-scrolls to newest. 256-entry ring buffer, pre-allocated |

## Common Usage Patterns

### 1. Integration in Application (init / beginFrame / render / shutdown cycle)

```cpp
#ifdef FFE_EDITOR
    ffe::editor::EditorOverlay m_editor;
#endif

// During startup, after GLFW window creation:
#ifdef FFE_EDITOR
    m_editor.init(window);
#endif

// In the main loop, once per frame:
#ifdef FFE_EDITOR
    m_editor.beginFrame();
#endif

// ... game systems tick, game rendering happens ...

// After game rendering, before glfwSwapBuffers:
#ifdef FFE_EDITOR
    m_editor.render(world);
#endif

// During shutdown:
#ifdef FFE_EDITOR
    m_editor.shutdown();
#endif
```

### 2. Toggle with F1

```cpp
if (ffe::isKeyPressed(ffe::Key::F1)) {
#ifdef FFE_EDITOR
    m_editor.toggle();
#endif
}
```

### 3. Suppressing game input when overlay is active

```cpp
#ifdef FFE_EDITOR
    const bool editorWantsMouse = m_editor.wantsMouse();
    const bool editorWantsKeyboard = m_editor.wantsKeyboard();
#else
    constexpr bool editorWantsMouse = false;
    constexpr bool editorWantsKeyboard = false;
#endif

if (!editorWantsMouse && ffe::isMouseButtonPressed(ffe::MouseButton::LEFT)) {
    // Handle game click
}
if (!editorWantsKeyboard && ffe::isKeyPressed(ffe::Key::SPACE)) {
    // Handle game action
}
```

## What Not To Do

### Do not call beginFrame or render without calling init first

Both methods silently no-op if the overlay is not initialised, but relying on this is a logic error. Always call `init()` during startup.

### Do not use EditorOverlay in headless mode

The overlay requires a live GLFW window and an OpenGL 3.3 context. Passing `nullptr` to `init()` or calling it without a valid GL context will crash ImGui's backend initialisation. In headless builds (`ApplicationConfig::headless = true`), do not instantiate or call the overlay.

### Do not call render before beginFrame in a given frame

ImGui requires `NewFrame()` before `Render()`. Calling `render()` without a preceding `beginFrame()` in the same frame is undefined behaviour in ImGui.

### Do not forget to guard all usage with `#ifdef FFE_EDITOR`

The `EditorOverlay` class does not exist in Release builds. Any unguarded reference will cause a compilation error.

### Do not render game content after the overlay render call

The overlay must be the last thing drawn before the buffer swap so that it appears on top of all game content.

# LLM Integration Panel ‚Äî System Context

*Phase 10 M3. File: `engine/editor/llm_panel.h` / `engine/editor/llm_panel.cpp`.*

---

## System Purpose

`LLMPanel` is an editor-only ImGui docked panel that gives developers an in-editor AI assistant without leaving the FFE standalone editor application. The panel sends the developer's natural-language question to an external OpenAI-compatible LLM endpoint, automatically assembling a system prompt from the FFE `.context.md` documentation files relevant to the currently selected entity. The LLM's response is displayed in a scrollable region, and the developer can inject it directly into the active script editor slot with a single button click.

This feature is strictly editor-only ‚Äî guarded by `#ifdef FFE_EDITOR` throughout. It adds zero runtime cost to shipped games. The LLM is always an external cloud endpoint; local inference is not supported. Credentials are stored in `~/.ffe/preferences.json` (mode `0600`), outside the project directory, so they cannot be accidentally committed to version control.

---

## Public API

Header: `engine/editor/llm_panel.h`
Namespace: `ffe::editor`
Guard: `#ifdef FFE_EDITOR`

### `LLMConfig` struct

Populated from `~/.ffe/preferences.json` at editor startup. Pass to `LLMPanel::setConfig()`.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `baseUrl` | `std::string` | `""` | LLM endpoint base URL. **Must begin with `https://`** ‚Äî requests to `http://` URLs are rejected. Example: `"https://api.openai.com"` |
| `apiKey` | `std::string` | `""` | Bearer token. Never logged. Passed as an HTTP `Authorization: Bearer` header only |
| `model` | `std::string` | `"gpt-4o"` | Model identifier. Set by editing `~/.ffe/preferences.json` directly; no UI picker in M3 |
| `timeoutSecs` | `int` | `30` | HTTP request timeout in seconds |
| `maxResponseBytes` | `size_t` | `32768` | Hard cap on response body (32 KB). Clamped to `[512, 65536]` when loaded from preferences |
| `caBundle` | `std::string` | `""` | Path to a custom CA certificate bundle. Empty = system default CA store. If the path is non-empty but fails `stat()` validation, the request proceeds with the system store and a warning is logged |

### `ScriptEditorSlot` interface

Inherit this in the concrete script editor implementation to allow `LLMPanel` to inject text:

| Method | Signature | Description |
|--------|-----------|-------------|
| `appendText` | `virtual void appendText(std::string_view text) = 0` | Appends text at the end of the editor buffer. Does not execute the code |
| `replaceSelection` | `virtual void replaceSelection(std::string_view text) = 0` | Replaces the current selection with text. Falls back to `appendText` if no selection exists |

Virtual dispatch on `ScriptEditorSlot` is acceptable because it is called only on an explicit user button click ‚Äî never in the render hot path.

### `LLMPanel` class

Non-copyable, non-movable (owns thread state).

| Method | Signature | Description |
|--------|-----------|-------------|
| `setConfig` | `void setConfig(LLMConfig cfg)` | Sets the LLM endpoint configuration. Must not be called while `isRequestPending() == true` |
| `setContextFiles` | `void setContextFiles(std::vector<std::string> paths)` | Sets the list of `.context.md` file paths to include in the next request's system prompt. The editor shell calls this each frame after resolving the selected entity's component set. Files are read at Submit time, not when this is called. Must not be called while `isRequestPending() == true` |
| `setScriptSlot` | `void setScriptSlot(ScriptEditorSlot* slot)` | Registers the active script editor slot for Lua snippet injection. Pass `nullptr` to hide the Insert button. Must not be called while `isRequestPending() == true` |
| `render` | `void render()` | Renders the LLM panel into the current ImGui dockspace. Call once per frame, inside the ImGui frame (after `beginFrame()`, before buffer swap) |
| `isRequestPending` | `bool isRequestPending() const` | Returns `true` if a background HTTP request is in flight. The Submit button is greyed out in this state |

### Context-file selection mapping

The editor shell determines which `.context.md` files to pass to `setContextFiles()` based on the selected entity's component set. `LLMPanel` does not inspect the ECS directly.

| Editor state | Files passed (priority order) |
|---|---|
| Nothing selected | `engine/core/.context.md` |
| 2D entity (Sprite / Tilemap / Particle) | `engine/core/.context.md`, `engine/renderer/.context.md` |
| 3D entity (Transform3D / Mesh / Material3D) | `engine/core/.context.md`, `engine/renderer/.context.md` |
| Entity with Collider2D | `engine/core/.context.md`, `engine/physics/.context.md` |
| Entity with networking component | `engine/core/.context.md`, `engine/networking/.context.md` |
| Script editor active (any entity) | All matching files above + `engine/scripting/.context.md` appended last |
| Multiple conditions | All matching files, deduplicated, in the order above |

Files are assembled into a system prompt capped at **4096 characters**. Each file is consumed in priority order and truncated at the last complete line that fits. The user's message is not counted against this budget.

## Lua Bindings (editor-only)

**These bindings are registered only in editor builds (`#ifdef FFE_EDITOR`). They are NOT present in `ffe_runtime` or any shipped game binary. Do not rely on them in game scripts.**

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.isLLMConfigured()` | `boolean` | Returns `true` if both `baseUrl` and `apiKey` are non-empty in the loaded preferences. Check this before calling `ffe.llmQuery` to avoid an error return |
| `ffe.llmQuery(prompt)` | `string\|nil, string\|nil` | Sends a synchronous LLM request on the Lua coroutine thread (not the render thread). Returns `responseText, nil` on success, or `nil, errorMsg` on failure. Returns `nil, "LLM not configured"` if no API key is set. The 1M-instruction budget is suspended during the HTTP wait. Timeout is **5 seconds** (independent of `LLMConfig.timeoutSecs`) |

Registration site: `engine/scripting/script_engine.cpp`, inside an `#ifdef FFE_EDITOR` block. The block is not compiled into any non-editor target.

Rationale: registering unrestricted outbound network bindings in `ffe_runtime` would allow any game script to exfiltrate data or make arbitrary HTTP calls, violating the sandbox guarantee (CLAUDE.md ¬ß5).

## Common Usage Patterns

### 1. Basic setup ‚Äî create panel, configure, wire into editor render loop

```cpp
#ifdef FFE_EDITOR
#include "editor/llm_panel.h"

// In your editor application class:
ffe::editor::LLMPanel m_llmPanel;

// During editor init (after preferences are loaded):
ffe::editor::LLMConfig cfg;
cfg.baseUrl       = prefs.llm.baseUrl;      // from ~/.ffe/preferences.json
cfg.apiKey        = prefs.llm.apiKey;
cfg.model         = prefs.llm.model;        // default "gpt-4o"
cfg.timeoutSecs   = prefs.llm.timeoutSecs;  // default 30
cfg.maxResponseBytes = prefs.llm.maxResponseBytes; // default 32768
m_llmPanel.setConfig(cfg);

// In the main loop, once per frame (inside ImGui frame):
m_llmPanel.render();
#endif
```

### 2. Context-aware query ‚Äî pass relevant `.context.md` files based on selection

```cpp
#ifdef FFE_EDITOR
// Resolve context files from the currently selected entity each frame.
// Do this before calling render() so the next Submit uses fresh context.
if (!m_llmPanel.isRequestPending()) {
    std::vector<std::string> contextPaths;
    contextPaths.push_back(projectRoot + "/engine/core/.context.md");

    if (selectedEntity.has<ffe::Mesh>() || selectedEntity.has<ffe::Transform3D>()) {
        contextPaths.push_back(projectRoot + "/engine/renderer/.context.md");
    }
    if (selectedEntity.has<ffe::Collider2D>()) {
        contextPaths.push_back(projectRoot + "/engine/physics/.context.md");
    }
    if (scriptEditorIsActive) {
        contextPaths.push_back(projectRoot + "/engine/scripting/.context.md");
    }

    m_llmPanel.setContextFiles(contextPaths);
}
#endif
```

The panel reads these files only when the user clicks Submit ‚Äî not every frame. The call to `setContextFiles` is cheap (stores the path list; no I/O).

### 3. Script insertion ‚Äî wire up a `ScriptEditorSlot` for the Insert button

```cpp
#ifdef FFE_EDITOR
// Your concrete script editor must implement ScriptEditorSlot:
class MyScriptEditor : public ffe::editor::ScriptEditorSlot {
public:
    void appendText(std::string_view text) override {
        // Append to your ImGui InputTextMultiline buffer.
        m_buffer += text;
    }
    void replaceSelection(std::string_view text) override {
        // Replace the selected range in your text buffer,
        // or fall back to appendText if no selection exists.
        if (m_selectionStart == m_selectionEnd) {
            appendText(text);
        } else {
            m_buffer.replace(m_selectionStart, m_selectionEnd - m_selectionStart, text);
        }
    }
private:
    std::string m_buffer;
    size_t m_selectionStart = 0;
    size_t m_selectionEnd   = 0;
};

MyScriptEditor m_scriptEditor;

// Wire the slot into the LLM panel once at init:
m_llmPanel.setScriptSlot(&m_scriptEditor);

// The user clicks Submit -> LLM responds -> user clicks Insert.
// LLMPanel calls m_scriptEditor.replaceSelection(responseText).
// No code is executed. The user must click Run in the script editor separately.
#endif
```

## What Not To Do

### Never log or store the API key

`LLMConfig::apiKey` must never appear in log output, stdout, stderr, error messages, or debug dumps ‚Äî not even the first four characters. `LLMResponse::errorMsg` is scrubbed before storage: any occurrence of the key value in an error string is replaced with `"[REDACTED]"`. If you add log statements anywhere near LLM request code, do not include `cfg.apiKey` or any field derived from it.

### Never commit `~/.ffe/preferences.json`

The preferences file lives outside the project root (`~/.ffe/preferences.json`) precisely so that it cannot be accidentally committed to version control. Do not copy it into the project directory, reference it in `.gitignore`, or add it to any build artifact. If a developer asks why their API key isn't working, check that the file is at `~/.ffe/preferences.json` ‚Äî not inside the project.

### Never call `render()` while checking `isRequestPending()` as a loop

`render()` contains the only per-frame check of the pending flag (`m_pending.load()`). Do not write a busy-wait or polling loop around `isRequestPending()` ‚Äî the render loop already polls it correctly. One call to `render()` per frame is the entire async coordination mechanism.

### Never auto-inject LLM responses

The Insert button requires an explicit user click. Do not call `m_scriptSlot->replaceSelection()` or `m_scriptSlot->appendText()` from any path other than the Insert button handler inside `render()`. Do not add keyboard shortcuts, timers, or automatic injection on request completion.

### Never disable TLS or allow `http://` endpoints

`LLMPanel` rejects any `baseUrl` that does not begin with `https://` (returns an error without making an HTTP call). Do not add a config flag or UI control to bypass certificate verification. The `caBundle` field exists for corporate proxy CAs ‚Äî not for disabling TLS.

### Never call `setConfig()`, `setContextFiles()`, or `setScriptSlot()` while a request is pending

These methods mutate panel state that the background thread reads. Calling them while `isRequestPending() == true` is a data race. Always guard with `if (!m_llmPanel.isRequestPending())` before mutating configuration.

### Lua LLM bindings are editor-only ‚Äî do not expect them in game builds

`ffe.llmQuery` and `ffe.isLLMConfigured` are not registered in `ffe_runtime`. Calling them from a game script will raise a Lua error ("attempt to call a nil value"). Always check `ffe.isLLMConfigured()` first, and only use these bindings in editor-hosted scripts (e.g., level-editing automation), not in shipping game logic.

## Tier Support

`LLMPanel` is **editor-only** and has no GPU dependency. All rendering is ImGui CPU-side rasterisation. The background HTTP thread is CPU and network I/O bound; it does not compete with the GPU timeline.

| Tier | Supported | Notes |
|------|-----------|-------|
| RETRO | No | Editor requires OpenGL 3.3 (ImGui backend) |
| LEGACY | Yes (editor minimum) | No GPU dependency for the LLM panel itself |
| STANDARD | Yes | |
| MODERN | Yes | |

The panel degrades gracefully on slow or unavailable network connections: the "Waiting..." indicator spins until the request completes or `timeoutSecs` fires. The render loop is never blocked.
