# engine/physics -- 2D Collision Detection Subsystem

AI-native context file. Written by api-designer.

---

## System Purpose

The physics subsystem provides 2D overlap detection for entities that have both a `Transform` and a `Collider2D` component. It supports two shapes (axis-aligned bounding box and circle), 16-bit layer/mask filtering, and trigger-only colliders. The system runs as an ECS system (`collisionSystem`) at priority 200 (after gameplay/scripting systems at 100-199), writes a per-frame `CollisionEventList` into the ECS registry context, and performs zero heap allocations -- all temporary storage comes from the per-frame `ArenaAllocator`.

This is **not** a physics simulation. There is no rigid-body dynamics, no forces, no friction, no restitution. The subsystem answers one question per frame: "which pairs of collider entities are overlapping?" Game code (C++ systems or Lua scripts) reads the resulting events and decides what to do.

## Public API

### Components and Types

Header: `engine/physics/collider2d.h`
Namespace: `ffe`

#### `ColliderShape` (enum class : u8)

| Value | Description |
|-------|-------------|
| `AABB` (0) | Axis-aligned bounding box. Parameterised by `halfWidth` and `halfHeight`. |
| `CIRCLE` (1) | Circle. Radius is stored in `halfWidth`; `halfHeight` is ignored. |

#### `Collider2D` (struct, 16 bytes POD)

Attached to entities alongside `Transform`. The collider is centered on the entity's `Transform` position (no local offset).

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `shape` | `ColliderShape` | `AABB` | Shape type. |
| `isTrigger` | `bool` | `false` | If true, generates overlap events but does not imply physics response. Game code decides what "trigger" means. |
| `layer` | `u16` | `0xFFFF` | Which collision layer this entity belongs to. 16-bit mask gives 16 layers. |
| `mask` | `u16` | `0xFFFF` | Which layers this entity tests against. |
| `halfWidth` | `f32` | `0.0` | AABB half-extent on X axis, or circle radius for `CIRCLE` shapes. |
| `halfHeight` | `f32` | `0.0` | AABB half-extent on Y axis. Ignored for `CIRCLE` shapes. |

`static_assert(sizeof(Collider2D) == 16)` -- guaranteed 16 bytes for cache-friendly iteration.

#### `CollisionEvent` (struct, 8 bytes)

Generated per frame for each overlapping pair. Arena-allocated, valid for the current frame only.

| Field | Type | Description |
|-------|------|-------------|
| `entityA` | `EntityId` | First entity in the pair. Always `entityA < entityB` (canonical ordering). |
| `entityB` | `EntityId` | Second entity in the pair. |

#### `CollisionEventList` (struct)

Stored in the ECS registry context by `collisionSystem`. Points to an arena-allocated array. Valid for the current frame only -- the arena is reset before the next frame.

| Field | Type | Description |
|-------|------|-------------|
| `events` | `CollisionEvent*` | Pointer to arena-allocated array of events. `nullptr` if no collisions. |
| `count` | `u32` | Number of events in the array. |

#### `CollisionCallbackRef` (struct)

Stored in the ECS registry context. Holds a Lua registry reference to the collision callback function. Used by `ScriptEngine::deliverCollisionEvents()` to invoke the Lua callback after all systems have run.

| Field | Type | Description |
|-------|------|-------------|
| `luaRef` | `int` | Lua registry reference. `-2` (LUA_NOREF) means no callback is registered. |

### Narrow-Phase Overlap Functions

Header: `engine/physics/narrow_phase.h`
Namespace: `ffe`

All functions are `inline`. No sqrt. Minimal branching. Positions are world-space centers from `Transform.position.x/y`.

| Function | Signature | Description |
|----------|-----------|-------------|
| `overlapAABB` | `bool overlapAABB(f32 posAx, f32 posAy, f32 halfAw, f32 halfAh, f32 posBx, f32 posBy, f32 halfBw, f32 halfBh)` | AABB vs AABB separating-axis test. |
| `overlapCircle` | `bool overlapCircle(f32 posAx, f32 posAy, f32 radiusA, f32 posBx, f32 posBy, f32 radiusB)` | Circle vs Circle using squared distance (no sqrt). |
| `overlapAABBCircle` | `bool overlapAABBCircle(f32 aabbPosX, f32 aabbPosY, f32 aabbHalfW, f32 aabbHalfH, f32 circlePosX, f32 circlePosY, f32 circleRadius)` | AABB vs Circle. Clamps circle center to AABB bounds, then distance check. |

### Collision System

Header: `engine/physics/collision_system.h`
Namespace: `ffe`

| Symbol | Type | Description |
|--------|------|-------------|
| `collisionSystem` | `void collisionSystem(World& world, float dt)` | ECS system function. Detects overlaps between all entities with `Transform` + `Collider2D`. Writes `CollisionEventList` to the ECS registry context. |
| `COLLISION_SYSTEM_PRIORITY` | `inline constexpr i32` (200) | Priority at which `collisionSystem` should be registered. Must run after gameplay/scripting systems (100-199). |

**Internal pipeline (per frame):**

1. Collect all entities with `Transform` + `Collider2D`.
2. Build a spatial hash grid (arena-allocated, 1024 buckets, 128-unit cell size) for broad-phase culling.
3. Generate candidate pairs from overlapping grid cells.
4. Sort and deduplicate pairs (canonical ordering: `entityA < entityB`).
5. Filter by layer/mask: `(A.layer & B.mask) != 0 && (B.layer & A.mask) != 0`.
6. Run narrow-phase overlap tests (AABB-AABB, Circle-Circle, AABB-Circle).
7. Write `CollisionEventList` to the ECS registry context.

## Common Usage Patterns

### 1. Add a collider to an entity (C++)

```cpp
#include "physics/collider2d.h"
#include "renderer/render_system.h" // Transform

// Create an entity with a Transform and an AABB collider.
auto entity = world.createEntity();
world.addComponent<Transform>(entity, Transform{
    .position = {100.0f, 200.0f, 0.0f},
    .scale    = {1.0f, 1.0f, 1.0f},
    .rotation = 0.0f
});
world.addComponent<Collider2D>(entity, Collider2D{
    .shape      = ColliderShape::AABB,
    .isTrigger  = false,
    .layer      = 0x0001,  // layer 0
    .mask       = 0xFFFF,  // collides with everything
    .halfWidth  = 16.0f,
    .halfHeight = 16.0f
});

// Circle collider (radius 12):
world.addComponent<Collider2D>(entity, Collider2D{
    .shape      = ColliderShape::CIRCLE,
    .halfWidth  = 12.0f,   // radius
    .halfHeight = 0.0f     // ignored for circles
});
```

### 2. Register the collision system

```cpp
#include "physics/collision_system.h"

// Register at the correct priority (200 = physics band).
world.addSystem("collisionSystem", ffe::collisionSystem, ffe::COLLISION_SYSTEM_PRIORITY);
```

### 3. Layer/mask filtering

Layers are 16-bit masks. Two entities A and B generate a collision event only if both directions pass:

```
(A.layer & B.mask) != 0  AND  (B.layer & A.mask) != 0
```

**Example -- player collides with pickups but not with other players:**

```cpp
// Player: layer bit 0, collides with layer bit 1 (pickups)
Collider2D playerCol{};
playerCol.layer = 0x0001;  // bit 0
playerCol.mask  = 0x0002;  // only collide with bit 1

// Pickup: layer bit 1, collides with layer bit 0 (players)
Collider2D pickupCol{};
pickupCol.layer = 0x0002;  // bit 1
pickupCol.mask  = 0x0001;  // only collide with bit 0
pickupCol.isTrigger = true;
```

### 4. Read collision events from C++

```cpp
// After world.tick() has run collisionSystem:
if (world.registry().ctx().contains<ffe::CollisionEventList>()) {
    const auto& events = world.registry().ctx().get<ffe::CollisionEventList>();
    for (u32 i = 0; i < events.count; ++i) {
        EntityId a = events.events[i].entityA;
        EntityId b = events.events[i].entityB;
        // Handle collision: destroy pickup, apply damage, play sound, etc.
    }
}
```

**Important:** The `events` pointer is arena-allocated and valid only until the arena is reset (before the next frame). Do not store the pointer across frames.

### 5. Handle collision events from Lua

```lua
-- Register a collision callback (called once per overlapping pair per frame).
ffe.setCollisionCallback(function(entityA, entityB)
    ffe.log("Collision: " .. tostring(entityA) .. " <-> " .. tostring(entityB))
    -- Respond: destroy one entity, play a sound, increment score, etc.
end)
```

The callback is invoked by `ScriptEngine::deliverCollisionEvents()` after all systems (including `collisionSystem`) have run. See the scripting `.context.md` for full Lua API details.

## What NOT to Do

### Do not heap-allocate in the collision system or its hot path

The collision system uses the per-frame `ArenaAllocator` exclusively. All spatial hash nodes, candidate pair buffers, and event arrays are arena-allocated. Do not add `new`, `delete`, `malloc`, `std::vector::push_back` (without `reserve`), or `std::function` to any code path inside `collisionSystem`. This is a hard rule from the engine constitution (Section 3).

### Do not call narrow-phase functions directly from Lua scripts

The narrow-phase overlap functions (`overlapAABB`, `overlapCircle`, `overlapAABBCircle`) are C++ inline functions intended for internal use by `collisionSystem`. They are not exposed to Lua. Game scripts should attach `Collider2D` components to entities and register a collision callback via `ffe.setCollisionCallback()` -- the engine handles the rest.

### Do not store CollisionEvent pointers across frames

`CollisionEventList.events` points to arena-allocated memory. The arena is reset before the next frame. Any pointer into the event array becomes dangling after the arena reset. Read the events within the same frame they were generated.

### Do not modify Collider2D components during collisionSystem execution

The collision system iterates over all `Collider2D` components during its run. Adding, removing, or modifying colliders from within a system that runs concurrently or at the same priority as `collisionSystem` produces undefined iteration behaviour. Modify colliders in gameplay systems (priority < 200) or in post-collision handlers (after `collisionSystem` has finished).

### Do not use layer 0x0000

Setting `layer = 0x0000` means the entity belongs to no layer. The bidirectional check `(A.layer & B.mask)` will always be 0, so the entity will never generate collision events regardless of other entities' masks. This is almost certainly a mistake. If you want an entity to not collide with anything, remove the `Collider2D` component instead.

### Do not expect physics response (push-back, bounce, friction)

This subsystem is overlap detection only. There is no rigid-body solver. If your game needs push-back (e.g., preventing a player from walking through walls), implement it in your gameplay system by reading collision events and adjusting positions manually.

## Tier Support

| Tier | Supported | Notes |
|------|-----------|-------|
| RETRO | Yes | CPU-only subsystem. No GPU dependency. |
| LEGACY | Yes | Primary target. Single-threaded spatial hash is well within budget for hundreds of entities. |
| STANDARD | Yes | Same implementation. |
| MODERN | Yes | Same implementation. Future: multi-threaded broad phase for thousands of entities. |

The collision system is CPU-only and has no GPU, OpenGL, or Vulkan dependency. Performance scales with entity count (broad phase is O(n) insert, narrow phase is proportional to overlapping pairs). The spatial hash grid (1024 buckets, 128-unit cells) is tuned for typical 2D game entity counts (tens to hundreds).
