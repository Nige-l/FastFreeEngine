## System Purpose

The scripting subsystem provides a sandboxed Lua scripting environment for game logic. It wraps LuaJIT via the raw Lua C API (sol2 is a declared dependency but not yet used in this file -- the raw C API is used directly). One `ScriptEngine` instance manages one Lua state per game context.

**What "sandboxed" means for game developers:** Scripts you load through `ScriptEngine` cannot read or write files, make network connections, or call OS functions. The engine blocks all of those libraries at init time, before your scripts ever run. Scripts also have a hard instruction budget of 1,000,000 instructions per call -- if a script enters an infinite loop, it raises a Lua error (caught by the engine), not a crash. From a game developer's perspective this means: scripts are safe to run even if they came from an untrusted source (a mod, a level file, a config).

Game behaviour belongs in Lua; engine internals stay in C++. The scripting layer is the primary interface for writing game logic against FFE without touching C++.

## Public API (C++)

Header: `engine/scripting/script_engine.h`
Namespace: `ffe`
Class: `ffe::ScriptEngine`

`ScriptEngine` is non-copyable and non-movable. One instance per game context. All methods must be called from the main thread.

| Method | Signature | Description |
|--------|-----------|-------------|
| `init` | `bool init()` | Create and sandbox the Lua state. Must be called before any other method. Returns false and logs if initialisation fails (out of memory). Safe to call once only -- a warning is logged if called twice. |
| `shutdown` | `void shutdown()` | Close the Lua state and free all resources. Before closing the state, calls the Lua global `shutdown()` function if it is defined (no arguments). If that callback errors, the error is logged and shutdown proceeds normally. If `shutdown` is not defined in Lua, this is a silent no-op. Safe to call if `init()` was never called or failed. Also called automatically by the destructor. |
| `setWorld` | `void setWorld(ffe::World* world)` | Sets the `World*` for ECS binding access. Call after `init()`, before any `doString`/`doFile`. Passing `nullptr` clears the stored world pointer. The World must remain valid for the lifetime of this ScriptEngine. Not callable from Lua -- C++ only. |
| `doString` | `bool doString(const char* script)` | Compile and execute a Lua script from a null-terminated string. Returns false and logs the error if compilation or execution fails. Errors are caught -- this call cannot crash the engine. |
| `doFile` | `bool doFile(const char* path, const char* assetRoot)` | Load and execute a Lua script from a file. `path` must be relative (no leading `/`, no `../` components). `assetRoot` must be a non-empty absolute directory path. The full path is constructed as `assetRoot + "/" + path`. Returns false and logs on failure. |
| `callFunction` | `bool callFunction(const char* funcName, ffe::i64 entityId, double dt)` | Call a named global Lua function with `(entityId, dt)` arguments. This is the correct way to invoke per-frame Lua callbacks — it uses `lua_getglobal` + `lua_pcall` directly, avoiding the compile step that `doString` runs every call. Returns false and logs if the function does not exist or errors. Requires `init()` and `setWorld()` to have been called first. |
| `isInitialised` | `bool isInitialised() const` | Returns true if `init()` has been called successfully and `shutdown()` has not yet been called. Use this to guard calls in conditional startup paths. |

## Lua Standard Libraries Available to Scripts

Scripts have access to a carefully chosen subset of the standard library. The engine opens only what is safe:

| Library | Available | Notes |
|---------|-----------|-------|
| `math` | Yes | Full math library |
| `string` | Yes | Full string library |
| `table` | Yes | Full table library |
| `base` (print, type, pairs, ipairs, tostring, tonumber, error, assert, pcall, xpcall, select, unpack, next) | Partial | Dangerous globals removed -- see Security section |
| `io` | **No** | Blocked entirely |
| `os` | **No** | Blocked entirely |
| `package` / `require` | **No** | Blocked entirely |
| `debug` | **No** | Blocked entirely |
| `ffi` | **No** | LuaJIT FFI blocked entirely |
| `jit` | **No** | LuaJIT JIT control blocked entirely |

## FFE Lua API (ffe.* table)

The engine installs an `ffe` global table in the Lua state with bindings to engine systems.

### Logging

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.log(message)` | nothing | Logs a message through the FFE logging system at INFO level, tagged with category "Lua". This is the only way scripts can emit diagnostic output -- scripts cannot write to stdout or files directly. |

### Engine Control

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.requestShutdown()` | nothing | Signals the engine to stop the game loop cleanly after the current tick completes. Sets `ShutdownSignal::requested = true` in the ECS registry context. Application checks this flag after every tick. Use this instead of `os.exit()` (which is blocked) or any other escape. |

```lua
-- ESC to quit cleanly from Lua:
if ffe.isKeyPressed(ffe.KEY_ESCAPE) then
    ffe.requestShutdown()
end
```

### Script Lifecycle Callbacks (optional global functions)

Scripts may define these global functions. The engine calls them automatically at the appropriate lifecycle points.

| Function | Called by C++ | Arguments | Description |
|----------|--------------|-----------|-------------|
| `update(entityId, dt)` | `ScriptEngine::callFunction("update", ...)` | entityId, dt | Per-frame update. Called by game code via `callFunction`. |
| `shutdown()` | `ScriptEngine::shutdown()` | none | Called once before the Lua state is closed. Use this to unload textures and clean up other resources. Errors are caught and logged — they do not prevent engine shutdown. |

### Input Queries (safe for per-frame use from Lua)

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.isKeyHeld(keyCode)` | bool | True while the key is held. `keyCode` is an integer -- use `ffe.KEY_*` constants. |
| `ffe.isKeyPressed(keyCode)` | bool | True on the frame the key first goes down. |
| `ffe.isKeyReleased(keyCode)` | bool | True on the frame the key goes up. |
| `ffe.getMouseX()` | number | Mouse X position in window pixels. |
| `ffe.getMouseY()` | number | Mouse Y position in window pixels. |

**Key constants** -- values match GLFW key codes (integers):

```lua
ffe.KEY_W, ffe.KEY_A, ffe.KEY_S, ffe.KEY_D  -- WASD
ffe.KEY_M                                    -- M key (music toggle)
ffe.KEY_SPACE, ffe.KEY_ESCAPE, ffe.KEY_ENTER
ffe.KEY_UP, ffe.KEY_DOWN, ffe.KEY_LEFT, ffe.KEY_RIGHT
```

Out-of-range key codes return `false` without error.

### Entity Lifecycle

Requires `setWorld()` to have been called from C++ before any script runs. If no World is registered, `createEntity` returns `nil` and the other bindings return `false` or are no-ops.

**entityId** is a Lua integer (the raw `u32` value of `EntityId`). Negative integers and values above `UINT32_MAX - 1` are always rejected.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.createEntity()` | integer or nil | Creates a new entity in the registered World. Returns an integer entity ID on success, `nil` if no World is registered or the ECS is out of entity slots. |
| `ffe.destroyEntity(entityId)` | nothing | Destroys an entity and all its components. No-op if `entityId` is negative, out of range, or no longer valid (already destroyed). |
| `ffe.addTransform(entityId, x, y, rotation, scaleX, scaleY)` | bool | Adds a `Transform` component to the entity. If the entity already has a `Transform`, it is overwritten (a warning is logged). Returns `false` if the entity ID is invalid or out of range, or if any float argument is non-finite (NaN or Infinity). Returns `true` on success. All arguments after `entityId` are numbers (floats). |
| `ffe.addSprite(entityId, texHandle, width, height, r, g, b, a, layer)` | bool | Adds a `Sprite` component to the entity. If the entity already has a `Sprite`, it is overwritten (a warning is logged). Returns `false` if the entity ID is invalid, `texHandle` is `<= 0`, or `width`/`height` is non-finite or `<= 0`. Color values `r`, `g`, `b`, `a` are clamped to `[0.0, 1.0]` (NaN is treated as `0.0`). `layer` is clamped to `[0, 15]`. `texHandle` is an opaque integer — the `u32 id` field of a `rhi::TextureHandle` created on the C++ side. |
| `ffe.addPreviousTransform(entityId)` | bool | Opts the entity in to render interpolation by adding a `PreviousTransform` component. If the entity has a `Transform`, `PreviousTransform` is initialised from it. If no `Transform` exists, `PreviousTransform` is initialised to zero position/unit scale and a warning is logged — call `addTransform` first. Returns `false` if the entity ID is invalid or out of range. |

**Overwrite behaviour:** `addTransform`, `addSprite`, and `addPreviousTransform` all use `emplace_or_replace` internally — calling them on an entity that already has the component overwrites the existing component and logs a warning. This is safe but is an indication of a logic error in most game code.

### Texture Lifecycle

These functions load and unload GPU textures from Lua scripts. The C++ `setAssetRoot()` (or equivalent asset root configuration) must be called before any texture can be loaded -- if no asset root is configured, `ffe.loadTexture` returns `nil`.

Texture handles returned by `ffe.loadTexture` are opaque integers -- the `u32 id` field of a `rhi::TextureHandle`. Pass them directly to `ffe.addSprite`. Do not modify or fabricate handle values.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.loadTexture(path)` | integer or nil | Load a texture file relative to the C++-configured global asset root. `path` must be a string -- passing a non-string returns `nil` immediately without logging. Returns an integer handle id on success, `nil` on failure. Failure reasons are logged by the C++ layer (file not found, unsupported format, GPU upload failure). The C++ `setAssetRoot()` must be called before this function works. |
| `ffe.unloadTexture(handle)` | nothing | Destroy a GPU texture created by `ffe.loadTexture`. Safe no-op for out-of-range handles (0, negative, or above `UINT32_MAX`). After calling this, do not pass the handle to `ffe.addSprite` -- the underlying GPU texture no longer exists. |

**Texture lifecycle example -- load at scene start, use for sprites, unload at teardown:**

```lua
-- scene_init.lua
-- Called once when the scene loads, NOT per frame.

local playerTex = ffe.loadTexture("sprites/player.png")
if playerTex == nil then
    ffe.log("ERROR: failed to load player texture -- sprites will be invisible")
    return
end

local enemyTex = ffe.loadTexture("sprites/enemy.png")

-- Create entities and attach sprites using the loaded handles.
local player = ffe.createEntity()
ffe.addTransform(player, 100.0, 200.0, 0.0, 1.0, 1.0)
ffe.addSprite(player, playerTex, 32.0, 32.0, 1.0, 1.0, 1.0, 1.0, 0)
ffe.addPreviousTransform(player)

local enemy = ffe.createEntity()
ffe.addTransform(enemy, 400.0, 300.0, 0.0, 1.0, 1.0)
ffe.addSprite(enemy, enemyTex, 32.0, 32.0, 1.0, 0.2, 0.2, 1.0, 0)

-- Store handles for later cleanup.
_G.sceneTextures = { playerTex, enemyTex }
_G.playerEntityId = player
_G.enemyEntityId  = enemy
```

```lua
-- scene_teardown.lua
-- Called once when the scene unloads.
-- Destroy sprite entities BEFORE unloading their textures.

if _G.playerEntityId then ffe.destroyEntity(_G.playerEntityId) end
if _G.enemyEntityId  then ffe.destroyEntity(_G.enemyEntityId)  end

-- Now safe to release GPU textures.
if _G.sceneTextures then
    for _, tex in ipairs(_G.sceneTextures) do
        ffe.unloadTexture(tex)
    end
    _G.sceneTextures = nil
end
```

### Music Playback

These functions wrap `ffe::audio` music playback. One music track at a time. Music volume is independent of the sound-effects master volume. All functions are no-ops in headless mode (CI) or when audio is unavailable.

Music track handles can be loaded from Lua using `ffe.loadSound(path)` (see Sound Effects section below) or from C++ via `ffe::audio::loadSound()` and passed to the script as an integer global.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.playMusic(soundHandle, loop)` | nothing | Start playing a music track. `soundHandle` is the integer from `ffe::audio::SoundHandle::id` (loaded in C++). `loop` is boolean (default `true` if omitted). If a track is already playing, it stops first. No-op if handle is invalid or audio is unavailable. |
| `ffe.stopMusic()` | nothing | Stop the current music track immediately. No-op if no music is playing. |
| `ffe.setMusicVolume(volume)` | nothing | Set music volume `[0.0, 1.0]`. Clamped; NaN/Inf treated as 0.0. |
| `ffe.getMusicVolume()` | number | Query current music volume `[0.0, 1.0]`. Safe to call per-frame (reads an atomic). |
| `ffe.isMusicPlaying()` | boolean | Returns true if a music track is active. Safe to call per-frame (reads an atomic). |

**Key constants added for music controls:**

```lua
ffe.KEY_M     -- M key (76) — common music toggle binding
-- (ffe.KEY_UP / ffe.KEY_DOWN also available for volume control)
```

**Example — load music from Lua, start looping, and add toggle controls:**

```lua
-- game.lua
-- Load the music track at script load time (once).
local musicHandle = ffe.loadSound("music/theme.ogg")
local musicStarted = false

function update(entityId, dt)
    -- Start music on first tick
    if not musicStarted and musicHandle then
        ffe.playMusic(musicHandle, true)  -- loop = true
        ffe.setMusicVolume(0.5)
        musicStarted = true
    end

    -- M key: toggle music
    if ffe.isKeyPressed(ffe.KEY_M) then
        if ffe.isMusicPlaying() then
            ffe.stopMusic()
        else
            ffe.playMusic(musicHandle, true)
        end
    end
end

function shutdown()
    ffe.stopMusic()
    if musicHandle then
        ffe.unloadSound(musicHandle)
        musicHandle = nil
    end
end
```

**Alternative — pass the handle from C++ (still supported):**

```lua
-- C++ side sets g_musicHandle as a Lua global before loading game.lua:
--   ffe::audio::SoundHandle h = ffe::audio::loadSound("music/theme.ogg", audioRoot);
--   lua_pushinteger(L, h.id);
--   lua_setglobal(L, "g_musicHandle");

-- Lua side can then use g_musicHandle directly with ffe.playMusic().
```

### Sound Effects

These functions load, unload, and play one-shot sound effects from Lua. Sound handles follow the same security model as texture handles: path traversal is rejected, the asset root is controlled by C++ (not scriptable), and non-string arguments to `loadSound` return `nil` immediately.

All functions are no-ops in headless mode (CI) or when audio is unavailable.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.loadSound(path)` | integer or nil | Load a WAV or OGG audio file relative to the C++-configured asset root. `path` must be a string -- passing a non-string returns `nil` immediately and logs an error. Returns an integer handle on success, `nil` on failure. Failure reasons are logged by the C++ audio layer. Same security model as `ffe.loadTexture`: explicit type guard (rejects numbers/booleans silently coerced to strings), path traversal rejection (no `../`, no absolute paths), and asset root controlled exclusively by C++. |
| `ffe.unloadSound(handle)` | nothing | Unload a previously loaded sound and free its decoded PCM buffer. `handle` must be a positive integer in `(0, UINT32_MAX]`. Out-of-range handles (0, negative, or above `UINT32_MAX`) are logged as errors and ignored. After calling this, do not pass the handle to `ffe.playSound` -- the underlying audio data no longer exists. |
| `ffe.playSound(handle [, volume])` | nothing | Play a one-shot sound effect. `handle` is an integer returned by `ffe.loadSound`. Optional `volume` argument (default `1.0`) controls playback volume. Invalid handles are logged as errors and ignored. |
| `ffe.setMasterVolume(volume)` | nothing | Set the master volume for all audio output. `volume` is a number in `[0.0, 1.0]`. Values are passed directly to the C++ audio layer which clamps them. |

**Sound lifecycle example -- load at scene start, play during gameplay, unload at teardown:**

```lua
-- scene_init.lua (called once when the scene loads)
local jumpSfx  = ffe.loadSound("sfx/jump.wav")
local coinSfx  = ffe.loadSound("sfx/coin.ogg")

if jumpSfx == nil then
    ffe.log("WARNING: failed to load jump sound effect")
end

-- Store handles for later use.
_G.sceneSounds = { jump = jumpSfx, coin = coinSfx }
```

```lua
-- game_logic.lua (per-frame update)
function update(entityId, dt)
    if ffe.isKeyPressed(ffe.KEY_SPACE) and _G.sceneSounds.jump then
        ffe.playSound(_G.sceneSounds.jump, 0.8)  -- play at 80% volume
    end

    -- Collect a coin → play coin sound at full volume
    if coinCollected and _G.sceneSounds.coin then
        ffe.playSound(_G.sceneSounds.coin)  -- default volume 1.0
    end
end
```

```lua
-- scene_teardown.lua (called once when the scene unloads)
if _G.sceneSounds then
    for _, handle in pairs(_G.sceneSounds) do
        if handle then ffe.unloadSound(handle) end
    end
    _G.sceneSounds = nil
end
```

```lua
-- Volume control from Lua
ffe.setMasterVolume(0.5)  -- set master volume to 50%
```

### ECS Transform Access

Requires `setWorld()` to have been called from C++ before any script runs. If no World is registered, `getTransform` returns `nil` and `setTransform` is a silent no-op.

**entityId** is a Lua integer (the raw `u32` value of `EntityId`). Pass it from C++ via `static_cast<lua_Integer>(entityId)`.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.getTransform(entityId)` | table or nil | Returns `{x, y, z, scaleX, scaleY, scaleZ, rotation}` for a valid entity that has a Transform component. Returns `nil` if the entity ID is invalid, the entity has no Transform, or no World is registered. **Allocates a new Lua table on every call** -- use `fillTransform` in hot loops to avoid GC pressure. |
| `ffe.fillTransform(entityId, table)` | boolean | Zero-allocation alternative to `getTransform`. Writes `x`, `y`, `rotation`, `scaleX`, `scaleY` into a caller-provided table. Returns `true` on success, `false` if the entity ID is invalid, the entity has no Transform, or no World is registered. The second argument must be a table (raises a Lua error if not). **Use this in per-frame update loops** -- pre-allocate the table once and reuse it every frame to eliminate GC pressure. |
| `ffe.setTransform(entityId, x, y, rotation, scaleX, scaleY)` | nothing | Sets the transform fields on the entity. No-op for invalid entity or missing component. Rejects NaN and Infinity -- they are logged as errors and discarded. Note: `z` and `scaleZ` are not settable from Lua in this API; they retain their existing values. |

## Common Usage Patterns

### 1. Engine startup -- initialise ScriptEngine, register World, and load a game script

```cpp
#include "scripting/script_engine.h"

// In Application::startup() or equivalent:
ffe::ScriptEngine scriptEngine;

if (!scriptEngine.init()) {
    FFE_LOG_ERROR("Game", "Failed to initialise script engine -- aborting");
    return false;
}

// Register the ECS World so Lua scripts can read/write transforms.
// world must remain valid for the lifetime of scriptEngine.
scriptEngine.setWorld(&world);

// Load the main game script from the assets directory.
// path is relative; assetRoot is the absolute scripts directory.
const char* assetRoot = "/home/user/mygame/assets/scripts";
if (!scriptEngine.doFile("main.lua", assetRoot)) {
    FFE_LOG_ERROR("Game", "Failed to load main.lua");
    // Decide whether this is fatal. doFile logs the specific error.
}
```

### 2. doString for in-game events (tooltip text, dialogue, NPC responses)

`doString` is appropriate for short, known-safe Lua fragments generated at load time -- not per-frame. A common pattern is evaluating dialogue strings or trigger conditions that were authored in a data file and loaded into memory at scene load.

```cpp
// At scene load time -- NOT per-frame.
// For example, evaluating a trigger condition stored as a string in a level file.
const char* triggerScript = "if playerHealth < 20 then ffe.log('Low health warning') end";

if (!scriptEngine.doString(triggerScript)) {
    // doString already logged the error. Decide whether to skip or abort.
    FFE_LOG_WARN("Game", "Trigger script failed -- skipping trigger");
}
```

Do NOT use `doString` to run scripts that were provided at runtime by an end user without validation -- the sandbox protects the OS, but the engine still spends CPU compiling and running the script.

### 3. Lua-side ffe.log usage (what game devs write in their scripts)

Game scripts use `ffe.log` to emit diagnostic output. The message appears in the engine log under the "Lua" category at INFO level.

```lua
-- main.lua
ffe.log("main.lua loaded successfully")

local function onEnemySpawned(enemyId)
    ffe.log("Enemy spawned: " .. tostring(enemyId))
end

-- ffe.log accepts any value coerced to string by tostring():
local score = 42
ffe.log("Current score: " .. tostring(score))
```

### 4. Error handling -- what to do when doString or doFile returns false

Both `doString` and `doFile` return `false` on failure and log the specific Lua error via `FFE_LOG_ERROR`. Your code only needs to decide what to do next -- you do not need to re-read the error yourself.

```cpp
// Example: load optional config script; fall back to defaults if it fails.
const bool ok = scriptEngine.doFile("config.lua", assetRoot);
if (!ok) {
    // The engine already logged: "Lua compile error: ..." or "Lua runtime error: ..."
    // Apply hardcoded defaults instead.
    FFE_LOG_WARN("Game", "config.lua failed to load -- using built-in defaults");
    applyDefaultConfig();
}

// Example: treat a missing required script as a fatal error.
if (!scriptEngine.doFile("game_rules.lua", assetRoot)) {
    FFE_LOG_ERROR("Game", "game_rules.lua is required -- cannot start");
    return false;
}
```

### 5. Per-frame Lua callbacks using callFunction

Define a named function in Lua at load time, then call it every frame from a C++ system using `callFunction`. This is the correct, zero-compile-overhead pattern for per-frame game logic.

**Lua side (loaded once at startup via doFile):**

```lua
-- game_logic.lua
local SPEED = 200.0

-- Pre-allocate a reusable table ONCE at load time (outside the function).
-- fillTransform writes into this table every frame — zero GC pressure.
local t = {}

function update(entityId, dt)
    if not ffe.fillTransform(entityId, t) then return end

    local dx = 0.0
    local dy = 0.0

    if ffe.isKeyHeld(ffe.KEY_D) then dx = dx + SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_A) then dx = dx - SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_S) then dy = dy + SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_W) then dy = dy - SPEED * dt end

    ffe.setTransform(entityId, t.x + dx, t.y + dy, t.rotation, t.scaleX, t.scaleY)

    if ffe.isKeyPressed(ffe.KEY_ESCAPE) then
        ffe.requestShutdown()
    end
end
```

**C++ side (called per frame from a system):**

```cpp
// Load the script once at startup:
scriptEngine.doFile("game_logic.lua", assetRoot);

// In a C++ system or per-frame update:
// callFunction looks up "update" by name (lua_getglobal), pushes the
// two arguments, and calls lua_pcall. No string construction or compilation.
scriptEngine.callFunction("update", static_cast<ffe::i64>(playerEntity), dt);
```

**Key rule:** `doFile` compiles the function body once. `callFunction` only looks it up and invokes it. The ratio is: 1× `doFile` at load time, N× `callFunction` at 60 fps. Never call `doString("update(...)")` per frame.

### 6. Creating an entity from Lua with Transform, Sprite, and interpolation

This pattern shows how to spawn a game entity entirely from a Lua script. The texture handle is an opaque integer obtained from C++ and passed into the script as an argument or a Lua global set before the script runs.

**C++ side — pass the texture handle to the script as a global:**

```cpp
// After loading the script with doFile, set a global the script can read.
// texHandle.id is the u32 from rhi::TextureHandle.
lua_State* L = ...;  // do not do this from outside the scripting subsystem in production code
// Preferred: expose a C++ binding (e.g., ffe.loadTexture) that returns the handle.
// For this example, set a known handle as a Lua global before calling the spawn function.
scriptEngine.doString("PLAYER_TEX = 1");  // handle 1 is the white default texture

// Load the script that defines spawnPlayer():
scriptEngine.doFile("spawn.lua", assetRoot);

// Spawn the player by calling a named Lua function.
// The entity ID produced is returned via callFunction's return path if needed,
// or stored in a Lua global by the script itself.
scriptEngine.callFunction("spawnPlayer", 0 /*unused entity arg*/, 0.0 /*dt*/);
```

**Lua side (spawn.lua):**

```lua
-- spawn.lua
-- PLAYER_TEX must be set by C++ before this script runs.

local function spawnPlayer()
    -- Create the entity. Returns nil if the ECS is full.
    local id = ffe.createEntity()
    if id == nil then
        ffe.log("spawnPlayer: ECS is full — could not create player entity")
        return
    end

    -- Add a Transform at position (100, 200), no rotation, scale 1x1.
    local ok = ffe.addTransform(id, 100.0, 200.0, 0.0, 1.0, 1.0)
    if not ok then
        ffe.log("spawnPlayer: addTransform failed for entity " .. tostring(id))
        ffe.destroyEntity(id)
        return
    end

    -- Add a Sprite using the texture handle set by C++.
    -- Width=32, height=32, white tint (r=1,g=1,b=1,a=1), layer 0.
    ok = ffe.addSprite(id, PLAYER_TEX, 32.0, 32.0, 1.0, 1.0, 1.0, 1.0, 0)
    if not ok then
        ffe.log("spawnPlayer: addSprite failed for entity " .. tostring(id))
        ffe.destroyEntity(id)
        return
    end

    -- Opt in to render interpolation. Call this AFTER addTransform so that
    -- PreviousTransform is initialised from the Transform we just set.
    ok = ffe.addPreviousTransform(id)
    if not ok then
        ffe.log("spawnPlayer: addPreviousTransform failed for entity " .. tostring(id))
        -- Not fatal — entity still renders; interpolation just won't work.
    end

    ffe.log("spawnPlayer: created entity " .. tostring(id))
    -- Store in a module-level variable for later use.
    _G.playerEntityId = id
end

-- Top-level call: this runs when callFunction("spawnPlayer", ...) is invoked.
spawnPlayer()
```

**Key sequencing rule:** always call `addTransform` before `addPreviousTransform`. If `addPreviousTransform` runs first, it initialises to zero position and logs a warning, so the first interpolated frame will have an incorrect starting position.

### 7. Using the shutdown() callback to release resources before the Lua state closes

Define a global `shutdown` function in your script to clean up textures and other resources when the engine exits. `ScriptEngine::shutdown()` calls it automatically — you do not need to invoke it yourself.

```lua
-- game_logic.lua
-- Loaded once at startup via doFile.

local playerTex = ffe.loadTexture("sprites/player.png")

function update(entityId, dt)
    -- ... game logic ...
end

function shutdown()
    -- Called by ScriptEngine::shutdown() before the Lua state is closed.
    -- Use this to unload textures and release any other resources your script owns.
    if playerTex then
        ffe.unloadTexture(playerTex)
        playerTex = nil
    end
    ffe.log("Script shutdown complete")
end
```

`shutdown()` is optional. If it is not defined, `ScriptEngine::shutdown()` proceeds silently. If `shutdown()` raises a Lua error, the error is logged and the engine continues to close the Lua state — the error does not abort shutdown.

## What NOT to Do

### Never call doString() or doFile() per-frame or per-entity

Both functions compile Lua bytecode and execute it. Calling them inside a system update -- even once per frame -- will destroy performance. They are not lightweight function calls; `doString` calls `luaL_loadstring` (compiler) followed by `lua_pcall` (executor). Call them at load time, in response to one-shot events, or during scene transitions. Never inside a hot loop.

```cpp
// BAD -- called per frame: doString compiles Lua bytecode every call
void mySystem(ffe::World& world, float dt) {
    scriptEngine.doString("updateAI()");  // WRONG: compiles every frame
}

// GOOD -- define the function once at startup; call it per frame via callFunction
// At startup:
scriptEngine.doFile("ai.lua", assetRoot);   // compiles "function updateAI(...)" once

// Per frame in a C++ system:
scriptEngine.callFunction("updateAI", static_cast<ffe::i64>(entityId), dt);
// callFunction uses lua_getglobal + lua_pcall — no compile step, no string construction.
```

### Do not call ffe.loadTexture() per frame

`ffe.loadTexture` triggers file I/O on disk and a GPU texture upload. Calling it per frame is a severe performance violation: one disk read + GPU upload every 16 ms will exhaust GPU memory, stall the render pipeline, and drop frame rate to unusable levels on any hardware tier. Load textures once at scene or application startup, keep the integer handle, and reuse it.

```lua
-- BAD -- called every frame inside an update function
function update(entityId, dt)
    local tex = ffe.loadTexture("sprites/player.png")  -- WRONG: file I/O + GPU upload every frame
    ffe.addSprite(entityId, tex, 32, 32, 1, 1, 1, 1, 0)
end

-- GOOD -- loaded once at startup; handle reused per frame
-- At scene init (called once):
_G.playerTex = ffe.loadTexture("sprites/player.png")

-- Per event (not per frame):
ffe.addSprite(entityId, _G.playerTex, 32, 32, 1, 1, 1, 1, 0)
```

### Do not open additional Lua libraries (io, os, package, debug, ffi, jit)

The sandbox is constructed by opening only safe libraries. If you open `io`, `os`, or `package` after `init()`, scripts gain access to the filesystem, OS commands, and `require`. This defeats the entire security model and breaks the promise to game developers that their scripts are sandboxed. The instruction budget also does not protect against file I/O blocking.

```cpp
// BAD -- opens os library, breaks sandbox
lua_State* L = ...; // do not extract the Lua state
luaopen_os(L);      // scripts can now call os.execute("rm -rf /")

// GOOD -- do not touch the Lua state directly; use only ScriptEngine's public API
```

(The Lua state is stored as `void*` in the header intentionally -- you are not meant to cast it back and call raw Lua C API functions from outside the scripting subsystem.)

### Do not expose raw pointers to Lua

If you add bindings that hand C++ objects to Lua scripts, never pass a raw pointer. A script could hold a stale pointer after the object is freed, producing use-after-free. Use opaque integer handles (like `EntityId`) that the engine validates before use.

```lua
-- BAD (hypothetical binding that passes raw pointer)
local entity = ffe.getRawPointer(42)  -- would be a use-after-free risk

-- GOOD (opaque handle -- engine validates before use)
local entity = ffe.getEntity(42)  -- engine checks validity before touching memory
```

### Do not call doString() or doFile() before init()

Calling either function before `init()` returns `false` immediately and logs an error. The Lua state does not exist until `init()` succeeds.

```cpp
// BAD
ffe::ScriptEngine engine;
engine.doFile("main.lua", assetRoot);  // init() never called -- logs error, returns false

// GOOD
ffe::ScriptEngine engine;
if (!engine.init()) { return false; }
engine.doFile("main.lua", assetRoot);
```

### Do not call setWorld() before init()

`setWorld()` requires an initialised Lua state. Calling it before `init()` logs an error and has no effect.

```cpp
// BAD
ffe::ScriptEngine engine;
engine.setWorld(&world);  // logs error, does nothing -- no Lua state yet

// GOOD
ffe::ScriptEngine engine;
if (!engine.init()) { return false; }
engine.setWorld(&world);
engine.doFile("main.lua", assetRoot);
```

### Do not try to work around the sandbox

The sandbox blocks specific Lua globals and standard libraries for security reasons documented in ADR-004. If you need a capability that the sandbox blocks -- such as reading a file from Lua -- implement it as a new C++ binding in `registerEcsBindings()` with proper path validation, and expose it through the `ffe.*` table. Do not try to re-open blocked libraries or restore removed globals. They are removed, not just hidden.

### Do not cache the result of getTransform across frames

`getTransform` returns a copy of the Transform data at the time of the call, not a live reference. Caching it across frames means you are working with stale data.

```lua
-- BAD -- cached result is stale next frame
local t = ffe.getTransform(entityId)  -- called once
-- ... later that frame or next frame:
ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
-- t.x might not reflect changes made by other systems since the last call

-- GOOD -- always fetch fresh data at the point of use
local t = ffe.getTransform(entityId)
if t == nil then return end
ffe.setTransform(entityId, t.x + dx, t.y + dy, t.rotation, t.scaleX, t.scaleY)
```

### Do not pass entity IDs across ScriptEngine instances

Entity IDs are only valid within the World they were created in. Passing an ID from one ScriptEngine's World to another ScriptEngine will either silently operate on the wrong entity or return nil.

### Do not pass NaN or math.huge to setTransform

Non-finite values (`math.huge`, `-math.huge`, `0/0`) are rejected by `setTransform` and logged as errors at INFO level. The transform is not modified. This protects against silent state corruption.

```lua
-- BAD -- NaN propagation silently corrupts position
ffe.setTransform(id, 0/0, 0, 0, 1, 1)   -- rejected and logged

-- GOOD -- validate values before setting
local newX = computeX()
if newX ~= newX then return end  -- NaN check: NaN ~= NaN is true in Lua
ffe.setTransform(id, newX, 0, 0, 1, 1)
```

### Do not hold texture handles across scene transitions where C++ may unload them (dangling handle risk)

`ffe.addSprite` accepts a texture handle as an opaque integer. The integer is the `u32 id` field of a `rhi::TextureHandle` created on the C++ side. The scripting layer cannot verify whether the GPU texture behind the handle is still live — it only validates that the integer is in the valid range (`> 0` and `<= UINT32_MAX`).

If C++ calls `unloadTexture` (or equivalent) on a texture while a `Sprite` component that was set from Lua still references that handle, the sprite will attempt to render with a dangling handle. On the headless renderer this silently draws nothing. On a real GPU renderer, behaviour is undefined and may produce visual corruption or a GPU error.

**The rule:** never unload a texture while any entity has a `Sprite` component that references its handle. This applies whether the sprite was created from C++ or from Lua.

```lua
-- BAD: storing a texture handle across a scene boundary where C++ might unload it
local myTexHandle = 42    -- obtained from a C++ binding at scene load
-- ... later, after a scene transition that unloaded the texture ...
ffe.addSprite(id, myTexHandle, 32, 32, 1, 1, 1, 1, 0)  -- dangling handle!

-- GOOD: texture handles only used within the lifetime guaranteed by C++.
-- Obtain fresh handles at scene load, stop using them before scene teardown.
-- Do not hold texture handle integers in Lua globals that survive scene transitions.
```

If your game uses scene loading/unloading, make the following explicit in your C++ scene teardown code:
1. Remove (or destroy) all entities that have `Sprite` components referencing the textures you are about to unload.
2. Only then call `unloadTexture` (or the equivalent in your asset manager).
3. Re-create the sprites in the new scene after the new textures are loaded.

### Do not use shutdown() for per-level cleanup or assume it runs when the player presses ESC

`shutdown()` is called exactly once: when `ScriptEngine::shutdown()` is explicitly invoked, which happens when the `Application` exits (the engine is closing). It is NOT called when the player presses ESC, when the game loop stops temporarily, or when a scene is unloaded mid-session.

Use `shutdown()` only for final, application-level cleanup — releasing textures and other resources that your script holds for the entire session lifetime.

For per-level cleanup (unloading scene-specific textures when transitioning between levels), call a named Lua function explicitly from your C++ scene manager via `callFunction`. Do not rely on `shutdown()` for this — it will not be called until the application fully exits.

```lua
-- BAD: treating shutdown() as a per-level teardown hook
function shutdown()
    -- This will NOT be called when a level ends and a new one loads.
    -- It is only called when Application exits entirely.
    unloadLevelTextures()  -- called too late for mid-session scene transitions
end

-- GOOD: per-level cleanup via an explicitly named function
function onLevelUnload()
    -- Called from C++ scene management: scriptEngine.callFunction("onLevelUnload", ...)
    unloadLevelTextures()
end

-- GOOD: final application cleanup
function shutdown()
    -- Only session-lifetime resources belong here.
    if _G.uiAtlasTex then
        ffe.unloadTexture(_G.uiAtlasTex)
        _G.uiAtlasTex = nil
    end
    ffe.log("Application shutdown complete")
end
```

### Use fillTransform, not getTransform, in per-frame update loops

`getTransform` allocates a new Lua table on every call. At 60 fps with multiple entities, this creates significant GC pressure — the Lua garbage collector must eventually sweep all those short-lived tables, causing frame-time spikes. `fillTransform` writes into a caller-provided table, producing zero allocations per call.

```lua
-- BAD — getTransform allocates a new table every frame per entity
function update(entityId, dt)
    local t = ffe.getTransform(entityId)  -- new table allocation every call
    if t == nil then return end
    ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
end

-- GOOD — fillTransform reuses a pre-allocated table (zero GC pressure)
local t = {}  -- allocated once at load time

function update(entityId, dt)
    if not ffe.fillTransform(entityId, t) then return end
    ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
end
```

`getTransform` is still appropriate for one-shot queries (scene init, event handlers, debug inspection) where allocation cost is negligible. Use `fillTransform` whenever the call is inside a per-frame or per-entity loop.

### Do not call ffe.loadSound() per frame

`ffe.loadSound` triggers file I/O on disk and decodes the entire audio file into a PCM buffer. Calling it per frame is a severe performance violation — identical to the `ffe.loadTexture` anti-pattern. Load sounds once at scene or application startup, keep the integer handle, and reuse it.

```lua
-- BAD — called every frame
function update(entityId, dt)
    local sfx = ffe.loadSound("sfx/jump.wav")  -- WRONG: file I/O + decode every frame
    ffe.playSound(sfx)
end

-- GOOD — loaded once at startup; handle reused
local jumpSfx = ffe.loadSound("sfx/jump.wav")

function update(entityId, dt)
    if ffe.isKeyPressed(ffe.KEY_SPACE) and jumpSfx then
        ffe.playSound(jumpSfx)
    end
end
```

### Do not call getTransform/setTransform in tight inner loops

The World pointer registry lookup happens on every `getTransform`/`setTransform` call (one `lua_gettable` on the Lua registry per call). Additionally, `getTransform` allocates a new Lua table on every call, creating GC pressure. For high-frequency entity updates, batch your script updates: update multiple entities per Lua call rather than making one Lua call per entity.

## Security

The sandbox applies two defence layers:

**Layer 1 -- Library blocking:** Only `base` (partial), `math`, `string`, and `table` are opened. `io`, `os`, `package`, `debug`, `ffi`, and `jit` are never opened. This is done by calling `luaopen_*` selectively, not by opening everything and removing it afterwards.

**Layer 2 -- Global removal:** Dangerous globals brought in by `luaopen_base` are explicitly set to nil after the library is opened:

| Global removed | Reason |
|----------------|--------|
| `dofile`, `loadfile` | Direct filesystem access |
| `load` | CRITICAL: can load Lua bytecode with arbitrary upvalues, bypassing string-based sandbox |
| `loadstring` | LuaJIT/Lua 5.1 alias for `load` -- same risk |
| `require` | Package system access |
| `collectgarbage` | DoS via GC pressure |
| `rawset`, `rawget` | Bypass `__newindex`/`__index` metamethod guards |
| `rawequal`, `rawlen` | Related raw access |
| `setmetatable`, `getmetatable` | Sandbox escape via metamethods |
| `debug` | Covered by library blocking but also explicitly nil'd |

**Instruction budget:** 1,000,000 instructions per `doString`/`doFile` call. This is set via `lua_sethook` before any libraries are opened. It is active in all build configurations (not just debug). Exceeding the budget raises a Lua error, caught by `pcall` -- the engine does not crash.

**ECS binding input validation:** All entity IDs received from Lua are validated before use: negative values and out-of-range IDs (above `UINT32_MAX - 1`) are rejected immediately. `world->isValid()` is checked before any component access. Non-finite float values passed to `setTransform` and `addTransform` are rejected and logged. `addSprite` rejects texture handles `<= 0` and non-finite or non-positive width/height values. `addSprite` clamps color channels to `[0.0, 1.0]` (NaN treated as 0.0) and layer to `[0, 15]`. `addPreviousTransform` warns when called without a prior `addTransform`.

**Audio binding input validation:** `ffe.loadSound` applies the same type guard as `ffe.loadTexture` -- non-string arguments return `nil` immediately without calling into C++ (prevents silent coercion of numbers/booleans to strings). Path traversal and absolute paths are rejected by the C++ audio layer. `ffe.unloadSound` and `ffe.playSound` validate that the handle is a positive integer in `(0, UINT32_MAX]`; out-of-range handles are logged and ignored. Scripts cannot control the asset root -- it is always the engine's internal root set by C++.

Game developers should not attempt to work around the sandbox. If a script requires a capability the sandbox blocks, implement it as a new `ffe.*` C++ binding with appropriate validation.
