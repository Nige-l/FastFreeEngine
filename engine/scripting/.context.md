# engine/scripting — Lua Scripting Subsystem

AI-native context file. Status: Session 5 implementation complete — full documentation pending api-designer review.

## System Purpose

Provides a sandboxed Lua scripting environment for game logic. Scripts cannot access the filesystem, network, or OS directly. The scripting layer is the primary interface for game developers — engine internals stay in C++; game behaviour lives in Lua.

## Tier Support

LEGACY and above. Tested on OpenGL 3.3 hardware (1 GB VRAM, single-threaded).

## Public API (C++)

See `engine/scripting/script_engine.h`.

Key class: `ffe::ScriptEngine`

- `bool init()` — create and sandbox the Lua state; call once at startup
- `void shutdown()` — close the Lua state; safe to call without a prior `init()`
- `bool doString(const char* script)` — run a script from a string; errors are logged, not thrown
- `bool doFile(const char* path, const char* assetRoot)` — run a script from a relative path within assetRoot
- `bool isInitialised() const` — query readiness

## Security

Sandbox blocks these Lua standard libraries: `io`, `os`, `package`, `debug`, `ffi`, `jit`.

Dangerous globals removed from the base library: `dofile`, `loadfile`, `load`, `require`, `collectgarbage`, `rawset`, `rawget`, `rawequal`, `rawlen`, `setmetatable`, `getmetatable`.

Instruction budget: 1,000,000 instructions per call. Active in all build configurations. Exceeding the budget raises a Lua error (caught by pcall — does not crash the engine).

## Dependencies

- `ffe_core` (logging, types)
- sol2 v3.x (header-only binding layer, vcpkg)
- LuaJIT (system package: `libluajit-5.1-dev`)

## What NOT to Do

- **Never call doString() or doFile() per-frame or per-entity.** Both functions compile and execute Lua bytecode. Calling them in a hot path will destroy performance. Execute scripts at load time or in response to events, never inside a system's per-frame update.
