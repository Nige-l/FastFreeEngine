## System Purpose

The scripting subsystem provides a sandboxed Lua scripting environment for game logic. It wraps LuaJIT via the raw Lua C API (sol2 is a declared dependency but not yet used in this file -- the raw C API is used directly). One `ScriptEngine` instance manages one Lua state per game context.

**What "sandboxed" means for game developers:** Scripts you load through `ScriptEngine` cannot read or write files, make network connections, or call OS functions. The engine blocks all of those libraries at init time, before your scripts ever run. Scripts also have a hard instruction budget of 1,000,000 instructions per call -- if a script enters an infinite loop, it raises a Lua error (caught by the engine), not a crash. From a game developer's perspective this means: scripts are safe to run even if they came from an untrusted source (a mod, a level file, a config).

Game behaviour belongs in Lua; engine internals stay in C++. The scripting layer is the primary interface for writing game logic against FFE without touching C++.

## Public API (C++)

Header: `engine/scripting/script_engine.h`
Namespace: `ffe`
Class: `ffe::ScriptEngine`

`ScriptEngine` is non-copyable and non-movable. One instance per game context. All methods must be called from the main thread.

| Method | Signature | Description |
|--------|-----------|-------------|
| `init` | `bool init()` | Create and sandbox the Lua state. Must be called before any other method. Returns false and logs if initialisation fails (out of memory). Safe to call once only -- a warning is logged if called twice. |
| `shutdown` | `void shutdown()` | Close the Lua state and free all resources. Before closing the state, calls the Lua global `shutdown()` function if it is defined (no arguments). If that callback errors, the error is logged and shutdown proceeds normally. If `shutdown` is not defined in Lua, this is a silent no-op. Safe to call if `init()` was never called or failed. Also called automatically by the destructor. |
| `setWorld` | `void setWorld(ffe::World* world)` | Sets the `World*` for ECS binding access. Call after `init()`, before any `doString`/`doFile`. Passing `nullptr` clears the stored world pointer. The World must remain valid for the lifetime of this ScriptEngine. Not callable from Lua -- C++ only. |
| `doString` | `bool doString(const char* script)` | Compile and execute a Lua script from a null-terminated string. Returns false and logs the error if compilation or execution fails. Errors are caught -- this call cannot crash the engine. |
| `doFile` | `bool doFile(const char* path, const char* assetRoot)` | Load and execute a Lua script from a file. `path` must be relative (no leading `/`, no `../` components). `assetRoot` must be a non-empty absolute directory path. The full path is constructed as `assetRoot + "/" + path`. Returns false and logs on failure. |
| `callFunction` | `bool callFunction(const char* funcName, ffe::i64 entityId, double dt)` | Call a named global Lua function with `(entityId, dt)` arguments. This is the correct way to invoke per-frame Lua callbacks — it uses `lua_getglobal` + `lua_pcall` directly, avoiding the compile step that `doString` runs every call. Returns false and logs if the function does not exist or errors. Requires `init()` and `setWorld()` to have been called first. |
| `deliverCollisionEvents` | `void deliverCollisionEvents(World& world)` | Reads `CollisionEventList` from the World's ECS registry context and calls the Lua collision callback (set via `ffe.setCollisionCallback`) for each event. Call this after all systems have run (after `tick()`), before arena reset. No-op if no callback is registered or no events exist. |
| `setScriptRoot` | `bool setScriptRoot(const char* absolutePath)` | Sets the script asset root directory for `ffe.loadScene()`. Must be an absolute path. Write-once: returns false if already set, null, or empty. Not callable from Lua -- C++ only. Call this from `main.cpp` before loading any scripts. |
| `scriptRoot` | `const char* scriptRoot() const` | Returns the stored script root, or `nullptr` if not set. |
| `setSaveRoot` | `bool setSaveRoot(const char* absolutePath)` | Sets the save directory root for `ffe.saveData()`/`ffe.loadData()`. Save files are stored under `<saveRoot>/saves/`. Must be an absolute path. Write-once: returns false if already set, null, or empty. Not callable from Lua -- C++ only. Call this from `main.cpp` before loading scripts. |
| `saveRoot` | `const char* saveRoot() const` | Returns the stored save root, or `nullptr` if not set. |
| `isInitialised` | `bool isInitialised() const` | Returns true if `init()` has been called successfully and `shutdown()` has not yet been called. Use this to guard calls in conditional startup paths. |

## Lua Standard Libraries Available to Scripts

Scripts have access to a carefully chosen subset of the standard library. The engine opens only what is safe:

| Library | Available | Notes |
|---------|-----------|-------|
| `math` | Yes | Full math library |
| `string` | Yes | Full string library |
| `table` | Yes | Full table library |
| `base` (print, type, pairs, ipairs, tostring, tonumber, error, assert, pcall, xpcall, select, unpack, next) | Partial | Dangerous globals removed -- see Security section |
| `io` | **No** | Blocked entirely |
| `os` | **No** | Blocked entirely |
| `package` / `require` | **No** | Blocked entirely |
| `debug` | **No** | Blocked entirely |
| `ffi` | **No** | LuaJIT FFI blocked entirely |
| `jit` | **No** | LuaJIT JIT control blocked entirely |

## FFE Lua API (ffe.* table)

The engine installs an `ffe` global table in the Lua state with bindings to engine systems.

### Logging

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.log(message)` | nothing | Logs a message through the FFE logging system at INFO level, tagged with category "Lua". This is the only way scripts can emit diagnostic output -- scripts cannot write to stdout or files directly. |

### Engine Control

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.requestShutdown()` | nothing | Signals the engine to stop the game loop cleanly after the current tick completes. Sets `ShutdownSignal::requested = true` in the ECS registry context. Application checks this flag after every tick. Use this instead of `os.exit()` (which is blocked) or any other escape. |
| `ffe.cameraShake(intensity, duration)` | nothing | Triggers a camera shake effect. `intensity` is the base pixel offset (clamped to [0, 100] on input, but the **effective on-screen offset is hard-capped at 3 pixels** by the render path). `duration` is seconds (clamped to [0, 5]). If a shake is already active, the new one replaces it. NaN/Inf values are silently rejected. The shake decays exponentially (punchy start, smooth fade). In practice, intensity values between 0.2 and 2.0 cover the useful range. |
| `ffe.setBackgroundColor(r, g, b)` | nothing | Sets the clear/background color. All values clamped to [0, 1]. NaN/Inf silently rejected. Takes effect next frame. |
| `ffe.getEntityCount()` | integer | Returns the number of alive entities. Returns 0 if no World is registered. |
| `ffe.after(seconds, callback)` | timerId or nil | Schedules a one-shot timer. The callback (a function with no arguments) fires once after `seconds` elapse. Returns an integer timer ID for cancellation. Returns `nil` if `seconds` is negative, NaN, or Inf, or if the max timer limit (256) is reached. Zero delay fires on the next tick. |
| `ffe.every(seconds, callback)` | timerId or nil | Schedules a repeating timer. The callback fires every `seconds`. `seconds` must be > 0. Returns an integer timer ID. Returns `nil` on invalid input or timer limit reached. Timing is accurate — overshoot is accumulated, not lost. |
| `ffe.cancelTimer(timerId)` | nothing | Cancels an active timer by ID. No-op for invalid, negative, or already-cancelled IDs. Safe to call from within a timer callback. |
| `ffe.addTilemap(entityId, width, height, tileW, tileH, textureHandle, columns, tileCount [, layer])` | bool | Creates a Tilemap component on the entity. `width`/`height` are grid dimensions (max 1024x1024). `tileW`/`tileH` are tile size in world units. `textureHandle` is the tileset atlas. `columns`/`tileCount` describe the atlas grid. `layer` defaults to 0. All tiles initialised to 0 (empty). Returns false on invalid input. |
| `ffe.setTile(entityId, x, y, tileIndex)` | nothing | Sets a tile at grid position (x, y). Index 0 = empty/transparent. Indices 1+ map to atlas frames (1-based). No-op if out of bounds or entity has no Tilemap. |
| `ffe.getTile(entityId, x, y)` | integer or nil | Returns the tile index at grid position (x, y). Returns nil if out of bounds or entity has no Tilemap. |
| `ffe.destroyAllEntities()` | nothing | Nuclear reset for scene transitions. Frees Tilemap heap data, clears the collision callback, cancels all active timers (unrefs Lua callbacks), then destroys all entities and components via `World::clearAllEntities()`. ECS context values (ClearColor, CameraShake) survive the clear. Lua globals persist. Loaded textures and sounds are NOT unloaded -- call `ffe.unloadTexture()` / `ffe.unloadSound()` explicitly if needed. Safe to call twice. No-op if no World is registered. |
| `ffe.cancelAllTimers()` | nothing | Cancels every active timer without touching entities. All Lua callback references are properly released. Useful for game-phase transitions where entities should persist but scheduled callbacks should not. No-op if no ScriptEngine is registered. |
| `ffe.loadScene(scriptPath)` | nothing | Loads and executes a Lua script file for scene transitions. `scriptPath` is validated with the same path safety checks as `doFile` (no `../`, no absolute paths). The script root must be set via `ScriptEngine::setScriptRoot()` from C++ before this can be used. Each loaded script gets a fresh 1M instruction budget. Re-entrancy is guarded: max depth of 4 nested `loadScene` calls. Failures are logged, not thrown. |
| `ffe.loadSceneJSON(jsonPath)` | boolean | Loads a JSON scene file exported by the editor's build pipeline. Clears all existing entities first (`World::clearAllEntities()`), then deserialises the JSON scene into the World via `ffe::scene::loadScene()`. `jsonPath` is relative to the script root (set via `ScriptEngine::setScriptRoot()` from C++). Path is validated with the same safety checks as other file-loading bindings (no `../`, no absolute paths). Returns `true` on success, `false` on failure (missing file, parse error, no World registered, no script root set). This is the binding the exported runtime's generated `main.lua` calls to load the entry scene. |
| `ffe.drawText(text, x, y [, scale, r, g, b, a])` | nothing | Queues HUD text for screen-space rendering. `x`, `y` are screen pixels (origin top-left). `scale` defaults to 1.0 (8px glyphs); scale=2 gives 16px, etc. Color defaults to white (1,1,1,1). Color clamped to [0,1], scale clamped to [0.1,20]. NaN/Inf silently rejected. Text is rendered after world sprites, before the editor overlay. Uses a built-in 8x8 bitmap font (ASCII 32-126). Max 4096 glyphs per frame. Text persists across zero-tick render frames (the glyph buffer is only cleared when a fixed-timestep tick runs), so text drawn in `update()` remains visible even when the engine renders extra frames between ticks. |
| `ffe.getScreenWidth()` | number | Returns the screen width in pixels (e.g. 1280). Returns 0 if no World is registered. Useful for centering text. |
| `ffe.getScreenHeight()` | number | Returns the screen height in pixels (e.g. 720). Returns 0 if no World is registered. |

```lua
-- ESC to quit cleanly from Lua:
if ffe.isKeyPressed(ffe.KEY_ESCAPE) then
    ffe.requestShutdown()
end

-- Draw HUD text (call every frame from update):
ffe.drawText("SCORE: " .. tostring(score), 20, 20, 3, 1, 1, 1, 1)
ffe.drawText("GAME OVER", 430, 320, 5, 1, 0.3, 0.3, 1)
```

### Entity Lifecycle

Requires `setWorld()` to have been called from C++ before any script runs. If no World is registered, `createEntity` returns `nil` and the other bindings return `false` or are no-ops.

**entityId** is a Lua integer (the raw `u32` value of `EntityId`). Negative integers and values above `UINT32_MAX - 1` are always rejected.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.createEntity()` | integer or nil | Creates a new entity in the registered World. Returns an integer entity ID on success, `nil` if no World is registered or the ECS is out of entity slots. |
| `ffe.destroyEntity(entityId)` | nothing | Destroys an entity and all its components. No-op if `entityId` is negative, out of range, or no longer valid (already destroyed). |
| `ffe.addTransform(entityId, x, y, rotation, scaleX, scaleY)` | bool | Adds a `Transform` component to the entity. If the entity already has a `Transform`, it is overwritten (a warning is logged). Returns `false` if the entity ID is invalid or out of range, or if any float argument is non-finite (NaN or Infinity). Returns `true` on success. All arguments after `entityId` are numbers (floats). |
| `ffe.addSprite(entityId, texHandle, width, height, r, g, b, a, layer)` | bool | Adds a `Sprite` component to the entity. If the entity already has a `Sprite`, it is overwritten (a warning is logged). Returns `false` if the entity ID is invalid, `texHandle` is `<= 0`, or `width`/`height` is non-finite or `<= 0`. Color values `r`, `g`, `b`, `a` are clamped to `[0.0, 1.0]` (NaN is treated as `0.0`). `layer` is clamped to `[0, 15]`. `texHandle` is an opaque integer — the `u32 id` field of a `rhi::TextureHandle` created on the C++ side. |
| `ffe.addPreviousTransform(entityId)` | bool | Opts the entity in to render interpolation by adding a `PreviousTransform` component. If the entity has a `Transform`, `PreviousTransform` is initialised from it. If no `Transform` exists, `PreviousTransform` is initialised to zero position/unit scale and a warning is logged — call `addTransform` first. Returns `false` if the entity ID is invalid or out of range. |

| `ffe.setSpriteColor(entityId, r, g, b, a)` | nothing | Modifies the color of an existing Sprite component. No-op if the entity has no Sprite. All values must be finite (NaN/Inf rejected). Safe to call per-frame — no log spam. |
| `ffe.setSpriteSize(entityId, width, height)` | nothing | Modifies the size of an existing Sprite component. No-op if the entity has no Sprite. Width and height must be finite and > 0 (rejected otherwise). Safe to call per-frame. |
| `ffe.setSpriteFlip(entityId, flipX, flipY)` | nothing | Sets horizontal/vertical flip flags on a Sprite component. `flipX`/`flipY` are booleans. Flipping swaps UV coordinates at render time, so it works correctly with sprite animations. No-op if the entity has no Sprite. Safe to call per-frame. |

**Overwrite behaviour:** `addTransform`, `addSprite`, `addPreviousTransform`, and `addSpriteAnimation` all use `emplace_or_replace` internally — calling them on an entity that already has the component overwrites the existing component and logs a warning. This is safe but is an indication of a logic error in most game code. For per-frame changes, use `setTransform`, `setSpriteColor`, and `setSpriteSize` instead of the `add*` functions.

### Sprite Animation

These functions add and control atlas-based sprite animation on entities. The entity must already have a `Sprite` component — the animation system updates the sprite's `uvMin`/`uvMax` each frame based on the current frame index and grid layout.

Requires `setWorld()` to have been called from C++ before any script runs. All functions are no-ops (or return `false`) if no World is registered or the entity ID is invalid.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.addSpriteAnimation(entityId, frameCount, columns, frameTime, looping)` | boolean | Adds a `SpriteAnimation` component to the entity. `frameCount` is the total number of frames in the atlas (1-65535). `columns` is the number of columns in the atlas grid (1-frameCount). `frameTime` is seconds per frame (must be > 0 and finite). `looping` is a boolean — if true, the animation wraps around; if false, it stops on the last frame. The animation starts **not playing** — call `ffe.playAnimation` to start it. Returns `false` if any parameter is invalid or the entity does not exist. If the entity already has a `SpriteAnimation`, it is overwritten and a warning is logged. |
| `ffe.playAnimation(entityId)` | nothing | Start (or restart) the animation. Sets `playing = true` and resets the elapsed timer to 0. No-op if the entity has no `SpriteAnimation` component. |
| `ffe.stopAnimation(entityId)` | nothing | Pause the animation at the current frame. Sets `playing = false`. No-op if the entity has no `SpriteAnimation` component. |
| `ffe.setAnimationFrame(entityId, frame)` | nothing | Jump to a specific frame. `frame` is clamped to `[0, frameCount - 1]`. Also updates the sprite's UVs immediately (if the entity has a `Sprite` component). No-op if the entity has no `SpriteAnimation` component. |
| `ffe.isAnimationPlaying(entityId)` | boolean | Returns `true` if the animation is currently playing. Returns `false` if the entity has no `SpriteAnimation` component, the entity does not exist, or no World is registered. |

**Atlas layout convention:** The animation system divides the texture into a grid of `columns` x `rows` cells (rows are computed as `ceil(frameCount / columns)`). Frame 0 is the top-left cell. Frames advance left-to-right, top-to-bottom. The sprite's `uvMin`/`uvMax` are rewritten each frame by `animationUpdateSystem` — do not manually set UVs on an animated sprite.

**Sprite animation example — set up an animated sprite from Lua:**

```lua
-- Load a spritesheet with 8 frames arranged in 4 columns x 2 rows.
local tex = ffe.loadTexture("spritesheet.png")
local entity = ffe.createEntity()
ffe.addTransform(entity, 0, 0, 0, 1, 1)
ffe.addSprite(entity, tex, 32, 32, 1, 1, 1, 1, 0)
ffe.addSpriteAnimation(entity, 8, 4, 0.1, true)  -- 8 frames, 4 columns, 0.1s/frame, looping
ffe.playAnimation(entity)
```

**One-shot animation (play once, then stop):**

```lua
-- Explosion effect: 6 frames, 6 columns (single row), 0.05s/frame, NOT looping.
local explosionTex = ffe.loadTexture("effects/explosion.png")
local fx = ffe.createEntity()
ffe.addTransform(fx, hitX, hitY, 0, 1, 1)
ffe.addSprite(fx, explosionTex, 64, 64, 1, 1, 1, 1, 5)
ffe.addSpriteAnimation(fx, 6, 6, 0.05, false)
ffe.playAnimation(fx)

-- Later, check if the animation finished:
if not ffe.isAnimationPlaying(fx) then
    ffe.destroyEntity(fx)
end
```

**Manually setting a frame (e.g., idle pose selection):**

```lua
-- Jump to frame 3 (clamped to valid range). Updates UVs immediately.
ffe.stopAnimation(entity)
ffe.setAnimationFrame(entity, 3)
```

### Particle Emitter

These functions manage engine-side particle emitters attached to entities. Each emitter owns a fixed-size pool of up to 128 particles (no heap allocation). Particles are rendered as sprites using the default white texture or a custom texture.

Requires `setWorld()` to have been called from C++ before any script runs. The entity must have a `Transform` component before adding an emitter. All functions are no-ops if no World is registered or the entity ID is invalid.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.addEmitter(entityId [, config])` | boolean | Adds a `ParticleEmitter` component. Returns `false` if the entity has no `Transform`. Optional config table sets properties (see below). |
| `ffe.setEmitterConfig(entityId, config)` | nothing | Updates emitter properties from a config table. No-op if entity has no emitter. |
| `ffe.startEmitter(entityId)` | nothing | Start continuous particle emission. |
| `ffe.stopEmitter(entityId)` | nothing | Stop continuous emission. Existing particles continue to live out their lifetime. |
| `ffe.emitBurst(entityId, count)` | nothing | Emit a fixed number of particles immediately. Capped at 128. |
| `ffe.removeEmitter(entityId)` | nothing | Remove the `ParticleEmitter` component from the entity. |

**Config table fields (all optional):**

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `emitRate` | number | 10 | Particles per second (continuous mode) |
| `lifetimeMin` | number | 0.5 | Minimum particle lifetime in seconds |
| `lifetimeMax` | number | 1.5 | Maximum particle lifetime in seconds |
| `speedMin` | number | 20 | Minimum initial speed |
| `speedMax` | number | 80 | Maximum initial speed |
| `angleMin` | number | 0 | Minimum emission angle in radians (0 = right) |
| `angleMax` | number | 6.28 | Maximum emission angle in radians (full circle) |
| `sizeStart` | number | 4 | Particle size at birth |
| `sizeEnd` | number | 0 | Particle size at death |
| `gravityY` | number | 0 | Gravity applied to Y velocity each tick |
| `offsetX` | number | 0 | Spawn offset from entity position (X) |
| `offsetY` | number | 0 | Spawn offset from entity position (Y) |
| `texture` | integer | (none) | Texture handle from `ffe.loadTexture()`. Uses default white if omitted. |
| `layer` | integer | 0 | Render layer for particles |
| `sortOrder` | integer | 0 | Sort order within the same layer (higher draws later) |
| `colorStartR/G/B/A` | number | 1,1,1,1 | Particle color at birth |
| `colorEndR/G/B/A` | number | 1,1,1,0 | Particle color at death (fades to transparent by default) |

**Usage example:**

```lua
-- Create an explosion effect
local explosion = ffe.createEntity()
ffe.addTransform(explosion, 200, 100)
ffe.addEmitter(explosion, {
    emitRate = 0,
    lifetimeMin = 0.3,
    lifetimeMax = 0.8,
    speedMin = 50,
    speedMax = 200,
    sizeStart = 6,
    sizeEnd = 1,
    gravityY = -50,
    colorStartR = 1, colorStartG = 0.8, colorStartB = 0.2, colorStartA = 1,
    colorEndR = 1, colorEndG = 0.1, colorEndB = 0, colorEndA = 0
})
ffe.emitBurst(explosion, 50)

-- Create a continuous fire effect
local fire = ffe.createEntity()
ffe.addTransform(fire, 0, -200)
ffe.addEmitter(fire, {
    emitRate = 30,
    lifetimeMin = 0.5,
    lifetimeMax = 1.0,
    speedMin = 20,
    speedMax = 60,
    angleMin = 1.2,  -- ~70 degrees (upward cone)
    angleMax = 1.9,  -- ~110 degrees
    sizeStart = 5,
    sizeEnd = 1,
    gravityY = 30,   -- particles float up
    colorStartR = 1, colorStartG = 0.6, colorStartB = 0.1, colorStartA = 1,
    colorEndR = 0.3, colorEndG = 0.1, colorEndB = 0, colorEndA = 0
})
ffe.startEmitter(fire)
```

### Collision / Physics

These functions manage 2D colliders on entities and register a callback for collision events. The collision system (`collisionSystem`, priority 200) detects overlaps each frame and writes a `CollisionEventList` to the ECS context. After all systems have run, `ScriptEngine::deliverCollisionEvents()` reads those events and invokes the registered Lua callback once per overlapping pair.

Requires `setWorld()` to have been called from C++ before any script runs. All functions are no-ops (or return `false`) if no World is registered or the entity ID is invalid.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.addCollider(entityId, shape, halfW, halfH [, layer, mask, isTrigger])` | boolean | Adds a `Collider2D` component to the entity. `shape` is a string: `"aabb"` or `"circle"`. `halfW` and `halfH` are positive, finite numbers (for circles, `halfW` is the radius and `halfH` is ignored but must be >= 0). Optional `layer` and `mask` are integers in `[0, 65535]` (default `0xFFFF` -- collides with everything). Optional `isTrigger` is a boolean (default `false`). Returns `false` if the entity ID is invalid, the shape string is unrecognised, or numeric arguments are non-finite or out of range. If the entity already has a `Collider2D`, it is overwritten and a warning is logged. |
| `ffe.removeCollider(entityId)` | nothing | Removes the `Collider2D` component from the entity. No-op if the entity has no collider or the entity ID is invalid. |
| `ffe.setCollisionCallback(func)` | nothing | Registers a Lua function to be called for each collision event each frame. The function receives two arguments: `(entityA, entityB)` where `entityA < entityB` (canonical ordering). Pass `nil` to unregister the callback. Only one callback can be registered at a time -- setting a new one replaces the previous. |

**Layer/mask filtering:** Two entities A and B generate a collision event only if both `(A.layer & B.mask) ~= 0` and `(B.layer & A.mask) ~= 0`. Default values (`0xFFFF`) mean "collide with everything".

**Collision callback example -- simple pickup system:**

```lua
-- game.lua
-- Set up a player and a coin with colliders.
local player = ffe.createEntity()
ffe.addTransform(player, 100, 200, 0, 1, 1)
ffe.addCollider(player, "aabb", 16, 16, 1, 2)  -- layer 1, collides with layer 2

local coin = ffe.createEntity()
ffe.addTransform(coin, 300, 200, 0, 1, 1)
ffe.addCollider(coin, "circle", 8, 0, 2, 1, true)  -- layer 2, collides with layer 1, trigger

local score = 0

ffe.setCollisionCallback(function(entityA, entityB)
    -- Check if this collision involves the coin.
    if entityA == coin or entityB == coin then
        ffe.destroyEntity(coin)
        score = score + 1
        ffe.log("Coin collected! Score: " .. tostring(score))
    end
end)
```

**Important:** The collision callback is invoked after all ECS systems have finished for the frame. Entity destruction inside the callback is safe -- the destroyed entity will simply not appear in the next frame's collision pass.

### Texture Lifecycle

These functions load and unload GPU textures from Lua scripts. The C++ `setAssetRoot()` (or equivalent asset root configuration) must be called before any texture can be loaded -- if no asset root is configured, `ffe.loadTexture` returns `nil`.

Texture handles returned by `ffe.loadTexture` are opaque integers -- the `u32 id` field of a `rhi::TextureHandle`. Pass them directly to `ffe.addSprite`. Do not modify or fabricate handle values.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.loadTexture(path)` | integer or nil | Load a texture file relative to the C++-configured global asset root. `path` must be a string -- passing a non-string returns `nil` immediately without logging. Returns an integer handle id on success, `nil` on failure. Failure reasons are logged by the C++ layer (file not found, unsupported format, GPU upload failure). The C++ `setAssetRoot()` must be called before this function works. |
| `ffe.unloadTexture(handle)` | nothing | Destroy a GPU texture created by `ffe.loadTexture`. Safe no-op for out-of-range handles (0, negative, or above `UINT32_MAX`). After calling this, do not pass the handle to `ffe.addSprite` -- the underlying GPU texture no longer exists. |

**Texture lifecycle example -- load at scene start, use for sprites, unload at teardown:**

```lua
-- scene_init.lua
-- Called once when the scene loads, NOT per frame.

local playerTex = ffe.loadTexture("sprites/player.png")
if playerTex == nil then
    ffe.log("ERROR: failed to load player texture -- sprites will be invisible")
    return
end

local enemyTex = ffe.loadTexture("sprites/enemy.png")

-- Create entities and attach sprites using the loaded handles.
local player = ffe.createEntity()
ffe.addTransform(player, 100.0, 200.0, 0.0, 1.0, 1.0)
ffe.addSprite(player, playerTex, 32.0, 32.0, 1.0, 1.0, 1.0, 1.0, 0)
ffe.addPreviousTransform(player)

local enemy = ffe.createEntity()
ffe.addTransform(enemy, 400.0, 300.0, 0.0, 1.0, 1.0)
ffe.addSprite(enemy, enemyTex, 32.0, 32.0, 1.0, 0.2, 0.2, 1.0, 0)

-- Store handles for later cleanup.
_G.sceneTextures = { playerTex, enemyTex }
_G.playerEntityId = player
_G.enemyEntityId  = enemy
```

```lua
-- scene_teardown.lua
-- Called once when the scene unloads.
-- Destroy sprite entities BEFORE unloading their textures.

if _G.playerEntityId then ffe.destroyEntity(_G.playerEntityId) end
if _G.enemyEntityId  then ffe.destroyEntity(_G.enemyEntityId)  end

-- Now safe to release GPU textures.
if _G.sceneTextures then
    for _, tex in ipairs(_G.sceneTextures) do
        ffe.unloadTexture(tex)
    end
    _G.sceneTextures = nil
end
```

### Music Playback

These functions wrap `ffe::audio` music playback. One music track at a time. Music volume is independent of the sound-effects master volume. All functions are no-ops in headless mode (CI) or when audio is unavailable.

Music track handles should be loaded from Lua using `ffe.loadMusic(path)`, which validates the path and stores it for streaming without fully decoding the file. This avoids the 10 MB decoded-size limit that `ffe.loadSound()` enforces. The returned handle can be passed to `ffe.playMusic()` but NOT to `ffe.playSound()`. For small music files (under 10 MB decoded), `ffe.loadSound()` also works.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.playMusic(soundHandle, loop)` | nothing | Start playing a music track. `soundHandle` is the integer from `ffe::audio::SoundHandle::id` (loaded in C++). `loop` is boolean (default `true` if omitted). If a track is already playing, it stops first. No-op if handle is invalid or audio is unavailable. |
| `ffe.stopMusic()` | nothing | Stop the current music track immediately. No-op if no music is playing. |
| `ffe.setMusicVolume(volume)` | nothing | Set music volume `[0.0, 1.0]`. Clamped; NaN/Inf treated as 0.0. |
| `ffe.getMusicVolume()` | number | Query current music volume `[0.0, 1.0]`. Safe to call per-frame (reads an atomic). |
| `ffe.isMusicPlaying()` | boolean | Returns true if a music track is active. Safe to call per-frame (reads an atomic). |

**Key constants added for music controls:**

```lua
ffe.KEY_M     -- M key (76) — common music toggle binding
-- (ffe.KEY_UP / ffe.KEY_DOWN also available for volume control)
```

**Example — load music from Lua, start looping, and add toggle controls:**

```lua
-- game.lua
-- Load the music track at script load time (once).
local musicHandle = ffe.loadMusic("music/theme.ogg")
local musicStarted = false

function update(entityId, dt)
    -- Start music on first tick
    if not musicStarted and musicHandle then
        ffe.playMusic(musicHandle, true)  -- loop = true
        ffe.setMusicVolume(0.5)
        musicStarted = true
    end

    -- M key: toggle music
    if ffe.isKeyPressed(ffe.KEY_M) then
        if ffe.isMusicPlaying() then
            ffe.stopMusic()
        else
            ffe.playMusic(musicHandle, true)
        end
    end
end

function shutdown()
    ffe.stopMusic()
    if musicHandle then
        ffe.unloadSound(musicHandle)
        musicHandle = nil
    end
end
```

**Alternative — pass the handle from C++ (still supported):**

```lua
-- C++ side sets g_musicHandle as a Lua global before loading game.lua:
--   ffe::audio::SoundHandle h = ffe::audio::loadSound("music/theme.ogg", audioRoot);
--   lua_pushinteger(L, h.id);
--   lua_setglobal(L, "g_musicHandle");

-- Lua side can then use g_musicHandle directly with ffe.playMusic().
```

### Sound Effects

These functions load, unload, and play one-shot sound effects from Lua. Sound handles follow the same security model as texture handles: path traversal is rejected, the asset root is controlled by C++ (not scriptable), and non-string arguments to `loadSound` return `nil` immediately.

All functions are no-ops in headless mode (CI) or when audio is unavailable.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.loadSound(path)` | integer or nil | Load a WAV or OGG audio file relative to the C++-configured asset root. `path` must be a string -- passing a non-string returns `nil` immediately and logs an error. Returns an integer handle on success, `nil` on failure. Failure reasons are logged by the C++ audio layer. Same security model as `ffe.loadTexture`: explicit type guard (rejects numbers/booleans silently coerced to strings), path traversal rejection (no `../`, no absolute paths), and asset root controlled exclusively by C++. Subject to 10 MB decoded size limit. |
| `ffe.loadMusic(path)` | integer or nil | Lightweight music loader for streaming playback via `ffe.playMusic()`. Validates the path and stores it for streaming but does NOT decode the file to PCM. The 10 MB decoded size limit does not apply (only the 32 MB raw file size limit applies). The returned handle can be passed to `ffe.playMusic()` but NOT to `ffe.playSound()`. Same security model as `ffe.loadSound()`. |
| `ffe.unloadSound(handle)` | nothing | Unload a previously loaded sound and free its decoded PCM buffer. `handle` must be a positive integer in `(0, UINT32_MAX]`. Out-of-range handles (0, negative, or above `UINT32_MAX`) are logged as errors and ignored. After calling this, do not pass the handle to `ffe.playSound` or `ffe.playMusic` -- the underlying data no longer exists. Works for handles from both `ffe.loadSound()` and `ffe.loadMusic()`. |
| `ffe.playSound(handle [, volume])` | nothing | Play a one-shot sound effect. `handle` is an integer returned by `ffe.loadSound`. Optional `volume` argument (default `1.0`) controls playback volume. Invalid handles are logged as errors and ignored. |
| `ffe.setMasterVolume(volume)` | nothing | Set the master volume for all audio output. `volume` is a number in `[0.0, 1.0]`. Values are passed directly to the C++ audio layer which clamps them. |

**Sound lifecycle example -- load at scene start, play during gameplay, unload at teardown:**

```lua
-- scene_init.lua (called once when the scene loads)

    ffe.log("WARNING: failed to load jump sound effect")
end

-- Store handles for later use.

```

```lua
-- game_logic.lua (per-frame update)
function update(entityId, dt)
    if ffe.isKeyPressed(ffe.KEY_SPACE) and _G.sceneSounds.jump then
        ffe.playSound(_G.sceneSounds.jump, 0.8)  -- play at 80% volume
    end

    -- Collect a coin → play coin sound at full volume
    if coinCollected and _G.sceneSounds.coin then
        ffe.playSound(_G.sceneSounds.coin)  -- default volume 1.0
    end
end
```

```lua
-- scene_teardown.lua (called once when the scene unloads)
if _G.sceneSounds then
    for _, handle in pairs(_G.sceneSounds) do
        if handle then ffe.unloadSound(handle) end
    end
    _G.sceneSounds = nil
end
```


| `ffe.disableShadows()` | nothing | Disable shadow mapping and free GPU resources (depth texture + FBO). Safe to call when already disabled (no-op). |
| `ffe.setShadowBias(bias?)` | nothing | Set the depth bias to reduce shadow acne. Clamped to `[0.0, 0.1]`. Defaults to `0.005`. Non-finite values are rejected (warning logged). |
| `ffe.setShadowArea(width?, height?, near?, far?)` | nothing | Configure the orthographic frustum for the shadow light camera. All values must be positive, finite, and `near < far`. Defaults: `20, 20, 0.1, 50`. Invalid inputs are rejected (warning logged, previous values preserved). |

#### Skybox

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.loadSkybox(right, left, top, bottom, front, back)` | boolean | Load 6 face images into a cubemap texture and enable skybox rendering. All 6 arguments must be strings (face image paths relative to the asset root, same rules as `ffe.loadTexture`). Parameter order follows the standard OpenGL cubemap convention: `+X` (right), `-X` (left), `+Y` (top), `-Y` (bottom), `+Z` (front), `-Z` (back). Returns `true` on success, `false` on failure. If a skybox is already loaded, the previous cubemap is destroyed before creating the new one. Cold path -- call at scene init, not per frame. |
| `ffe.unloadSkybox()` | nothing | Delete the cubemap texture and disable skybox rendering. Frees GPU memory immediately. Safe to call when no skybox is loaded (no-op). Call this in `shutdown()` to avoid leaking GPU memory. |
| `ffe.setSkyboxEnabled(enabled)` | nothing | Toggle skybox rendering on or off without unloading the cubemap texture. `enabled` is a boolean. Useful for settings menus where the player can toggle the skybox. No-op if no `SkyboxConfig` is in the ECS context. |

**3D usage example — load a mesh, place it, set camera, lighting, materials, point lights, shadows, and skybox:**

```lua
-- Init: load assets once (not per frame)
local meshHandle = ffe.loadMesh("models/crate.glb")
if meshHandle == 0 then return end  -- loadMesh already logged the error

local entity = ffe.createEntity3D(meshHandle, 0, 0, 0)
ffe.setMeshColor(entity, 1, 0.8, 0.6, 1)    -- warm tint

-- Optional diffuse texture on the mesh:
local crateTex = ffe.loadTexture("textures/crate.png")
ffe.setMeshTexture(entity, crateTex)

-- Specular material: shiny surface with tight highlight
ffe.setMeshSpecular(entity, 1, 1, 1, 64)

-- Optional normal map and specular map for extra surface detail:
local normalTex = ffe.loadTexture("textures/crate_normal.png")
local specTex   = ffe.loadTexture("textures/crate_specular.png")
ffe.setMeshNormalMap(entity, normalTex)
ffe.setMeshSpecularMap(entity, specTex)

-- Directional light (sun)
ffe.setLightDirection(1, -1, 0.5)
ffe.setLightColor(1, 0.95, 0.8)
ffe.setAmbientColor(0.1, 0.1, 0.15)

-- Point lights: warm torch and cool accent
ffe.addPointLight(0,  2, 1, 0,  1.0, 0.8, 0.3,  8)   -- warm, radius 8
ffe.addPointLight(1, -3, 2, 1,  0.3, 0.5, 1.0, 12)   -- cool, radius 12

-- Enable shadows (optional -- 1024x1024 default)
ffe.enableShadows()
ffe.setShadowArea(15, 15, 0.1, 30)   -- tighten frustum for a small scene

-- Skybox (optional -- 6 face images: right, left, top, bottom, front, back)
ffe.loadSkybox(
    "skybox/right.png", "skybox/left.png",
    "skybox/top.png",   "skybox/bottom.png",
    "skybox/front.png", "skybox/back.png"
)

-- Orbit camera state (updated per frame)
local orbitYaw, orbitPitch = 0, 20
local time = 0

function update(entityId, dt)
    time = time + dt

    -- Spin the mesh
    local angle = 0
    angle = angle + 60 * dt
    ffe.setTransform3D(entity, 0, 0, 0,  0, angle, 0,  1, 1, 1)

    -- Animate point light 0 position (orbit around the mesh)
    local lx = math.cos(time) * 3
    local lz = math.sin(time) * 3
    ffe.setPointLightPosition(0, lx, 1, lz)

    -- Orbit camera from keyboard
    if ffe.isKeyHeld(ffe.KEY_LEFT)  then orbitYaw   = orbitYaw   - 60 * dt end
    if ffe.isKeyHeld(ffe.KEY_RIGHT) then orbitYaw   = orbitYaw   + 60 * dt end
    if ffe.isKeyHeld(ffe.KEY_UP)    then orbitPitch  = orbitPitch  + 40 * dt end
    if ffe.isKeyHeld(ffe.KEY_DOWN)  then orbitPitch  = orbitPitch  - 40 * dt end
    ffe.set3DCameraOrbit(0, 0, 0, 6, orbitYaw, orbitPitch)
end

function shutdown()
    ffe.removePointLight(0)
    ffe.removePointLight(1)
    ffe.unloadSkybox()
    ffe.unloadMesh(meshHandle)
    ffe.unloadTexture(crateTex)
    ffe.unloadTexture(normalTex)
    ffe.unloadTexture(specTex)
end
```

### 3D Skeletal Animation

These functions control bone-based skeletal animation on 3D mesh entities. The entity must have a `Mesh` component that references a `.glb` file containing skeleton and animation data. `playAnimation3D` automatically adds `Skeleton` and `AnimationState` ECS components if not already present.

Requires `setWorld()` to have been called from C++ before any script runs. All functions are no-ops (or return default values) if no World is registered or the entity ID is invalid.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.playAnimation3D(entityId, animIndex, loop)` | nothing | Start playing animation clip `animIndex` (0-based, default `0`) on the entity. `loop` is a boolean (default `true`). Resets playback time to 0. Adds `Skeleton` and `AnimationState` components if not present. Entity must have a `Mesh` component whose `.glb` contains skeleton data -- logs a warning and returns if not. `animIndex` is validated against the mesh's clip count -- logs an error if out of range. |
| `ffe.stopAnimation3D(entityId)` | nothing | Stop animation playback. The entity freezes at its current pose. No-op if entity has no `AnimationState` component. |
| `ffe.setAnimationSpeed3D(entityId, speed)` | nothing | Set the playback speed multiplier. `speed` defaults to `1.0`. Negative values are clamped to `0`. A speed of `0` effectively pauses the animation at the current frame. No-op if entity has no `AnimationState`. |
| `ffe.getAnimationProgress3D(entityId)` | number | Returns the current playback progress as a float in `[0.0, 1.0]` (computed as `time / clipDuration`). Returns `0.0` if the entity has no `AnimationState`, no `Mesh`, or the clip duration is zero. |
| `ffe.isAnimation3DPlaying(entityId)` | boolean | Returns `true` if the entity's animation is currently playing (`AnimationState.playing == true`). Returns `false` if the entity has no `AnimationState`, the entity does not exist, or no World is registered. |
| `ffe.getAnimationCount3D(entityId)` | integer | Returns the number of animation clips embedded in the entity's mesh. Returns `0` if the entity has no `Mesh` component or the mesh contains no animation data. Useful for validating `animIndex` before calling `playAnimation3D`. |

**Skeletal animation example -- load an animated character and play a walk cycle:**

```lua
-- Init: load assets once (not per frame)
local charMesh = ffe.loadMesh("models/character.glb")
if charMesh == 0 then return end

local character = ffe.createEntity3D(charMesh, 0, 0, 0)
ffe.setTransform3D(character, 0, 0, 0,  0, 0, 0,  1, 1, 1)

-- Check available animations
local animCount = ffe.getAnimationCount3D(character)
ffe.log("Character has " .. animCount .. " animation(s)")

-- Play the first animation (walk cycle), looping
if animCount > 0 then
    ffe.playAnimation3D(character, 0, true)
end

-- Camera
ffe.set3DCameraOrbit(0, 1, 0, 5, 30, 15)
ffe.setLightDirection(1, -1, 0.5)

function update(entityId, dt)
    -- Animation advances automatically via animationUpdateSystem3D
    -- Monitor progress if needed:
    local progress = ffe.getAnimationProgress3D(character)
    ffe.drawText(string.format("Anim: %.0f%%", progress * 100), 10, 10)

    -- Switch to run animation (clip 1) when holding Shift
    if ffe.isKeyPressed(ffe.KEY_LEFT_SHIFT) then
        if animCount > 1 then
            ffe.playAnimation3D(character, 1, true)
            ffe.setAnimationSpeed3D(character, 1.5)  -- faster playback
        end
    end
    if ffe.isKeyReleased(ffe.KEY_LEFT_SHIFT) then
        ffe.playAnimation3D(character, 0, true)  -- back to walk
        ffe.setAnimationSpeed3D(character, 1.0)
    end

    if ffe.isKeyPressed(ffe.KEY_ESCAPE) then
        ffe.requestShutdown()
    end
end

function shutdown()
    ffe.unloadMesh(charMesh)
end
```

**One-shot animation (play once, then destroy):**

```lua
-- Death animation: play clip 2, do NOT loop
ffe.playAnimation3D(character, 2, false)

-- In update, check when animation finishes:
function update(entityId, dt)
    if not ffe.isAnimation3DPlaying(character) then
        ffe.destroyEntity(character)
    end
end
```

### 3D Collision Callbacks

These functions register and manage Lua callbacks for 3D physics collision events. The callback receives detailed contact information (entity IDs, contact point, normal, event type). See `engine/physics/.context.md` for full collision system documentation and usage examples.

Requires 3D physics to be initialized. All functions are no-ops if no physics world exists.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.onCollision3D(callback)` | nothing | Register a Lua function to receive 3D collision events. Only one callback at a time — calling again replaces the previous one. Callback signature: `function(entityA, entityB, px, py, pz, nx, ny, nz, eventType)` where `eventType` is `"enter"`, `"stay"`, or `"exit"`. Unused trailing parameters can be omitted (Lua discards extras). |
| `ffe.removeCollision3DCallback()` | nothing | Unregister the 3D collision callback. Events are still buffered but not delivered to Lua. No-op if no callback is registered. |
| `ffe.dispatchCollision3DEvents()` | nothing | **Engine-internal.** Dispatches buffered collision events to the registered callback. Called automatically by the engine loop — game scripts should not call this. Exposed for testing only. |

### 3D Raycasting

These functions cast rays into the 3D physics world and return hit information. All inputs are validated — NaN/Inf values return `nil` (castRay) or an empty table (castRayAll) with a warning logged.

| Function | Returns | Description |
|----------|---------|-------------|
| `ffe.castRay(ox, oy, oz, dx, dy, dz, maxDist)` | `entity, hx, hy, hz, nx, ny, nz, dist` or `nil` | Cast a ray from origin in direction, up to maxDist. Returns 8 values (entity ID, hit point, hit normal, distance) on hit, `nil` on miss. Zero-allocation — safe for per-frame use. |
| `ffe.castRayAll(ox, oy, oz, dx, dy, dz, maxDist)` | table | Cast a ray and return all hits as an array of tables sorted by distance. Each entry: `{ entity=id, x=.., y=.., z=.., nx=.., ny=.., nz=.., distance=.. }`. Returns empty table on miss. Max 32 hits. Allocates tables — avoid calling in tight per-frame loops. |

## Common Usage Patterns

### 1. Engine startup -- initialise ScriptEngine, register World, and load a game script

```cpp
#include "scripting/script_engine.h"

// In Application::startup() or equivalent:
ffe::ScriptEngine scriptEngine;

if (!scriptEngine.init()) {
    FFE_LOG_ERROR("Game", "Failed to initialise script engine -- aborting");
    return false;
}

// Register the ECS World so Lua scripts can read/write transforms.
// world must remain valid for the lifetime of scriptEngine.
scriptEngine.setWorld(&world);

// Load the main game script from the assets directory.
// path is relative; assetRoot is the absolute scripts directory.
const char* assetRoot = "/home/user/mygame/assets/scripts";
if (!scriptEngine.doFile("main.lua", assetRoot)) {
    FFE_LOG_ERROR("Game", "Failed to load main.lua");
    // Decide whether this is fatal. doFile logs the specific error.
}
```

### 2. doString for in-game events (tooltip text, dialogue, NPC responses)

`doString` is appropriate for short, known-safe Lua fragments generated at load time -- not per-frame. A common pattern is evaluating dialogue strings or trigger conditions that were authored in a data file and loaded into memory at scene load.

```cpp
// At scene load time -- NOT per-frame.
// For example, evaluating a trigger condition stored as a string in a level file.
const char* triggerScript = "if playerHealth < 20 then ffe.log('Low health warning') end";

if (!scriptEngine.doString(triggerScript)) {
    // doString already logged the error. Decide whether to skip or abort.
    FFE_LOG_WARN("Game", "Trigger script failed -- skipping trigger");
}
```

Do NOT use `doString` to run scripts that were provided at runtime by an end user without validation -- the sandbox protects the OS, but the engine still spends CPU compiling and running the script.

### 3. Lua-side ffe.log usage (what game devs write in their scripts)

Game scripts use `ffe.log` to emit diagnostic output. The message appears in the engine log under the "Lua" category at INFO level.

```lua
-- main.lua
ffe.log("main.lua loaded successfully")

local function onEnemySpawned(enemyId)
    ffe.log("Enemy spawned: " .. tostring(enemyId))
end

-- ffe.log accepts any value coerced to string by tostring():
local score = 42
ffe.log("Current score: " .. tostring(score))
```

### 4. Error handling -- what to do when doString or doFile returns false

Both `doString` and `doFile` return `false` on failure and log the specific Lua error via `FFE_LOG_ERROR`. Your code only needs to decide what to do next -- you do not need to re-read the error yourself.

```cpp
// Example: load optional config script; fall back to defaults if it fails.
const bool ok = scriptEngine.doFile("config.lua", assetRoot);
if (!ok) {
    // The engine already logged: "Lua compile error: ..." or "Lua runtime error: ..."
    // Apply hardcoded defaults instead.
    FFE_LOG_WARN("Game", "config.lua failed to load -- using built-in defaults");
    applyDefaultConfig();
}

// Example: treat a missing required script as a fatal error.
if (!scriptEngine.doFile("game_rules.lua", assetRoot)) {
    FFE_LOG_ERROR("Game", "game_rules.lua is required -- cannot start");
    return false;
}
```

### 5. Per-frame Lua callbacks using callFunction

Define a named function in Lua at load time, then call it every frame from a C++ system using `callFunction`. This is the correct, zero-compile-overhead pattern for per-frame game logic.

**Lua side (loaded once at startup via doFile):**

```lua
-- game_logic.lua
local SPEED = 200.0

-- Pre-allocate a reusable table ONCE at load time (outside the function).
-- fillTransform writes into this table every frame — zero GC pressure.
local t = {}

function update(entityId, dt)
    if not ffe.fillTransform(entityId, t) then return end

    local dx = 0.0
    local dy = 0.0

    if ffe.isKeyHeld(ffe.KEY_D) then dx = dx + SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_A) then dx = dx - SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_S) then dy = dy + SPEED * dt end
    if ffe.isKeyHeld(ffe.KEY_W) then dy = dy - SPEED * dt end

    ffe.setTransform(entityId, t.x + dx, t.y + dy, t.rotation, t.scaleX, t.scaleY)

    if ffe.isKeyPressed(ffe.KEY_ESCAPE) then
        ffe.requestShutdown()
    end
end
```

**C++ side (called per frame from a system):**

```cpp
// Load the script once at startup:
scriptEngine.doFile("game_logic.lua", assetRoot);

// In a C++ system or per-frame update:
// callFunction looks up "update" by name (lua_getglobal), pushes the
// two arguments, and calls lua_pcall. No string construction or compilation.
scriptEngine.callFunction("update", static_cast<ffe::i64>(playerEntity), dt);
```

**Key rule:** `doFile` compiles the function body once. `callFunction` only looks it up and invokes it. The ratio is: 1× `doFile` at load time, N× `callFunction` at 60 fps. Never call `doString("update(...)")` per frame.

### 6. Creating an entity from Lua with Transform, Sprite, and interpolation

This pattern shows how to spawn a game entity entirely from a Lua script. The texture handle is an opaque integer obtained from C++ and passed into the script as an argument or a Lua global set before the script runs.

**C++ side — pass the texture handle to the script as a global:**

```cpp
// After loading the script with doFile, set a global the script can read.
// texHandle.id is the u32 from rhi::TextureHandle.
lua_State* L = ...;  // do not do this from outside the scripting subsystem in production code
// Preferred: expose a C++ binding (e.g., ffe.loadTexture) that returns the handle.
// For this example, set a known handle as a Lua global before calling the spawn function.
scriptEngine.doString("PLAYER_TEX = 1");  // handle 1 is the white default texture

// Load the script that defines spawnPlayer():
scriptEngine.doFile("spawn.lua", assetRoot);

// Spawn the player by calling a named Lua function.
// The entity ID produced is returned via callFunction's return path if needed,
// or stored in a Lua global by the script itself.
scriptEngine.callFunction("spawnPlayer", 0 /*unused entity arg*/, 0.0 /*dt*/);
```

**Lua side (spawn.lua):**

```lua
-- spawn.lua
-- PLAYER_TEX must be set by C++ before this script runs.

local function spawnPlayer()
    -- Create the entity. Returns nil if the ECS is full.
    local id = ffe.createEntity()
    if id == nil then
        ffe.log("spawnPlayer: ECS is full — could not create player entity")
        return
    end

    -- Add a Transform at position (100, 200), no rotation, scale 1x1.
    local ok = ffe.addTransform(id, 100.0, 200.0, 0.0, 1.0, 1.0)
    if not ok then
        ffe.log("spawnPlayer: addTransform failed for entity " .. tostring(id))
        ffe.destroyEntity(id)
        return
    end

    -- Add a Sprite using the texture handle set by C++.
    -- Width=32, height=32, white tint (r=1,g=1,b=1,a=1), layer 0.
    ok = ffe.addSprite(id, PLAYER_TEX, 32.0, 32.0, 1.0, 1.0, 1.0, 1.0, 0)
    if not ok then
        ffe.log("spawnPlayer: addSprite failed for entity " .. tostring(id))
        ffe.destroyEntity(id)
        return
    end

    -- Opt in to render interpolation. Call this AFTER addTransform so that
    -- PreviousTransform is initialised from the Transform we just set.
    ok = ffe.addPreviousTransform(id)
    if not ok then
        ffe.log("spawnPlayer: addPreviousTransform failed for entity " .. tostring(id))
        -- Not fatal — entity still renders; interpolation just won't work.
    end

    ffe.log("spawnPlayer: created entity " .. tostring(id))
    -- Store in a module-level variable for later use.
    _G.playerEntityId = id
end

-- Top-level call: this runs when callFunction("spawnPlayer", ...) is invoked.
spawnPlayer()
```

**Key sequencing rule:** always call `addTransform` before `addPreviousTransform`. If `addPreviousTransform` runs first, it initialises to zero position and logs a warning, so the first interpolated frame will have an incorrect starting position.

### 7. Using the shutdown() callback to release resources before the Lua state closes

Define a global `shutdown` function in your script to clean up textures and other resources when the engine exits. `ScriptEngine::shutdown()` calls it automatically — you do not need to invoke it yourself.

```lua
-- game_logic.lua
-- Loaded once at startup via doFile.

local playerTex = ffe.loadTexture("sprites/player.png")

function update(entityId, dt)
    -- ... game logic ...
end

function shutdown()
    -- Called by ScriptEngine::shutdown() before the Lua state is closed.
    -- Use this to unload textures and release any other resources your script owns.
    if playerTex then
        ffe.unloadTexture(playerTex)
        playerTex = nil
    end
    ffe.log("Script shutdown complete")
end
```

`shutdown()` is optional. If it is not defined, `ScriptEngine::shutdown()` proceeds silently. If `shutdown()` raises a Lua error, the error is logged and the engine continues to close the Lua state — the error does not abort shutdown.

### 8. Scene transitions with destroyAllEntities and loadScene

Use `ffe.destroyAllEntities()` to wipe the current scene state, then `ffe.loadScene()` to load a new scene script. The new script defines its own `update()` function, which replaces the previous one as a Lua global.

**C++ side -- set up the script root once at startup:**

```cpp
// In main.cpp, after scriptEngine.init() and before doFile():
scriptEngine.setScriptRoot("/home/user/mygame/assets/scripts");
```

**Lua side -- title screen that transitions to gameplay:**

```lua
-- title.lua (loaded as the initial script)
local titleTex = ffe.loadTexture("textures/title.png")
local titleEntity = ffe.createEntity()
ffe.addTransform(titleEntity, 640, 360, 0, 1, 1)
ffe.addSprite(titleEntity, titleTex, 800, 600, 1, 1, 1, 1, 0)

function update(entityId, dt)
    ffe.drawText("PRESS ENTER TO START", 400, 500, 3, 1, 1, 1, 1)
    if ffe.isKeyPressed(ffe.KEY_ENTER) then
        -- Clean up: unload title-only assets, destroy all entities, load gameplay
        ffe.unloadTexture(titleTex)
        titleTex = nil
        ffe.destroyAllEntities()
        ffe.loadScene("gameplay.lua")
    end
end
```

```lua
-- gameplay.lua (loaded by ffe.loadScene from title.lua)
local playerTex = ffe.loadTexture("textures/player.png")
local player = ffe.createEntity()
ffe.addTransform(player, 400, 300, 0, 1, 1)
ffe.addSprite(player, playerTex, 32, 32, 1, 1, 1, 1, 0)

function update(entityId, dt)
    -- gameplay logic here
    -- This update() replaces title.lua's update() as the Lua global
end
```

**What `destroyAllEntities` resets:** all entities/components, all timers, the collision callback. **What it does NOT reset:** Lua globals, loaded textures/sounds, background color, camera shake, music playback. Unload assets explicitly if needed.

### 8b. Loading editor-exported JSON scenes with loadSceneJSON

Use `ffe.loadSceneJSON()` to load a scene file that was exported by the editor's build pipeline. Unlike `ffe.loadScene()` (which executes a Lua script), `loadSceneJSON()` deserialises a JSON scene file into the World. It clears all existing entities first.

```lua
-- main.lua (generated by the editor's exporter)
-- Load the entry scene exported from the editor
local ok = ffe.loadSceneJSON("scenes/level1.json")
if not ok then
    ffe.log("ERROR: failed to load level1.json")
end

function update(entityId, dt)
    -- gameplay logic for the loaded scene
    if ffe.isKeyPressed(ffe.KEY_N) then
        -- Transition to another JSON scene
        local ok2 = ffe.loadSceneJSON("scenes/level2.json")
        if not ok2 then
            ffe.log("ERROR: failed to load level2.json")
        end
    end
end
```

**Key difference from `ffe.loadScene()`:** `loadScene` executes a Lua script (which defines entities and logic procedurally). `loadSceneJSON` deserialises a JSON file (which was serialised from the editor). Use `loadScene` for hand-written Lua scene scripts; use `loadSceneJSON` for editor-exported scenes.

## What NOT to Do

### Never call doString() or doFile() per-frame or per-entity

Both functions compile Lua bytecode and execute it. Calling them inside a system update -- even once per frame -- will destroy performance. They are not lightweight function calls; `doString` calls `luaL_loadstring` (compiler) followed by `lua_pcall` (executor). Call them at load time, in response to one-shot events, or during scene transitions. Never inside a hot loop.

```cpp
// BAD -- called per frame: doString compiles Lua bytecode every call
void mySystem(ffe::World& world, float dt) {
    scriptEngine.doString("updateAI()");  // WRONG: compiles every frame
}

// GOOD -- define the function once at startup; call it per frame via callFunction
// At startup:
scriptEngine.doFile("ai.lua", assetRoot);   // compiles "function updateAI(...)" once

// Per frame in a C++ system:
scriptEngine.callFunction("updateAI", static_cast<ffe::i64>(entityId), dt);
// callFunction uses lua_getglobal + lua_pcall — no compile step, no string construction.
```

### Do not call ffe.loadTexture() per frame

`ffe.loadTexture` triggers file I/O on disk and a GPU texture upload. Calling it per frame is a severe performance violation: one disk read + GPU upload every 16 ms will exhaust GPU memory, stall the render pipeline, and drop frame rate to unusable levels on any hardware tier. Load textures once at scene or application startup, keep the integer handle, and reuse it.

```lua
-- BAD -- called every frame inside an update function
function update(entityId, dt)
    local tex = ffe.loadTexture("sprites/player.png")  -- WRONG: file I/O + GPU upload every frame
    ffe.addSprite(entityId, tex, 32, 32, 1, 1, 1, 1, 0)
end

-- GOOD -- loaded once at startup; handle reused per frame
-- At scene init (called once):
_G.playerTex = ffe.loadTexture("sprites/player.png")

-- Per event (not per frame):
ffe.addSprite(entityId, _G.playerTex, 32, 32, 1, 1, 1, 1, 0)
```

### Do not open additional Lua libraries (io, os, package, debug, ffi, jit)

The sandbox is constructed by opening only safe libraries. If you open `io`, `os`, or `package` after `init()`, scripts gain access to the filesystem, OS commands, and `require`. This defeats the entire security model and breaks the promise to game developers that their scripts are sandboxed. The instruction budget also does not protect against file I/O blocking.

```cpp
// BAD -- opens os library, breaks sandbox
lua_State* L = ...; // do not extract the Lua state
luaopen_os(L);      // scripts can now call os.execute("rm -rf /")

// GOOD -- do not touch the Lua state directly; use only ScriptEngine's public API
```

(The Lua state is stored as `void*` in the header intentionally -- you are not meant to cast it back and call raw Lua C API functions from outside the scripting subsystem.)

### Do not expose raw pointers to Lua

If you add bindings that hand C++ objects to Lua scripts, never pass a raw pointer. A script could hold a stale pointer after the object is freed, producing use-after-free. Use opaque integer handles (like `EntityId`) that the engine validates before use.

```lua
-- BAD (hypothetical binding that passes raw pointer)
local entity = ffe.getRawPointer(42)  -- would be a use-after-free risk

-- GOOD (opaque handle -- engine validates before use)
local entity = ffe.getEntity(42)  -- engine checks validity before touching memory
```

### Do not call doString() or doFile() before init()

Calling either function before `init()` returns `false` immediately and logs an error. The Lua state does not exist until `init()` succeeds.

```cpp
// BAD
ffe::ScriptEngine engine;
engine.doFile("main.lua", assetRoot);  // init() never called -- logs error, returns false

// GOOD
ffe::ScriptEngine engine;
if (!engine.init()) { return false; }
engine.doFile("main.lua", assetRoot);
```

### Do not call setWorld() before init()

`setWorld()` requires an initialised Lua state. Calling it before `init()` logs an error and has no effect.

```cpp
// BAD
ffe::ScriptEngine engine;
engine.setWorld(&world);  // logs error, does nothing -- no Lua state yet

// GOOD
ffe::ScriptEngine engine;
if (!engine.init()) { return false; }
engine.setWorld(&world);
engine.doFile("main.lua", assetRoot);
```

### Do not try to work around the sandbox

The sandbox blocks specific Lua globals and standard libraries for security reasons documented in ADR-004. If you need a capability that the sandbox blocks -- such as reading a file from Lua -- implement it as a new C++ binding in `registerEcsBindings()` with proper path validation, and expose it through the `ffe.*` table. Do not try to re-open blocked libraries or restore removed globals. They are removed, not just hidden.

### Do not cache the result of getTransform across frames

`getTransform` returns a copy of the Transform data at the time of the call, not a live reference. Caching it across frames means you are working with stale data.

```lua
-- BAD -- cached result is stale next frame
local t = ffe.getTransform(entityId)  -- called once
-- ... later that frame or next frame:
ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
-- t.x might not reflect changes made by other systems since the last call

-- GOOD -- always fetch fresh data at the point of use
local t = ffe.getTransform(entityId)
if t == nil then return end
ffe.setTransform(entityId, t.x + dx, t.y + dy, t.rotation, t.scaleX, t.scaleY)
```

### Do not pass entity IDs across ScriptEngine instances

Entity IDs are only valid within the World they were created in. Passing an ID from one ScriptEngine's World to another ScriptEngine will either silently operate on the wrong entity or return nil.

### Do not pass NaN or math.huge to setTransform

Non-finite values (`math.huge`, `-math.huge`, `0/0`) are rejected by `setTransform` and logged as errors at INFO level. The transform is not modified. This protects against silent state corruption.

```lua
-- BAD -- NaN propagation silently corrupts position
ffe.setTransform(id, 0/0, 0, 0, 1, 1)   -- rejected and logged

-- GOOD -- validate values before setting
local newX = computeX()
if newX ~= newX then return end  -- NaN check: NaN ~= NaN is true in Lua
ffe.setTransform(id, newX, 0, 0, 1, 1)
```

### Do not call set3DCameraFPS or set3DCameraOrbit with hardcoded angles every frame

Both functions compute a target point from angle inputs and write it to the camera. Calling them with constant values every frame is valid but achieves nothing — the camera never rotates. The correct pattern is to accumulate yaw/pitch in Lua variables and update them from input (mouse delta, joystick, arrow keys) each frame.

```lua
-- BAD -- hardcoded angles; camera is static even though the function is called every frame
function update(entityId, dt)
    ffe.set3DCameraFPS(0, 1.8, 0, 45, 10)     -- WRONG: same angles every frame
    ffe.set3DCameraOrbit(0, 0, 0, 6, 30, 20)  -- WRONG: same angles every frame
end

-- GOOD -- accumulate angles in variables, update from input
local yaw, pitch = 0, 0
local camX, camY, camZ = 0, 1.8, 0

function update(entityId, dt)
    local speed = 60  -- degrees per second
    if ffe.isKeyHeld(ffe.KEY_LEFT)  then yaw   = yaw   - speed * dt end
    if ffe.isKeyHeld(ffe.KEY_RIGHT) then yaw   = yaw   + speed * dt end
    if ffe.isKeyHeld(ffe.KEY_UP)    then pitch  = pitch  + speed * dt end
    if ffe.isKeyHeld(ffe.KEY_DOWN)  then pitch  = pitch  - speed * dt end
    ffe.set3DCameraFPS(camX, camY, camZ, yaw, pitch)
end
```

### Do not call ffe.playAnimation3D() every frame

`playAnimation3D` resets playback time to 0 on each call. Calling it every frame means the animation restarts from the beginning each tick and never advances past the first frame. Call it once to start playback, then let `animationUpdateSystem3D` advance the animation automatically.

```lua
-- BAD -- resets animation to frame 0 every tick; animation appears frozen
function update(entityId, dt)
    ffe.playAnimation3D(entity, 0, true)  -- WRONG: called every frame
end

-- GOOD -- call once at init; animation advances automatically
ffe.playAnimation3D(entity, 0, true)  -- called once

function update(entityId, dt)
    -- animationUpdateSystem3D handles playback progression
end
```

### Do not call ffe.loadScene() per frame

`loadScene` calls `doFile` internally, which compiles and executes a Lua script. It is a cold-path operation for scene transitions, not a per-frame call. Calling it per frame will recompile and re-execute the entire scene script at 60Hz, destroying performance.

```lua
-- BAD -- called every frame
function update(entityId, dt)
    if gameState == "menu" then
        ffe.loadScene("menu.lua")  -- WRONG: recompiles + executes at 60 fps
    end
end

-- GOOD -- called once on state transition
function update(entityId, dt)
    if gameState == "gameplay" and ffe.isKeyPressed(ffe.KEY_ESCAPE) then
        gameState = "menu"
        ffe.destroyAllEntities()
        ffe.loadScene("menu.lua")  -- called once, on the frame ESC is pressed
    end
end
```

### Do not call ffe.loadSceneJSON() per frame

Same principle as `ffe.loadScene()`. `loadSceneJSON` clears all entities and deserialises a JSON file. It is a cold-path operation for scene transitions. Calling it per frame will destroy and recreate the entire scene at 60 Hz.

```lua
-- BAD
function update(entityId, dt)
    ffe.loadSceneJSON("scenes/level1.json")  -- WRONG: clears + rebuilds every frame
end

-- GOOD
function update(entityId, dt)
    if ffe.isKeyPressed(ffe.KEY_N) then
        ffe.loadSceneJSON("scenes/level2.json")  -- called once on key press
    end
end
```

### Do not hold texture handles across scene transitions where C++ may unload them (dangling handle risk)

`ffe.addSprite` accepts a texture handle as an opaque integer. The integer is the `u32 id` field of a `rhi::TextureHandle` created on the C++ side. The scripting layer cannot verify whether the GPU texture behind the handle is still live — it only validates that the integer is in the valid range (`> 0` and `<= UINT32_MAX`).

If C++ calls `unloadTexture` (or equivalent) on a texture while a `Sprite` component that was set from Lua still references that handle, the sprite will attempt to render with a dangling handle. On the headless renderer this silently draws nothing. On a real GPU renderer, behaviour is undefined and may produce visual corruption or a GPU error.

**The rule:** never unload a texture while any entity has a `Sprite` component that references its handle. This applies whether the sprite was created from C++ or from Lua.

```lua
-- BAD: storing a texture handle across a scene boundary where C++ might unload it
local myTexHandle = 42    -- obtained from a C++ binding at scene load
-- ... later, after a scene transition that unloaded the texture ...
ffe.addSprite(id, myTexHandle, 32, 32, 1, 1, 1, 1, 0)  -- dangling handle!

-- GOOD: texture handles only used within the lifetime guaranteed by C++.
-- Obtain fresh handles at scene load, stop using them before scene teardown.
-- Do not hold texture handle integers in Lua globals that survive scene transitions.
```

If your game uses scene loading/unloading, make the following explicit in your C++ scene teardown code:
1. Remove (or destroy) all entities that have `Sprite` components referencing the textures you are about to unload.
2. Only then call `unloadTexture` (or the equivalent in your asset manager).
3. Re-create the sprites in the new scene after the new textures are loaded.

### Do not use shutdown() for per-level cleanup or assume it runs when the player presses ESC

`shutdown()` is called exactly once: when `ScriptEngine::shutdown()` is explicitly invoked, which happens when the `Application` exits (the engine is closing). It is NOT called when the player presses ESC, when the game loop stops temporarily, or when a scene is unloaded mid-session.

Use `shutdown()` only for final, application-level cleanup — releasing textures and other resources that your script holds for the entire session lifetime.

For per-level cleanup (unloading scene-specific textures when transitioning between levels), call a named Lua function explicitly from your C++ scene manager via `callFunction`. Do not rely on `shutdown()` for this — it will not be called until the application fully exits.

```lua
-- BAD: treating shutdown() as a per-level teardown hook
function shutdown()
    -- This will NOT be called when a level ends and a new one loads.
    -- It is only called when Application exits entirely.
    unloadLevelTextures()  -- called too late for mid-session scene transitions
end

-- GOOD: per-level cleanup via an explicitly named function
function onLevelUnload()
    -- Called from C++ scene management: scriptEngine.callFunction("onLevelUnload", ...)
    unloadLevelTextures()
end

-- GOOD: final application cleanup
function shutdown()
    -- Only session-lifetime resources belong here.
    if _G.uiAtlasTex then
        ffe.unloadTexture(_G.uiAtlasTex)
        _G.uiAtlasTex = nil
    end
    ffe.log("Application shutdown complete")
end
```

### Use fillTransform, not getTransform, in per-frame update loops

`getTransform` allocates a new Lua table on every call. At 60 fps with multiple entities, this creates significant GC pressure — the Lua garbage collector must eventually sweep all those short-lived tables, causing frame-time spikes. `fillTransform` writes into a caller-provided table, producing zero allocations per call.

```lua
-- BAD — getTransform allocates a new table every frame per entity
function update(entityId, dt)
    local t = ffe.getTransform(entityId)  -- new table allocation every call
    if t == nil then return end
    ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
end

-- GOOD — fillTransform reuses a pre-allocated table (zero GC pressure)
local t = {}  -- allocated once at load time

function update(entityId, dt)
    if not ffe.fillTransform(entityId, t) then return end
    ffe.setTransform(entityId, t.x + 1, t.y, t.rotation, t.scaleX, t.scaleY)
end
```

`getTransform` is still appropriate for one-shot queries (scene init, event handlers, debug inspection) where allocation cost is negligible. Use `fillTransform` whenever the call is inside a per-frame or per-entity loop.

### Do not call ffe.loadSound() per frame

`ffe.loadSound` triggers file I/O on disk and decodes the entire audio file into a PCM buffer. Calling it per frame is a severe performance violation — identical to the `ffe.loadTexture` anti-pattern. Load sounds once at scene or application startup, keep the integer handle, and reuse it.

```lua
-- BAD — called every frame
function update(entityId, dt)
    local sfx = ffe.loadSound("sfx/jump.wav")  -- WRONG: file I/O + decode every frame
    ffe.playSound(sfx)
end

-- GOOD — loaded once at startup; handle reused
local jumpSfx = ffe.loadSound("sfx/jump.wav")

function update(entityId, dt)
    if ffe.isKeyPressed(ffe.KEY_SPACE) and jumpSfx then
        ffe.playSound(jumpSfx)
    end
end
```

### Do not pass negative texture handles to setMeshNormalMap or setMeshSpecularMap

Negative handles are rejected with a warning. Pass `0` to disable the map, or a positive handle from `ffe.loadTexture`.

```lua
-- BAD
ffe.setMeshNormalMap(entity, -1)  -- rejected, warning logged

-- GOOD
ffe.setMeshNormalMap(entity, 0)           -- disable normal mapping
ffe.setMeshNormalMap(entity, myNormTex)   -- positive handle from ffe.loadTexture
```

### Do not call ffe.addPointLight per frame to move a light

Use `setPointLightPosition` to update position, `setPointLightColor` to update color, or `setPointLightRadius` to update radius. `addPointLight` sets all properties at once and enables the light — it is a setup call, not a per-frame call.

```lua
-- BAD — re-sets all properties every frame just to move it
function update(entityId, dt)
    ffe.addPointLight(0, newX, newY, newZ, 1, 1, 1, 10)  -- wasteful
end

-- GOOD — update only what changed
function update(entityId, dt)
    ffe.setPointLightPosition(0, newX, newY, newZ)
end
```

### Do not exceed 8 point lights (indices 0-7)

The point light array is a fixed 8-slot array. Index 8 and above are rejected with a warning. If you need more dynamic lights, consider baking static lighting or combining fewer, higher-radius lights.

### Do not call ffe.enableShadows() every frame

`enableShadows` performs a no-op check when already enabled at the requested resolution, but calling it per-frame is wasteful and a code smell. Call it once in `init()` or a settings handler.

```lua
-- BAD — called every frame
function update(entityId, dt)
    ffe.enableShadows(1024)  -- pointless per-frame call
end

-- GOOD — called once at init
function init()
    ffe.enableShadows(1024)
end
```

### Do not call ffe.loadSkybox() every frame

`loadSkybox` reads 6 image files from disk and uploads a cubemap to the GPU. Calling it per-frame will destroy performance. Load the skybox once in `init()`.

```lua
-- BAD -- loading 6 images from disk every frame
function update(entityId, dt)
    ffe.loadSkybox("skybox/right.png", "skybox/left.png", ...)
end

-- GOOD -- load once at init
function init()
    ffe.loadSkybox("skybox/right.png", "skybox/left.png",
        "skybox/top.png", "skybox/bottom.png",
        "skybox/front.png", "skybox/back.png")
end
```

### Do not forget to call ffe.unloadSkybox() in shutdown

The cubemap texture is a scene-global GPU resource, not an entity component. It is not cleaned up by `ffe.destroyAllEntities()`. Call `ffe.unloadSkybox()` explicitly in your `shutdown()` callback or before loading a new scene.

### Do not call ffe.castRayAll() in a per-entity per-frame loop

`castRayAll` allocates a Lua table per call. Use `ffe.castRay()` (8 return values, zero allocation) for per-frame single-hit checks. Reserve `castRayAll` for event-driven queries like weapon fire.

### Do not call ffe.onCollision3D() every frame

The callback is replaced each time. Register it once at init time. Re-registering per frame churns Lua registry references and creates new closures needlessly.

### Do not call ffe.dispatchCollision3DEvents() from game scripts

This is an engine-internal function exposed for testing. The engine loop calls it automatically. Calling it from Lua re-delivers the same events, causing duplicate callbacks.

### Do not pass skybox face images in the wrong order

The parameter order is: right (+X), left (-X), top (+Y), bottom (-Y), front (+Z), back (-Z). Swapping any faces produces visible seams and incorrect reflections at cubemap edges.

### Do not call getTransform/setTransform in tight inner loops

The World pointer registry lookup happens on every `getTransform`/`setTransform` call (one `lua_gettable` on the Lua registry per call). Additionally, `getTransform` allocates a new Lua table on every call, creating GC pressure. For high-frequency entity updates, batch your script updates: update multiple entities per Lua call rather than making one Lua call per entity.

## Security

The sandbox applies two defence layers:

**Layer 1 -- Library blocking:** Only `base` (partial), `math`, `string`, and `table` are opened. `io`, `os`, `package`, `debug`, `ffi`, and `jit` are never opened. This is done by calling `luaopen_*` selectively, not by opening everything and removing it afterwards.

**Layer 2 -- Global removal:** Dangerous globals brought in by `luaopen_base` are explicitly set to nil after the library is opened:

| Global removed | Reason |
|----------------|--------|
| `dofile`, `loadfile` | Direct filesystem access |
| `load` | CRITICAL: can load Lua bytecode with arbitrary upvalues, bypassing string-based sandbox |
| `loadstring` | LuaJIT/Lua 5.1 alias for `load` -- same risk |
| `require` | Package system access |
| `collectgarbage` | DoS via GC pressure |
| `rawset`, `rawget` | Bypass `__newindex`/`__index` metamethod guards |
| `rawequal`, `rawlen` | Related raw access |
| `setmetatable`, `getmetatable` | Sandbox escape via metamethods |
| `debug` | Covered by library blocking but also explicitly nil'd |

**Instruction budget:** 1,000,000 instructions per `doString`/`doFile` call. This is set via `lua_sethook` before any libraries are opened. It is active in all build configurations (not just debug). Exceeding the budget raises a Lua error, caught by `pcall` -- the engine does not crash.

**ECS binding input validation:** All entity IDs received from Lua are validated before use: negative values and out-of-range IDs (above `UINT32_MAX - 1`) are rejected immediately. `world->isValid()` is checked before any component access. Non-finite float values passed to `setTransform` and `addTransform` are rejected and logged. `addSprite` rejects texture handles `<= 0` and non-finite or non-positive width/height values. `addSprite` clamps color channels to `[0.0, 1.0]` (NaN treated as 0.0) and layer to `[0, 15]`. `addPreviousTransform` warns when called without a prior `addTransform`. `addSpriteAnimation` validates that `frameCount` is in `[1, 65535]`, `columns` is in `[1, frameCount]`, and `frameTime` is positive and finite — invalid values return `false` and log an error. `setAnimationFrame` clamps the frame index to `[0, frameCount - 1]`.

**Collision binding input validation:** `ffe.addCollider` validates the shape string (must be `"aabb"` or `"circle"`), rejects non-finite or non-positive `halfWidth`, and rejects non-finite or negative `halfHeight`. Layer and mask values are clamped to `[0, 65535]` with a warning if out of range. If the entity already has a `Collider2D`, the component is overwritten via `emplace_or_replace` and a warning is logged. `ffe.removeCollider` is a safe no-op for entities without a collider. `ffe.setCollisionCallback` validates that the argument is a function (or `nil` to unregister); non-function values raise a Lua error via `luaL_checktype`.

**Audio binding input validation:** `ffe.loadSound` applies the same type guard as `ffe.loadTexture` -- non-string arguments return `nil` immediately without calling into C++ (prevents silent coercion of numbers/booleans to strings). Path traversal and absolute paths are rejected by the C++ audio layer. `ffe.unloadSound` and `ffe.playSound` validate that the handle is a positive integer in `(0, UINT32_MAX]`; out-of-range handles are logged and ignored. Scripts cannot control the asset root -- it is always the engine's internal root set by C++.

Game developers should not attempt to work around the sandbox. If a script requires a capability the sandbox blocks, implement it as a new `ffe.*` C++ binding with appropriate validation.
