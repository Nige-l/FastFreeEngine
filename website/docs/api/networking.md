# engine/networking -- System Context

## System Purpose

The networking subsystem provides a complete client-server multiplayer stack for FFE. It is built in three layers: (1) a binary packet format with bounds-checked serialisation (`PacketReader`/`PacketWriter`), (2) an ENet-based transport layer with per-connection rate limiting (`ServerTransport`/`ClientTransport`), and (3) a high-level game networking layer with automatic ECS replication, snapshot interpolation, and game messaging (`NetworkServer`/`NetworkClient`/`network_system`). The system follows a "server XOR client" model -- only one mode is active at a time. All external input (network packets) is treated as untrusted and validated before processing. No virtual functions, no `std::function`, and no per-frame heap allocations in hot paths. This subsystem has no GPU involvement and runs on all tiers from LEGACY upward.

### Lag Compensation (`networking/lag_compensation.h`)

Server-side lag compensation: records entity position history and performs rewound hit checks (ray-vs-sphere) at past ticks. Used for shooter-style games where the server must verify hits at the position the shooter actually saw.

#### Constants

| Constant | Value | Description |
|----------|-------|-------------|
| `MAX_HISTORY_TICKS` | `64` | Maximum history frames (~3.2 s at 20 Hz) |
| `MAX_TRACKED_ENTITIES` | `256` | Maximum entities tracked per frame |

#### EntityState Struct

Snapshot of one entity's position and collision radius.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `entityId` | `uint32_t` | `0xFFFFFFFF` | Entity identifier (invalid sentinel = `0xFFFFFFFF`) |
| `x` | `float` | `0.0f` | World X position |
| `y` | `float` | `0.0f` | World Y position |
| `z` | `float` | `0.0f` | World Z position |
| `radius` | `float` | `0.5f` | Collision sphere radius |
| `active` | `bool` | `false` | Whether this entity is active |

#### HistoryFrame Struct

All tracked entities at a single tick.

| Field | Type | Description |
|-------|------|-------------|
| `tick` | `uint32_t` | Network tick number |
| `entityCount` | `uint32_t` | Number of active entities in this frame |
| `entities` | `EntityState[256]` | Entity state array |

#### HitCheckResult Struct

Outcome of a lag-compensated ray-vs-sphere check.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `hit` | `bool` | `false` | Whether a hit was found |
| `hitEntityId` | `uint32_t` | `0xFFFFFFFF` | Entity that was hit (invalid if no hit) |
| `hitX` | `float` | `0.0f` | World X of the hit point |
| `hitY` | `float` | `0.0f` | World Y of the hit point |
| `hitZ` | `float` | `0.0f` | World Z of the hit point |
| `distance` | `float` | `0.0f` | Distance from ray origin to hit point |

#### HitConfirmFn Callback

| Type | Signature | Description |
|------|-----------|-------------|
| `HitConfirmFn` | `void(*)(uint32_t shooterEntityId, const HitCheckResult& result, void*)` | Called when a hit check is confirmed server-side |

#### LagCompensator Class

Circular buffer of `HistoryFrame` structs with ray-vs-sphere hit checking against rewound positions.

| Method | Description |
|--------|-------------|
| `void recordFrame(uint32_t tick, const EntityState* entities, uint32_t count)` | Record entity positions for a tick. Call every network tick on the server BEFORE applying new inputs. |
| `HitCheckResult performHitCheck(uint32_t atTick, float originX, float originY, float originZ, float dirX, float dirY, float dirZ, float maxDistance, uint32_t ignoreEntityId) const` | Rewind to `atTick` and perform a ray-vs-sphere check. Returns the closest hit. Ignores `ignoreEntityId` (typically the shooter). |
| `const HistoryFrame* getFrame(uint32_t tick) const` | Get the frame at a specific tick, or `nullptr` if not in history. |
| `uint32_t oldestTick() const` | Oldest tick still in the buffer. |
| `uint32_t newestTick() const` | Newest tick in the buffer. |
| `uint32_t frameCount() const` | Number of frames currently stored. |
| `void setMaxRewindTicks(uint32_t ticks)` | Set maximum rewind depth. Clamped to `MAX_HISTORY_TICKS`. |
| `uint32_t maxRewindTicks() const` | Current maximum rewind depth. |
| `void clear()` | Clear all history. |

## Common Usage Patterns

### 1. Lua: Starting a Multiplayer Server

```lua
function ffe.onInit()
    -- Start server on port 7777
    local ok = ffe.startServer(7777)
    if not ok then
        print("Failed to start server!")
        return
    end

    -- Handle player connections
    ffe.onClientConnected(function(clientId)
        print("Player " .. clientId .. " joined!")
        -- Spawn a player entity for them
    end)

    ffe.onClientDisconnected(function(clientId)
        print("Player " .. clientId .. " left!")
        -- Clean up their entities
    end)

    -- Handle game messages from clients
    ffe.onNetworkMessage(function(senderId, msgType, data)
        if msgType == 1 then  -- e.g., chat message
            print("Player " .. senderId .. " says: " .. data)
        end
    end)
end
```

### 4. C++: Registering Custom Components for Replication

```cpp
#include "networking/replication.h"
#include "networking/network_system.h"
using namespace ffe::networking;

// Custom component
struct Health { float current; float max; };
constexpr uint16_t COMPONENT_ID_HEALTH = 10;
constexpr uint16_t HEALTH_SERIALIZED_SIZE = 8; // 2 floats

uint16_t serializeHealth(const void* comp, uint8_t* buf, uint16_t maxSize) {
    if (maxSize < HEALTH_SERIALIZED_SIZE) return 0;
    const auto* h = static_cast<const Health*>(comp);
    std::memcpy(buf, &h->current, 4);
    std::memcpy(buf + 4, &h->max, 4);
    return HEALTH_SERIALIZED_SIZE;
}

bool deserializeHealth(void* comp, const uint8_t* buf, uint16_t size) {
    if (size < HEALTH_SERIALIZED_SIZE) return false;
    auto* h = static_cast<Health*>(comp);
    std::memcpy(&h->current, buf, 4);
    std::memcpy(&h->max, buf + 4, 4);
    return true;
}

// Register before starting server/client
auto& reg = getReplicationRegistry();
reg.registerComponent(COMPONENT_ID_HEALTH, HEALTH_SERIALIZED_SIZE,
                      serializeHealth, deserializeHealth, nullptr);
```

### 5. Lua: Client-Side Prediction (Sending Input)

```lua
function ffe.onInit()
    ffe.onConnected(function()
        -- Create local player entity
        local id = ffe.createEntity()
        ffe.addTransform(id, 0, 0, 0, 1, 1)
        ffe.addPreviousTransform(id)
        ffe.addSprite(id, 1, 40, 40, 0.3, 0.7, 1.0, 1.0, 1)

        -- Tell the prediction system which entity is ours
        ffe.setLocalPlayer(id)
    end)

    ffe.connectToServer("127.0.0.1", 7777)
end

local INPUT_UP    = 1
local INPUT_DOWN  = 2
local INPUT_LEFT  = 4
local INPUT_RIGHT = 8
local SPEED       = 250.0

function update(entityId, dt)
    -- Build input bits from keyboard state
    local bits = 0
    if ffe.isKeyHeld(ffe.KEY_W) then bits = bits + INPUT_UP end
    if ffe.isKeyHeld(ffe.KEY_S) then bits = bits + INPUT_DOWN end
    if ffe.isKeyHeld(ffe.KEY_A) then bits = bits + INPUT_LEFT end
    if ffe.isKeyHeld(ffe.KEY_D) then bits = bits + INPUT_RIGHT end

    -- Send to server (engine records for reconciliation)
    ffe.sendInput({bits = bits, aimX = 0, aimY = 0})

    -- Apply predicted movement locally for responsive feel
    -- (The server will reconcile if the result diverges)
    if bits ~= 0 and localEntity then
        local buf = {}
        if ffe.fillTransform(localEntity, buf) then
            local dx, dy = 0, 0
            if bits >= INPUT_RIGHT then dx = dx + SPEED * dt end
            if bits >= INPUT_LEFT  then dx = dx - SPEED * dt end
            if bits >= INPUT_UP    then dy = dy + SPEED * dt end
            if bits >= INPUT_DOWN  then dy = dy - SPEED * dt end
            ffe.setTransform(localEntity, buf.x + dx, buf.y + dy,
                             buf.rotation, buf.scaleX, buf.scaleY)
        end
    end

    -- Monitor prediction quality
    local err = ffe.getPredictionError()
    if err > 1.0 then
        -- Large error means server disagreed significantly with our prediction
        ffe.log("Prediction error: " .. err)
    end
end
```

### 6. Lua: Server-Side Input Handling

```lua
local SPEED = 250.0
local INPUT_UP    = 1
local INPUT_DOWN  = 2
local INPUT_LEFT  = 4
local INPUT_RIGHT = 8

local players = {}  -- clientId -> entityId

function ffe.onInit()
    ffe.startServer(7777)
    ffe.setNetworkTickRate(20)

    ffe.onClientConnected(function(clientId)
        -- Spawn a player entity for the new client
        local id = ffe.createEntity()
        ffe.addTransform(id, 0, 0, 0, 1, 1)
        ffe.addPreviousTransform(id)
        ffe.addSprite(id, 1, 40, 40, 1.0, 0.4, 0.3, 1.0, 1)
        players[clientId] = id
    end)

    ffe.onClientDisconnected(function(clientId)
        if players[clientId] then
            ffe.destroyEntity(players[clientId])
            players[clientId] = nil
        end
    end)

    -- Handle input commands from clients (authoritative movement)
    ffe.onServerInput(function(clientId, inputTable)
        local entity = players[clientId]
        if not entity then return end

        local bits = inputTable.bits or 0
        local dt   = inputTable.dt or (1.0 / 60.0)

        local dx, dy = 0, 0
        if bits >= INPUT_UP    and (bits % (INPUT_UP * 2))    >= INPUT_UP    then dy = dy + SPEED * dt end
        if bits >= INPUT_DOWN  and (bits % (INPUT_DOWN * 2))  >= INPUT_DOWN  then dy = dy - SPEED * dt end
        if bits >= INPUT_LEFT  and (bits % (INPUT_LEFT * 2))  >= INPUT_LEFT  then dx = dx - SPEED * dt end
        if bits >= INPUT_RIGHT and (bits % (INPUT_RIGHT * 2)) >= INPUT_RIGHT then dx = dx + SPEED * dt end

        if dx ~= 0 or dy ~= 0 then
            local buf = {}
            if ffe.fillTransform(entity, buf) then
                ffe.setTransform(entity, buf.x + dx, buf.y + dy,
                                 buf.rotation, buf.scaleX, buf.scaleY)
            end
        end
    end)
end
```

### 7. C++: Client-Side Prediction with MoveFn (Full Reconciliation)

```cpp
#include "networking/prediction.h"
#include "networking/network_system.h"
using namespace ffe::networking;

// Movement function — applied both during prediction and reconciliation replay
void movePlayer(ffe::World& world, uint32_t entityId,
                const InputCommand& cmd, void*) {
    auto* t = world.getComponent<ffe::Transform>(entityId);
    if (!t) return;

    constexpr float SPEED = 250.0f;
    if (cmd.inputBits & 0x01) t->y += SPEED * cmd.dt;  // up
    if (cmd.inputBits & 0x02) t->y -= SPEED * cmd.dt;  // down
    if (cmd.inputBits & 0x04) t->x -= SPEED * cmd.dt;  // left
    if (cmd.inputBits & 0x08) t->x += SPEED * cmd.dt;  // right
}

// At startup (client side):
ffe::networking::setMovementFunction(movePlayer, nullptr);
ffe::networking::setLocalPlayer(playerEntityId);

// Each frame (client side):
InputCommand cmd;
cmd.inputBits = gatherInputBits();

ffe::networking::sendInput(cmd);
```

```cpp
#include "networking/packet.h"
using namespace ffe::networking;

// Send
uint8_t buf[MAX_PACKET_SIZE];
PacketWriter writer(buf, MAX_PACKET_SIZE);

PacketHeader hdr{};
hdr.type     = PacketType::INPUT;
hdr.channel  = 0;
hdr.sequence = nextSeq++;
writeHeader(writer, hdr);

// Receive
void onReceive(const ReceivedPacket& pkt, void*) {
    PacketReader reader(pkt.data, pkt.dataLength);
    PacketHeader hdr;
    if (!readHeader(reader, hdr)) return;
    if (!isValidPacketType(static_cast<uint8_t>(hdr.type))) return;

    float x, y;
    uint8_t buttons;
    if (reader.readF32(x) && reader.readF32(y) && reader.readU8(buttons)) {
        handleInput(pkt.sender, x, y, buttons);
    }
}
```

## What Not to Do

### Do not skip `initNetworkSystem()` / `shutdownNetworkSystem()`

ENet requires global initialisation. The module-level `initNetworkSystem()` handles this plus default component registration. Calling `startServer` or `connectToServer` without it will fail or cause undefined behaviour. (Lua games get this automatically from the engine loop.)

### Do not hold `ReceivedPacket::data` beyond the callback

The `data` pointer in `ReceivedPacket` is a non-owning view into ENet's internal buffer. It is only valid for the duration of the receive callback. Copy it if you need the data later.

### Do not send packets larger than `MAX_PACKET_SIZE` (1200 bytes)

Packets are capped at 1200 bytes (MTU-safe). `PacketWriter::hasError()` returns true if you overflow. Split large data across multiple packets.

### Do not cast raw bytes to `PacketType` without validation

Always call `isValidPacketType(rawByte)` before casting a `uint8_t` to `PacketType`. Unvalidated casts produce undefined enum values.

### Do not ignore `PacketReader` / `PacketWriter` error flags

Every read/write returns `false` on failure. Once the error flag is set, all subsequent operations also fail. Always check `hasError()` or individual return values. Ignoring errors means processing garbage data.

### Do not use `readF32` output without checking the return value

`readF32` rejects NaN and Infinity for security. If it returns `false`, the float was non-finite or the buffer underflowed.

### Do not set callbacks after calling `start()` / `connect()`

Register callbacks before starting the transport. Setting them after may cause events to be missed during the handshake.

### Do not run both server and client simultaneously

The module-level API (`network_system.h`) enforces server XOR client. Calling `startServer` while a client is connected (or vice versa) will disconnect the current mode first. Design your game to be one or the other per process.

### Do not call `sendInput` on the server

`sendInput` is a client-only function. It records input in the prediction buffer and sends it to the server. On the server, use `onServerInput` to receive input from clients and apply authoritative movement directly. Calling `sendInput` on the server returns false and does nothing.

### Do not call `setLocalPlayer` before creating the entity

`setLocalPlayer` stores the entity ID for prediction. If the entity does not exist yet, reconciliation will fail silently. Always create the entity and add its Transform component first, then call `setLocalPlayer`.

### Do not manually reconcile predicted positions

The prediction system handles reconciliation automatically when server snapshots arrive. If the server position diverges from the predicted position by more than 0.1 units, the system snaps to the server position and replays buffered inputs. Do not write your own snap/lerp logic on top of this -- it will fight the built-in reconciliation.

### Do not call `performHitCheck` every frame

`performHitCheck` is for discrete fire events, not continuous scanning. It performs a ray-vs-sphere test against all tracked entities at a rewound tick. Calling it every frame wastes CPU. Call it only when the player fires.

### Do not forget to call `recordFrame` every network tick (server-side)

The `LagCompensator` needs entity positions recorded each tick to have history to rewind to. If you skip `recordFrame` calls, hit checks will find no data at the requested tick and return misses. The module-level `updateNetworkSystem` handles this automatically, but if you use `LagCompensator` directly, you must call `recordFrame` before applying new inputs each tick.

### Do not create a lobby before starting the server

`createLobby` is server-only. It requires an active server (`startServer` must succeed first). Calling it in client mode does nothing and returns false.

### Do not poll `getLobbyPlayers` on the client without `onLobbyUpdate`

The client's lobby state is only updated when the server broadcasts it. Polling `getLobbyPlayers` between broadcasts returns stale data. Use `onLobbyUpdate` to react to state changes.

### Do not call `startLobbyGame` without checking readiness

`startLobbyGame` sets the `gameStarted` flag unconditionally. It does not check whether all players are ready. Check readiness yourself (e.g., iterate `getLobbyPlayers` and verify all are ready) before calling it.

### Do not use `std::function` or virtual dispatch for packet handling

Per CLAUDE.md Section 3, hot-path code avoids `std::function` (hidden heap allocations) and virtual calls. The API uses raw function pointers with `void* userData`. Follow the same pattern in game-level handlers.

### Do not process packets without rate-limit awareness (server side)

`ServerTransport` enforces rate limiting automatically (100 packets/sec, 64 KB/sec per connection). If you build custom server logic outside of `ServerTransport`, enforce rate limiting yourself.

## Tier Support

| Tier | Supported | Notes |
|------|-----------|-------|
| **RETRO** | No | ENet requires socket APIs that may not be available on minimal RETRO targets |
| **LEGACY** | **Yes (primary)** | No GPU involvement. Pure CPU + sockets. |
| **STANDARD** | Yes | Same implementation as LEGACY |
| **MODERN** | Yes | Same implementation as LEGACY |

The networking subsystem has no GPU involvement. Performance is bounded by socket I/O and packet processing, not rendering tier.
