# editor/ -- System Context


## System Purpose

The FastFreeEngine Editor is a standalone desktop application (separate binary from the game runtime) for visual scene editing, entity inspection, and asset management. It embeds the engine via the editor-hosted API (`initSubsystems`, `tickOnce`, `renderOnce`) and renders its UI with Dear ImGui over the engine's OpenGL 3.3 context. The editor owns the GLFW window and main loop; the engine runs as a guest inside it.

The editor follows the command pattern for all destructive operations, providing full undo/redo support. It is designed to be extended with new panels and new command types as the engine grows.

## Public API

### EditorApp (`editor/editor_app.h`, namespace `ffe::editor_app`)

The top-level editor object. Creates the window, initialises ImGui and the engine, runs the editor main loop.

| Method | Signature | Description |
|--------|-----------|-------------|
| constructor | `EditorApp()` | Default constructor |
| `init` | `bool init()` | Create GLFW window, load OpenGL, init ImGui, init engine subsystems. Returns false on failure |
| `run` | `void run()` | Enter the editor main loop (blocks until window close). Polls events, renders ImGui UI, swaps buffers |
| `shutdown` | `void shutdown()` | Shut down ImGui, engine subsystems, and GLFW. Call after `run()` returns |

Non-copyable, non-movable. There is exactly one EditorApp.

### Command (`editor/commands/command.h`, namespace `ffe::editor`)

Abstract base class for undoable operations. Virtual dispatch is acceptable here because commands execute on user interaction (mouse click, keyboard shortcut), not per-frame.

| Method | Signature | Description |
|--------|-----------|-------------|
| `execute` | `virtual void execute() = 0` | Perform the operation |
| `undo` | `virtual void undo() = 0` | Reverse the operation |
| `description` | `virtual const char* description() const = 0` | Human-readable label for undo/redo menus |

### CommandHistory (`editor/commands/command_history.h`, namespace `ffe::editor`)

| `undo` | `void undo()` | Undo the most recent command. No-op if empty |
| `redo` | `void redo()` | Redo the most recently undone command. No-op if empty |

#### Entity Commands (`editor/commands/entity_commands.h`, namespace `ffe::editor`)

|---------|------------|-------------|

#### ModifyComponentCommand (`editor/commands/component_commands.h`, namespace `ffe::editor`)

Generic template command for modifying any ECS component value through the undo/redo system. Stores old and new values by copy. Safe against entity deletion (checks `registry().valid()` before writing).


| Method | Description |
|--------|-------------|
| `execute()` | Writes `newValue` into the entity's component `T` |
| `undo()` | Writes `oldValue` back into the entity's component `T` |
| `description()` | Returns `"Modify Component"` |

#### AddComponentCommand (`editor/commands/component_commands.h`, namespace `ffe::editor`)

Template command for adding a component to an entity via the undo/redo system. Execute adds the component with default construction. Undo removes it. Safe against entity deletion and duplicate-add (checks `registry().valid()` and `all_of<T>` before writing).


| Method | Description |
|--------|-------------|
| `execute()` | Emplaces a default-constructed `T` on the entity (no-op if already present) |
| `undo()` | Removes the `T` component from the entity |
| `description()` | Returns `"Add Component"` |

#### RemoveComponentCommand (`editor/commands/component_commands.h`, namespace `ffe::editor`)

Template command for removing a component from an entity via the undo/redo system. Snapshots the component value at construction time so undo can restore it exactly. Safe against entity deletion.

```cpp
template<typename T>
RemoveComponentCommand(World& world, entt::entity entity);
```

| Method | Description |
|--------|-------------|
| `execute()` | Re-snapshots the component value, then removes `T` from the entity |
| `undo()` | Restores the snapshotted `T` value onto the entity (no-op if component already present or entity invalid) |
| `description()` | Returns `"Remove Component"` |

#### ReparentCommand (`editor/commands/reparent_command.h`, namespace `ffe::editor`)

Moves an entity to a new parent (or to root) in the scene graph hierarchy. Stores old and new parent so the operation is fully reversible. Uses `ffe::scene::setParent` / `ffe::scene::removeParent` for the actual hierarchy mutation.

```cpp
ReparentCommand(World& world, entt::entity entity, entt::entity newParent);
```

- `newParent`: the new parent entity, or `entt::null` to make the entity a root.

| Method | Description |
|--------|-------------|
| `execute()` | Sets the entity's parent to `newParent` (or removes parent if `entt::null`) |
| `undo()` | Restores the entity's original parent (captured at construction time) |
| `description()` | Returns `"Reparent Entity"` |

### PlayMode (`editor/play_mode.h`, namespace `ffe::editor`)

Manages the play-in-editor lifecycle. Snapshots the ECS World as an in-memory JSON string before entering play mode, and restores it on stop. No file I/O occurs.

**PlayState enum:** `EDITING`, `PLAYING`, `PAUSED`

**State transitions:**
- `EDITING` --play()--> `PLAYING` (snapshot captured)
- `PLAYING` --pause()--> `PAUSED`
- `PAUSED` --resume()--> `PLAYING`
- `PLAYING` or `PAUSED` --stop()--> `EDITING` (snapshot restored)

| Method | Signature | Description |
|--------|-----------|-------------|
| `state` | `PlayState state() const` | Current play state |
| `play` | `void play(World& world)` | Snapshot the scene and enter PLAYING. No-op if already PLAYING or PAUSED |
| `pause` | `void pause()` | Pause gameplay (keep modified state). Only valid when PLAYING |
| `resume` | `void resume()` | Resume from PAUSED back to PLAYING. Only valid when PAUSED |
| `stop` | `void stop(World& world)` | Restore the snapshot taken at `play()`, return to EDITING. Clears all entities then deserialises the snapshot. Only valid when PLAYING or PAUSED |

## Common Usage Patterns

### 1. Running the editor

```cpp
#include "editor_app.h"

int main() {
    ffe::editor_app::EditorApp editor;
    if (!editor.init()) {
        return 1;
    }
    editor.run();
    editor.shutdown();
    return 0;
}
```

### 2. Executing an undoable command

```cpp
#include "commands/command_history.h"
#include "commands/entity_commands.h"

ffe::editor::CommandHistory history;

// Create an entity (undoable)
history.executeCommand(
    std::make_unique<ffe::editor::CreateEntityCommand>(world, "Player")
);

// Undo it
if (history.canUndo()) {
    history.undo();  // Entity is destroyed
}

// Redo it
if (history.canRedo()) {
    history.redo();  // Entity is re-created (new ID)
}
```

### 3. Writing a custom command

```cpp
#include "commands/command.h"

class RenameEntityCommand : public ffe::editor::Command {
public:
    RenameEntityCommand(World& world, EntityId entity, const char* newName)
        : m_world(world), m_entity(entity) {
        std::strncpy(m_newName, newName, 63);
        m_newName[63] = '\0';
        // Snapshot the old name for undo
        const auto& name = m_world.getComponent<Name>(m_entity);
        std::memcpy(m_oldName, name.name, 64);
    }

    void execute() override {
        auto& name = m_world.getComponent<Name>(m_entity);
        std::memcpy(name.name, m_newName, 64);
    }

    void undo() override {
        auto& name = m_world.getComponent<Name>(m_entity);
        std::memcpy(name.name, m_oldName, 64);
    }

    const char* description() const override { return "Rename Entity"; }

private:
    World& m_world;
    EntityId m_entity;
    char m_newName[64] = {};
    char m_oldName[64] = {};
};
```

### 14. Drag-and-drop from AssetBrowser to InspectorPanel

The AssetBrowser sets an ImGui drag-and-drop payload when a file is clicked. The InspectorPanel's Mesh and Material3D sections act as drop targets for compatible asset types.

```cpp
// AssetBrowser (drag source) — sets payload with the asset file path.
// InspectorPanel (drop target) — accepts the payload and filters by extension:
//   - Image files (.png, .jpg, .bmp, .tga) → diffuse/normal/specular texture slots
//   - Mesh files (.glb, .gltf, .obj) → mesh asset slot
// The drawAssetDropTarget() helper handles payload acceptance and extension validation.
```

Drop targets are display-only for now — they store the path string but do not trigger asset loading. Full resource system integration is planned for a future session.

### 15. Integrating with the engine's editor-hosted mode

The editor creates the GLFW window and OpenGL context itself, then hands the window to the engine:

```cpp
// EditorApp::init() does this internally:
auto app = std::make_unique<ffe::Application>(config);
app->setWindow(m_window);      // Hand over the externally-created window
app->initSubsystems();         // Init ECS, renderer, audio, etc. (no window creation)

// In the main loop (EditorApp::run()):
app->tickOnce(dt);             // Run one fixed-timestep update
app->renderOnce(alpha);        // Render one frame into the shared GL context
```

## What Not To Do

### Do not create a second GLFW window for the engine

The editor owns the window. The engine receives it via `setWindow()`. Creating two windows causes OpenGL context conflicts.

### Do not modify entities without commands in the editor

All destructive operations (create, delete, modify) must go through `CommandHistory::executeCommand()` so they are undoable. Direct `world.createEntity()` calls in editor code bypass undo/redo.

```cpp
// BAD -- not undoable
world.createEntity();

// GOOD -- undoable
history.executeCommand(
    std::make_unique<CreateEntityCommand>(world, "New Entity")
);
```

### Do not call Application::run() from the editor

`run()` enters the engine's own main loop and blocks. The editor owns the main loop and calls `tickOnce()` / `renderOnce()` instead.

### Do not forget to call shutdown() after run()

`shutdown()` tears down ImGui, engine subsystems, and GLFW in the correct order. Skipping it leaks GPU resources and the window.

### Do not forget to restore glViewport after unbinding the FBO

`EditorFramebuffer::unbind()` restores framebuffer 0 but does NOT restore the previous viewport. After unbinding, the caller must call `glViewport()` with the window dimensions, or ImGui rendering will be clipped to the FBO size.

### Do not render into the FBO without checking isValid()

If `init()` failed (e.g., no GL context yet), the FBO handle is 0 and `bind()` would bind the default framebuffer. Always check `isValid()` before using the FBO in error-recovery paths.

### Do not modify the scene while in play mode

During PLAYING or PAUSED state, the scene belongs to gameplay. Inspector edits and entity creation/deletion should be disabled or guarded. Modifications made during play mode are discarded when the user hits Stop (the snapshot overwrites everything). If you allow edits during play, users will lose work with no warning.

### Do not call play() without a valid serialiser

`PlayMode::play()` snapshots the scene via the scene serialiser. If the serialiser is not initialised or the world has components the serialiser cannot handle, the snapshot will be incomplete and `stop()` will restore a broken scene.

### Do not allow AssetBrowser to navigate above the project root

`AssetBrowser` enforces this internally via `isWithinProjectRoot()`, but if you build a custom asset panel, always validate paths against the project root boundary. Same principle as FileDialog.

### Do not allow FileDialog to navigate above the project root

`FileDialog` enforces this internally, but if you build a custom file browser, always validate paths with a project-root boundary check. Scene files loaded from arbitrary paths are a path-traversal risk.

### Do not exceed MAX_UNDO_DEPTH without awareness

CommandHistory silently drops the oldest command when it exceeds 256 entries. This is by design, but commands that hold expensive resources should be aware they may be dropped without explicit cleanup (the destructor runs when dropped).

### Do not snapshot unknown component types in DestroyEntityCommand

Currently only Name, Transform, Transform3D, Sprite, Material3D, and Mesh are snapshotted. If you add a new component type, update `DestroyEntityCommand::snapshotComponents()` and `restoreComponents()` to preserve it across undo/redo.

### Do not reparent entities without using ReparentCommand in the editor

Direct calls to `ffe::scene::setParent()` from editor code bypass undo/redo. Always use `ReparentCommand` through `CommandHistory` so the operation is reversible.

```cpp
// BAD — not undoable
ffe::scene::setParent(world, child, newParent);

// GOOD — undoable
history.executeCommand(
    std::make_unique<ReparentCommand>(world, child, newParent)
);
```

### Do not reparent an entity to its own descendant

`ffe::scene::setParent()` rejects circular references and returns false. `ReparentCommand` uses `setParent()` internally, so the cycle check is automatic. However, be aware that if `setParent()` fails silently inside `execute()`, the undo stack still has the command. This is harmless (undo/redo of a no-op) but worth knowing.

### Do not assume drag-and-drop asset paths trigger resource loading

The InspectorPanel's drop targets currently store the asset path string for display only. They do not load textures or meshes. Full asset loading integration is planned for a future session. Do not write code that depends on the dropped path being loaded into GPU resources.

### Do not apply gizmo deltas without checking `update()` return value

`GizmoSystem::update()` returns true when the gizmo consumed mouse input. Only read `delta()` and apply it to the entity when `update()` returns true. Otherwise, forward mouse input to the scene camera as normal.

### Do not call `triggered()` before `update()` on ShortcutManager

`ShortcutManager::update()` polls ImGui IO state and marks which bindings fired. Calling `triggered()` before `update()` reads stale state from the previous frame.

### Do not use AddComponentCommand on types that require constructor arguments

`AddComponentCommand` default-constructs the component via `registry().emplace<T>()`. For component types that need non-default initialisation, use a custom command that emplaces with arguments.

### Do not export without setting the entry scene

`BuildConfig::entryScene` defaults to an empty string. If you call `exportGame()` without setting it, the generated `main.lua` will call `ffe.loadSceneJSON("")`, which will fail at runtime. Always set `entryScene` to a valid scene JSON path before exporting.

### Do not assume the runtime binary path is correct

`BuildPanel` defaults `m_runtimePath` to `"ffe_runtime"`, which assumes the binary is on `PATH` or in the current directory. If the export fails with a copy error, the runtime binary path is likely wrong. The exporter copies the file verbatim -- it does not search `PATH`.

## Tier Support

The editor targets the **LEGACY** tier (OpenGL 3.3, 1 GB VRAM). It uses the same OpenGL context as the engine. ImGui rendering is lightweight and does not affect the engine's performance budget.

The editor is a development tool, not a shipped game component. It is not expected to run on end-user machines at 60 fps, though it should remain responsive on LEGACY hardware.
