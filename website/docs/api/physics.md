# engine/physics -- Collision Detection & 3D Physics Subsystem

AI-native context file. Written by api-designer.

## System Purpose

The physics subsystem provides two capabilities:

1. **2D Collision Detection** — overlap detection for entities with `Transform` + `Collider2D`. No rigid-body dynamics. Answers "which pairs overlap?" each frame. Game code reads events and responds.

2. **3D Rigid-Body Physics** — full dynamics simulation powered by Jolt Physics. Supports body creation/destruction, force/impulse application, velocity control, and gravity configuration. Bodies are linked to ECS entities via the `RigidBody3D` component, and their transforms are synced to `Transform3D` each frame automatically.

### 2D Collision System

Header: `engine/physics/collision_system.h`
Namespace: `ffe`

| Symbol | Type | Description |
|--------|------|-------------|
| `collisionSystem` | `void collisionSystem(World& world, float dt)` | ECS system function. Detects overlaps between all entities with `Transform` + `Collider2D`. Writes `CollisionEventList` to the ECS registry context. |
| `COLLISION_SYSTEM_PRIORITY` | `inline constexpr i32` (200) | Priority at which `collisionSystem` should be registered. Must run after gameplay/scripting systems (100-199). |

**Internal pipeline (per frame):**

1. Collect all entities with `Transform` + `Collider2D`.
2. Build a spatial hash grid (arena-allocated, 1024 buckets, 128-unit cell size) for broad-phase culling.
3. Generate candidate pairs from overlapping grid cells.
4. Sort and deduplicate pairs (canonical ordering: `entityA < entityB`).
5. Filter by layer/mask: `(A.layer & B.mask) != 0 && (B.layer & A.mask) != 0`.
6. Run narrow-phase overlap tests (AABB-AABB, Circle-Circle, AABB-Circle).
7. Write `CollisionEventList` to the ECS registry context.

### 2D Collision Usage Patterns

#### 1. Add a collider to an entity (C++)

```cpp
#include "physics/collider2d.h"
#include "renderer/render_system.h" // Transform

// Create an entity with a Transform and an AABB collider.
auto entity = world.createEntity();
world.addComponent<Transform>(entity, Transform{
    .position = {100.0f, 200.0f, 0.0f},
    .scale    = {1.0f, 1.0f, 1.0f},
    .rotation = 0.0f
});
world.addComponent<Collider2D>(entity, Collider2D{
    .shape      = ColliderShape::AABB,
    .isTrigger  = false,
    .layer      = 0x0001,  // layer 0
    .mask       = 0xFFFF,  // collides with everything
    .halfWidth  = 16.0f,
    .halfHeight = 16.0f
});

// Circle collider (radius 12):
world.addComponent<Collider2D>(entity, Collider2D{
    .shape      = ColliderShape::CIRCLE,
    .halfWidth  = 12.0f,   // radius
    .halfHeight = 0.0f     // ignored for circles
});
```

#### 2. Register the collision system

```cpp
#include "physics/collision_system.h"

// Register at the correct priority (200 = physics band).
world.addSystem("collisionSystem", ffe::collisionSystem, ffe::COLLISION_SYSTEM_PRIORITY);
```

#### 3. Layer/mask filtering

Layers are 16-bit masks. Two entities A and B generate a collision event only if both directions pass:

```
(A.layer & B.mask) != 0  AND  (B.layer & A.mask) != 0
```

**Example -- player collides with pickups but not with other players:**

```cpp
// Player: layer bit 0, collides with layer bit 1 (pickups)
Collider2D playerCol{};
playerCol.layer = 0x0001;  // bit 0
playerCol.mask  = 0x0002;  // only collide with bit 1

// Pickup: layer bit 1, collides with layer bit 0 (players)
Collider2D pickupCol{};
pickupCol.layer = 0x0002;  // bit 1
pickupCol.mask  = 0x0001;  // only collide with bit 0
pickupCol.isTrigger = true;
```

#### 4. Read collision events from C++

```cpp
// After world.tick() has run collisionSystem:
if (world.registry().ctx().contains<ffe::CollisionEventList>()) {
    const auto& events = world.registry().ctx().get<ffe::CollisionEventList>();
    for (u32 i = 0; i < events.count; ++i) {
        EntityId a = events.events[i].entityA;
        EntityId b = events.events[i].entityB;
        // Handle collision: destroy pickup, apply damage, play sound, etc.
    }
}
```

**Important:** The `events` pointer is arena-allocated and valid only until the arena is reset (before the next frame). Do not store the pointer across frames.

#### 5. Handle collision events from Lua

```lua
-- Register a collision callback (called once per overlapping pair per frame).
ffe.setCollisionCallback(function(entityA, entityB)
    ffe.log("Collision: " .. tostring(entityA) .. " <-> " .. tostring(entityB))
    -- Respond: destroy one entity, play a sound, increment score, etc.
end)
```

The callback is invoked by `ScriptEngine::deliverCollisionEvents()` after all systems (including `collisionSystem`) have run. See the scripting `.context.md` for full Lua API details.

## 3D Rigid-Body Physics

### Overview

3D physics is powered by Jolt Physics with a single-threaded job system. No per-frame heap allocations -- uses a pre-allocated 10 MB temporary allocator internally. Supports box, sphere, and capsule shapes with static, kinematic, and dynamic motion types.

The system has two layers:
- **Low-level C++ API** (`engine/physics/physics3d.h`) -- direct body management via opaque `BodyHandle3D` handles.
- **ECS integration** (`engine/physics/physics3d_system.h`) -- `RigidBody3D` component links entities to physics bodies. `physics3dSyncSystem` copies Jolt transforms into `Transform3D` each frame at priority 60.
- **Lua bindings** -- 8 functions on the `ffe` table for scripted physics gameplay.

#### `ffe.destroyPhysicsBody(entityId)`

Destroys the Jolt body and removes the `RigidBody3D` component. Safe to call on entities without a body (no-op).

#### `ffe.applyForce(entityId, fx, fy, fz)`

Apply a continuous force (in Newtons) at center of mass. Use every frame for gravity guns, wind, thrusters. NaN/Inf values are silently ignored with a warning log.

#### `ffe.applyImpulse(entityId, ix, iy, iz)`

Apply an instantaneous impulse (Newton-seconds) at center of mass. Use for one-shot events: jumps, explosions, knockback. NaN/Inf values are silently ignored with a warning log.

#### `ffe.setLinearVelocity(entityId, vx, vy, vz)`

Set the body's velocity directly. Useful for character controllers or teleporting. NaN/Inf values are silently ignored with a warning log.

Returns three numbers. Returns `0, 0, 0` if the entity has no physics body.

#### `ffe.setGravity(gx, gy, gz)`

Change world gravity. Default is `(0, -9.81, 0)`. Affects all dynamic bodies. If called with no arguments, defaults to `(0, -9.81, 0)`. NaN/Inf values are silently ignored with a warning log.

Returns the current gravity vector as three numbers.

#### `ffe.onCollision3D(callback)`

Registers a Lua function to receive 3D collision events. Only one callback at a time — calling again replaces the previous one. The callback is invoked once per collision pair per frame by `ffe.dispatchCollision3DEvents()`.

**Callback signature:** `function(entityA, entityB, px, py, pz, nx, ny, nz, eventType)`

| Parameter | Type | Description |
|-----------|------|-------------|
| `entityA` | integer | First entity in the collision pair. |
| `entityB` | integer | Second entity in the collision pair. |
| `px, py, pz` | numbers | Contact point in world space. |
| `nx, ny, nz` | numbers | Contact normal in world space (points from A toward B). |
| `eventType` | string | `"enter"` (first frame of contact), `"stay"` (ongoing), or `"exit"` (contact ended). |

**Note on the 9-parameter callback:** The contact point, normal, and event type are essential for 3D physics gameplay (e.g., spawning sparks at the contact point, playing different sounds for enter vs. stay). If you only need entity IDs, ignore the extra parameters — Lua silently discards unused arguments.

#### `ffe.removeCollision3DCallback()`

Unregisters the 3D collision callback. Events are still buffered internally but not delivered to Lua. No-op if no callback is registered.

#### `ffe.dispatchCollision3DEvents()`

**Engine-internal.** Iterates the collision event buffer and calls the registered Lua callback for each event. This is called automatically by the engine's game loop after `stepPhysics3D()`. Game scripts should **not** call this directly — it exists on the `ffe` table for testing purposes only.

#### `ffe.castRay(ox, oy, oz, dx, dy, dz, maxDist) -> entity, hx, hy, hz, nx, ny, nz, dist | nil`

Cast a ray from origin `(ox, oy, oz)` in direction `(dx, dy, dz)` up to `maxDist` world units. Returns 8 values on hit, or `nil` on miss.

| Return value | Type | Description |
|-------------|------|-------------|
| `entity` | integer | Entity ID that was hit. |
| `hx, hy, hz` | numbers | Hit point in world space. |
| `nx, ny, nz` | numbers | Surface normal at the hit point. |
| `dist` | number | Distance from origin to hit point. |

Returns `nil` if nothing was hit or if any input is NaN/Inf.

**Why 8 return values instead of a table:** This avoids a Lua table allocation per call. For per-frame raycasts (e.g., ground checks), zero-allocation matters. Use `castRayAll` when you need multiple hits — it returns a table because the hit count is variable.

#### `ffe.castRayAll(ox, oy, oz, dx, dy, dz, maxDist) -> table`

Cast a ray and return all hits as an array of tables, sorted by distance (nearest first). Returns an empty table on miss or NaN/Inf input. Each entry:

```lua
{ entity = id, x = .., y = .., z = .., nx = .., ny = .., nz = .., distance = .. }
```

Maximum 32 hits (capped at `MAX_RAY_HITS`).

## What NOT to Do

### 2D Collision

#### Do not heap-allocate in the collision system or its hot path

The collision system uses the per-frame `ArenaAllocator` exclusively. All spatial hash nodes, candidate pair buffers, and event arrays are arena-allocated. Do not add `new`, `delete`, `malloc`, `std::vector::push_back` (without `reserve`), or `std::function` to any code path inside `collisionSystem`. This is a hard rule from the engine constitution (Section 3).

#### Do not call narrow-phase functions directly from Lua scripts

The narrow-phase overlap functions (`overlapAABB`, `overlapCircle`, `overlapAABBCircle`) are C++ inline functions intended for internal use by `collisionSystem`. They are not exposed to Lua. Game scripts should attach `Collider2D` components to entities and register a collision callback via `ffe.setCollisionCallback()` -- the engine handles the rest.

#### Do not store CollisionEvent pointers across frames

`CollisionEventList.events` points to arena-allocated memory. The arena is reset before the next frame. Any pointer into the event array becomes dangling after the arena reset. Read the events within the same frame they were generated.

#### Do not modify Collider2D components during collisionSystem execution

The collision system iterates over all `Collider2D` components during its run. Adding, removing, or modifying colliders from within a system that runs concurrently or at the same priority as `collisionSystem` produces undefined iteration behaviour. Modify colliders in gameplay systems (priority < 200) or in post-collision handlers (after `collisionSystem` has finished).

#### Do not use layer 0x0000

Setting `layer = 0x0000` means the entity belongs to no layer. The bidirectional check `(A.layer & B.mask)` will always be 0, so the entity will never generate collision events regardless of other entities' masks. This is almost certainly a mistake. If you want an entity to not collide with anything, remove the `Collider2D` component instead.

#### Do not expect physics response from the 2D collision system

This subsystem is overlap detection only. There is no rigid-body solver. If your game needs push-back (e.g., preventing a player from walking through walls), implement it in your gameplay system by reading collision events and adjusting positions manually. For rigid-body dynamics, use the 3D physics system.

### 3D Physics

#### Do not create physics bodies every frame

`ffe.createPhysicsBody()` allocates a Jolt body internally. Creating and destroying bodies per-frame will tank performance. Create bodies once (in `_init()` or when spawning an entity) and reuse them.

```lua
-- WRONG: creating a body every frame
function _update(dt)
    ffe.createPhysicsBody(bullet, { shape = "sphere" })  -- memory leak + slow
end

-- RIGHT: create once, then apply forces each frame
function _init()
    ffe.createPhysicsBody(bullet, { shape = "sphere" })
end
function _update(dt)
    ffe.applyForce(bullet, 0, 0, -100)
end
```

#### Do not forget to destroy physics bodies

When you destroy an entity, the Jolt body is **not** automatically cleaned up (this is a foundation limitation -- automatic cleanup is planned). Always call `ffe.destroyPhysicsBody(entityId)` before `ffe.destroyEntity(entityId)`.

```lua
-- WRONG: orphaned Jolt body leaks until shutdown
ffe.destroyEntity(enemy)

-- RIGHT: clean up physics first
ffe.destroyPhysicsBody(enemy)
ffe.destroyEntity(enemy)
```

#### Do not apply forces to static bodies

Static bodies (`motion = "static"`) are immovable by design. Calling `ffe.applyForce()` or `ffe.applyImpulse()` on a static body has no visible effect. If you need a body to move, use `"dynamic"` or `"kinematic"`.

#### Do not use applyForce for one-shot events

`applyForce()` applies a continuous force that accumulates over the timestep. For instant events (jumps, explosions), use `applyImpulse()` instead. Using `applyForce()` for a jump will produce inconsistent results depending on frame rate.

#### Do not pass NaN or Infinity to physics functions

All Lua physics bindings guard against non-finite values and will silently ignore the call with a warning log. If your physics objects are behaving unexpectedly, check the engine log for `"non-finite input ignored"` warnings -- this usually means a division by zero or uninitialized variable in your script.

#### Do not call ffe.castRayAll() every frame in a loop over many entities

`castRayAll` allocates a Lua table per call (one table per hit, plus the outer array). Calling it in a tight loop (e.g., once per enemy per frame) creates heavy GC pressure. Use `ffe.castRay()` (zero-allocation, returns multiple values) for per-frame single-hit checks like ground detection. Reserve `castRayAll` for occasional queries (weapon fire, line-of-sight checks triggered by events).

```lua
-- WRONG: castRayAll for every enemy every frame
function _update(dt)
    for _, enemy in ipairs(enemies) do
        local hits = ffe.castRayAll(...)  -- table allocation per enemy per frame!
    end
end

-- RIGHT: use castRay for per-frame checks (zero allocation)
function _update(dt)
    local entity, hx, hy, hz = ffe.castRay(px, py, pz, 0, -1, 0, 2)
    if entity then ... end
end
```

#### Do not call ffe.onCollision3D() every frame

`onCollision3D` replaces the previous callback each time it is called (the old Lua function reference is released). Calling it per frame churns Lua registry references pointlessly. Register the callback once in `_init()` or at script load time.

```lua
-- WRONG: re-registering every frame
function _update(dt)
    ffe.onCollision3D(function(a, b) ... end)  -- new closure + registry churn every frame
end

-- RIGHT: register once
ffe.onCollision3D(function(a, b, px, py, pz, nx, ny, nz, eventType)
    if eventType == "enter" then handleCollision(a, b) end
end)
```

#### Do not call ffe.dispatchCollision3DEvents() from game scripts

This function is called automatically by the engine loop after `stepPhysics3D()`. Calling it from Lua would re-deliver the same events, causing duplicate callback invocations. It is exposed on the `ffe` table for C++/test use only.

#### Do not store collision event data across frames

Collision events are buffered per-frame and cleared at the start of the next `stepPhysics3D()`. Process events inside the callback — do not store entity IDs or contact points expecting them to remain valid across frames (entities may be destroyed, positions change).

#### Do not mix up force and impulse units

- **Force** is in Newtons. Applied continuously. Depends on mass: `F = m * a`. A 10 kg object needs 98.1 N upward force just to counteract gravity.
- **Impulse** is in Newton-seconds. Applied instantly. Changes velocity by `impulse / mass`. A 10 kg object hit with impulse `(0, 50, 0)` gains 5 m/s upward velocity.

## Tier Support

| Tier | 2D Collision | 3D Physics | Notes |
|------|-------------|------------|-------|
| RETRO | Yes | Yes | CPU-only. No GPU dependency. |
| LEGACY | Yes | Yes | Primary target. Single-threaded Jolt job system. Comfortable with hundreds of bodies. |
| STANDARD | Yes | Yes | Same implementation. |
| MODERN | Yes | Yes | Same implementation. Future: multi-threaded Jolt job system for thousands of bodies. |

Both subsystems are CPU-only with no GPU, OpenGL, or Vulkan dependency. The 2D collision spatial hash (1024 buckets, 128-unit cells) is tuned for typical 2D entity counts (tens to hundreds). The 3D physics pre-allocates a 10 MB temporary allocator for Jolt's per-frame scratch memory.
