## System Purpose

The audio subsystem provides fire-and-forget sound effect playback, looping music streaming, and **3D positional audio** for FFE games. It wraps **miniaudio** (a public-domain single-header C library) for device output and **stb_vorbis** (another single-header library) for OGG decoding. WAV files are decoded by miniaudio directly.

The design follows the same global-module-singleton pattern as `ffe::rhi`: one audio device, one mixer, one output stream. Sound effects are decoded fully into float32 PCM at load time and held in a fixed-capacity table of 256 slots. Up to 32 voices can play simultaneously. Music tracks stream from disk via stb_vorbis per-callback decode -- one active music track at a time.

The audio callback runs on a background thread managed by miniaudio. Game code (main thread) communicates with the callback via a lock-free SPSC command ring buffer -- the main thread never blocks waiting for the audio thread, and the audio thread never stalls the game loop.

3D positional audio uses a built-in inverse-distance attenuation model with configurable min/max distances. The listener automatically tracks the active 3D camera each frame (no manual sync needed for the common case). Manual listener override is available for advanced scenarios.

**Supported formats:** WAV (RIFF magic) and OGG Vorbis (OggS magic). Format is detected by magic bytes, not file extension.

**Supported tier:** LEGACY and above. RETRO is not supported.

### Handle Type

```cpp
struct SoundHandle {
    u32 id = 0;  // 0 is invalid; non-zero is valid
```

`SoundHandle` is an opaque value. `id == 0` means invalid (load failed or handle was never set). Use `isValid()` to check before passing to any playback function.

### 3D Positional Audio

3D audio adds spatial positioning to fire-and-forget sound effects. Sounds are placed at a world-space position and attenuated based on distance from the listener, with left/right panning based on direction. The attenuation model is **inverse-distance** with configurable min and max range.

**Listener auto-sync:** By default, the listener position and orientation are automatically synchronised to the active 3D camera each frame (via `updateListenerFromCamera()` called from `Application::tick()`). This means **you do not need to manually set the listener** for the common case of a 3D game with a camera. If you call `setListenerPosition()` manually, the automatic sync is disabled until the next scene load.

| Function | Signature | Description |
|----------|-----------|-------------|
| `playSound3D` | `void playSound3D(SoundHandle handle, float x, float y, float z, float volume = 1.0f)` | Play a one-shot 3D positioned sound effect at `(x, y, z)` in world space. Fire-and-forget -- the position is fixed at creation time (no per-frame tracking for fire-and-forget voices). `volume` is an additional per-instance multiplier on top of distance attenuation. NaN/Inf coordinates are clamped to 0.0. Same voice budget as `playSound()` (shared 32-voice pool). |
| `setListenerPosition` | `void setListenerPosition(float x, float y, float z, float fx, float fy, float fz, float ux, float uy, float uz)` | Manually set the listener position and orientation. 9 parameters: position `(x, y, z)`, forward direction `(fx, fy, fz)`, up direction `(ux, uy, uz)`. Forward and up should be normalised. Calling this disables automatic camera sync. NaN/Inf values are replaced with defaults (position: 0,0,0; forward: 0,0,-1; up: 0,1,0). |
| `setSound3DMinDistance` | `void setSound3DMinDistance(float dist)` | Set the minimum attenuation distance. Within this distance, the sound plays at full volume (gain = 1.0). Clamped to `[0.01, current maxDistance)`. Default: implementation-defined (typically 1.0). |
| `setSound3DMaxDistance` | `void setSound3DMaxDistance(float dist)` | Set the maximum attenuation distance. Beyond this distance, the sound is silent (gain = 0.0). Clamped to `(current minDistance, 10000.0]`. Default: implementation-defined (typically 100.0). |
| `getSound3DMinDistance` | `float getSound3DMinDistance()` | Query the current 3D audio minimum attenuation distance. |
| `getSound3DMaxDistance` | `float getSound3DMaxDistance()` | Query the current 3D audio maximum attenuation distance. |
| `updateListenerFromCamera` | `void updateListenerFromCamera(float posX, float posY, float posZ, float fwdX, float fwdY, float fwdZ, float upX, float upY, float upZ)` | **Internal -- called by `Application::tick()`.** Syncs the listener to the 3D camera position and orientation each frame. Only takes effect if `setListenerPosition()` has not been called (automatic mode). Game code should not call this directly. |

#### Attenuation Model

The built-in spatialization uses inverse-distance attenuation between `minDistance` and `maxDistance`:

- **Distance < minDistance**: gain = 1.0 (full volume)
- **minDistance <= distance <= maxDistance**: gain falls off linearly from 1.0 to 0.0
- **Distance > maxDistance**: gain = 0.0 (silent)

Left/right stereo panning is computed from the angle between the listener's forward direction and the vector to the sound source, projected onto the listener's horizontal plane.

## Common Usage Patterns

### 1. Initialise, load a sound, play it, and clean up at shutdown

```cpp
#include "audio/audio.h"

// In Application::startup() or equivalent:
const bool audioOk = ffe::audio::init(false);
if (!audioOk) {
    // Audio is unavailable (no device, or device failed to open).
    // The engine continues without audio -- all playback calls become no-ops.
    FFE_LOG_WARN("Game", "Audio unavailable -- continuing without sound");
}

// Load sound effects at scene load time, not per-frame.
// path is relative; assetRoot is an absolute directory.
const char* assetRoot = "/home/user/mygame/assets/audio";
ffe::audio::SoundHandle jumpSfx = ffe::audio::loadSound("sfx/jump.wav", assetRoot);
ffe::audio::SoundHandle coinSfx = ffe::audio::loadSound("sfx/coin.ogg", assetRoot);

// Check that the sounds loaded before using them.
if (!ffe::audio::isValid(jumpSfx)) {
    FFE_LOG_WARN("Game", "jump.wav failed to load -- jump sound will be silent");
}

// Play sounds in response to game events -- not per-frame:
void onPlayerJump() {
    ffe::audio::playSound(jumpSfx);           // full volume (default 1.0)
}

void onCoinCollected() {
    ffe::audio::playSound(coinSfx, 0.7f);     // 70% volume
}

// At application shutdown -- after all audio calls:
ffe::audio::unloadSound(jumpSfx);
ffe::audio::unloadSound(coinSfx);
ffe::audio::shutdown();
```

### 2. Play a 3D positioned sound effect (C++)

```cpp
#include "audio/audio.h"

// Pre-load at scene start (not per-frame).
const char* assetRoot = "/home/user/mygame/assets";
ffe::audio::SoundHandle explosionSfx = ffe::audio::loadSound("sfx/explosion.wav", assetRoot);

// Configure attenuation range for your scene scale.
// Sounds within 2 units play at full volume; beyond 50 units they are silent.
ffe::audio::setSound3DMinDistance(2.0f);
ffe::audio::setSound3DMaxDistance(50.0f);

// When an explosion happens at world position (10, 0, -5):
ffe::audio::playSound3D(explosionSfx, 10.0f, 0.0f, -5.0f);

// With a custom volume multiplier (on top of distance attenuation):
ffe::audio::playSound3D(explosionSfx, 10.0f, 0.0f, -5.0f, 0.6f);
```

The listener is automatically synced to the 3D camera -- no manual setup required.

```cpp
// Set music volume independent of sound effects volume.

// On scene transition -- stop music before loading the next scene.

// At shutdown -- unload after stopping.
ffe::audio::stopMusic();
ffe::audio::unloadSound(bgMusic);
ffe::audio::shutdown();
```

### 9. Switch music track on scene transition

```cpp

ffe::audio::SoundHandle dungeonMusic   = ffe::audio::loadSound("music/dungeon.ogg",   assetRoot);

// When leaving the dungeon:
```

## What NOT to Do

### Do not call loadSound() per frame

`loadSound()` performs `stat()`, `realpath()`, `fopen()`, full PCM decode, and a heap allocation for the decoded PCM buffer. Calling it per frame is a severe performance violation that will cause drops on any hardware tier.

```cpp
// BAD -- per-frame loadSound is a critical performance violation
void gameSystem(ffe::World& world, float dt) {
    auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);  // WRONG: every frame
    ffe::audio::playSound(h);
    // Also leaks: the handle is never unloaded
}

// GOOD -- load once at scene start, keep the handle, play many times
// At scene load:
g_jumpSfx = ffe::audio::loadSound("sfx/jump.wav", assetRoot);

// In response to the game event (not per-frame):
ffe::audio::playSound(g_jumpSfx);
```

### Do not call ffe.playSound3D() in a per-frame loop from Lua

The Lua `ffe.playSound3D(path, ...)` binding auto-loads the sound by path on every call. This means file I/O on every invocation. Use it for event-triggered sounds (explosions, footsteps on contact), not for sounds that fire every frame.

```lua
-- BAD -- auto-loads the file every single frame
function update(entityId, dt)
    ffe.playSound3D("sfx/engine_hum.wav", px, py, pz)  -- WRONG: 60 calls/sec = 60 file loads
end

-- GOOD -- trigger on discrete events, not per-frame
function update(entityId, dt)
    if justLanded then
        ffe.playSound3D("sfx/footstep.wav", px, py, pz, 0.5)  -- once per landing event
    end
end
```

### Do not set minDistance >= maxDistance

The attenuation range requires `minDistance < maxDistance`. Setting them equal or inverted produces undefined gain behaviour. The C++ API clamps to prevent this, but relying on clamping indicates a logic error.

```lua
-- BAD -- min equals max, clamped but meaningless
ffe.setSound3DMinDistance(10.0)
ffe.setSound3DMaxDistance(10.0)  -- clamped to just above 10.0 -- not useful

-- GOOD -- clear separation between full-volume and silent distances
ffe.setSound3DMinDistance(2.0)
ffe.setSound3DMaxDistance(50.0)
```

### Do not use a SoundHandle after unloadSound()

`unloadSound()` frees the decoded PCM buffer. Any voice using that buffer is stopped under the mutex before the free -- there is no use-after-free. But after the call, the handle's slot is recycled and will be reused by a future `loadSound()`. Always clear the handle variable after unloading.

```cpp
// BAD -- use after unload
ffe::audio::unloadSound(jumpSfx);
ffe::audio::playSound(jumpSfx);   // undefined behaviour -- handle slot was recycled

// GOOD -- null the handle after unloading
ffe::audio::unloadSound(jumpSfx);
jumpSfx = ffe::audio::SoundHandle{};   // id = 0; isValid() now returns false
```

### Do not call audio functions before init()

All audio functions check `s_state.initialised` internally and are silent no-ops if `init()` has not been called. However, silently doing nothing when you expect sound is a logic error. Always call `init()` first.

```cpp
// BAD -- no init before use
auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);  // returns SoundHandle{0}
ffe::audio::playSound(h);   // no-op (not initialised) -- silent failure

// GOOD
ffe::audio::init();
auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);
ffe::audio::playSound(h);
```

### Do not call audio functions from the audio callback thread

`playSound()` posts to an SPSC ring buffer consumed by the audio callback. Calling `playSound()` from within the callback would write from the consumer side, violating the SPSC contract and producing a data race. Game code should never run inside the miniaudio callback.

### Do not pass absolute paths or traversal sequences to loadSound()

`isAudioPathSafe()` is the first operation in `loadSound()`, before any syscall. Paths starting with `/` or `\`, containing `../`, or containing drive letters (`C:`) are rejected immediately.

```cpp
// BAD -- rejected before any filesystem access
auto h = ffe::audio::loadSound("/etc/passwd", assetRoot);         // SoundHandle{0}
auto h = ffe::audio::loadSound("../../../etc/passwd", assetRoot); // SoundHandle{0}

// GOOD -- relative path within the asset root
auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);
```

### Do not call playMusic() with a handle that was already unloaded

`playMusic()` retrieves the canonical file path from the sound table. If the handle was unloaded via `unloadSound()`, the table slot is cleared. The PLAY_MUSIC command will be silently dropped in the callback with an error log.

```cpp
// BAD -- use after unload
ffe::audio::unloadSound(bgMusic);
ffe::audio::playMusic(bgMusic);  // error logged, no music plays

// GOOD -- only unload music handles when you are truly done with them
ffe::audio::stopMusic();
ffe::audio::unloadSound(bgMusic);
bgMusic = ffe::audio::SoundHandle{};
```

### Do not use playMusic() for short one-shot sound effects

`playMusic()` opens a stb_vorbis streaming state -- there is ~10 us overhead per `playMusic()` call as the decoder opens the file and reads the OGG headers. For short sound effects that trigger many times per second, use `playSound()` instead.

### Do not ignore the return value of loadSound()

`loadSound()` can fail for many reasons: file not found, unsupported format, decoded size exceeds the 10 MB limit, sound table full (max 256 sounds). Always check `isValid()`. The specific error reason is in the log.

```cpp
// BAD -- silently uses an invalid handle
auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);
ffe::audio::playSound(h);   // silent no-op if load failed

// GOOD -- check and handle failure explicitly
auto h = ffe::audio::loadSound("sfx/jump.wav", assetRoot);
if (!ffe::audio::isValid(h)) {
    FFE_LOG_WARN("Game", "jump.wav failed to load -- jump will be silent");
}
ffe::audio::playSound(h);   // still safe (no-op on invalid handle), but now the bug is visible
```

### Do not manually call setListenerPosition() unless you need to decouple from the camera

The listener auto-syncs to the 3D camera every frame. Calling `setListenerPosition()` disables this. Only use the manual override for advanced cases like split-screen, VR, or a listener attached to a player entity rather than the camera.

```lua
-- BAD (unnecessary) -- the camera already syncs the listener
function update(entityId, dt)
```

## Security

The audio subsystem enforces the following constraints on file loading. All of these run before the decoder is invoked:

1. **Path safety check first (SEC-1):** `isAudioPathSafe()` is the first operation in `loadSound()`. Rejects null, empty, absolute paths (Unix and Windows), drive letters, UNC paths, and all traversal sequences (`../`, `..\`, `/..`, `\..`). Uses `strnlen` bounded at `MAX_AUDIO_PATH` -- not `strlen`.
2. **realpath() canonicalisation (SEC-1):** After string validation, `realpath()` resolves symlinks and any encoding tricks the string check missed. The canonical path is then verified to start with the `assetRoot` prefix. If it does not, the load is rejected.
3. **File size limit before decoding (SEC-2):** `stat()` is called before `fopen()`. Files larger than `AUDIO_MAX_FILE_BYTES` (32 MB) are rejected before the decoder touches them.
4. **Format allowlist by magic bytes (SEC-3):** Only WAV (RIFF magic: `52 49 46 46`) and OGG (OggS magic: `4F 67 67 53`) are accepted. The first 4 bytes of the file are read and checked before the decoder is invoked.
5. **Decoder output validation before cast (HIGH-2 / SEC-5):** `frameCount`, `channels`, and `sampleRate` from the decoder are validated as positive and sane before any cast to `u64`. This prevents a negative `int` from the decoder being silently cast to `0xFFFFFFFFFFFFFFFF` in the subsequent size multiplication.
6. **Decoded size limit in u64 arithmetic (SEC-4):** Decoded PCM size is computed in `u64` arithmetic after validation and compared to `AUDIO_MAX_DECODED_BYTES` (10 MB).
7. **Volume clamping (SEC-6):** All volume inputs pass through `clampVolume()`. NaN and Inf are replaced with 0.0.
8. **Atomic voice/buffer teardown (LOW-4):** `unloadSound()` deactivates any active voice AND frees the PCM buffer in the same mutex critical section -- no window exists between deactivation and free.
9. **3D coordinate sanitization:** `playSound3D()` and `setListenerPosition()` clamp NaN/Inf inputs to safe defaults (0.0 for position, (0,0,-1) for forward, (0,1,0) for up). The Lua bindings additionally reject NaN/Inf coordinates with a warning log before calling the C++ API.

## Tier Support

| Tier | Supported | Notes |
|------|-----------|-------|
| RETRO | **No** | Not supported. RETRO (~2005) hardware may lack a reliable audio API compatible with miniaudio. Games targeting RETRO have no audio. This is an explicit design decision, not an oversight. |
| LEGACY | **Yes** (primary target) | 44.1 kHz float32 stereo output; WAV and OGG formats; up to 32 simultaneous SFX voices (shared between 2D and 3D); music streaming via stb_vorbis (one track at a time); 3D spatialization with inverse-distance attenuation; no per-frame allocations; lock-free SPSC command ring. miniaudio's ALSA/PulseAudio/CoreAudio/WASAPI backends are stable on 2012-era hardware. |
| STANDARD | **Yes** | Identical behaviour to LEGACY. |
| MODERN | **Yes** | Identical behaviour to LEGACY. |

The audio callback runs on its own thread managed by miniaudio. The game loop's performance impact is: one lock-free ring-buffer write per `playSound()` / `playSound3D()` / `playMusic()` call (on game-event boundaries, not per-frame), plus a single mutex acquisition per callback that covers both SFX mixing (including 3D gain computation) and music decode. The `computeSpatialGain()` function runs per active 3D voice per callback -- this is a few multiplies and a square root, well within LEGACY-tier budget for 32 voices.

## Dependencies

| Dependency | Source | Role |
|------------|--------|------|
| **miniaudio v0.11.25** | `third_party/miniaudio.h` (embedded, public domain) | Platform audio device abstraction; WAV decoding; audio callback thread management |
| **stb_vorbis v1.22** | `third_party/stb_vorbis.c` (embedded, public domain) | OGG Vorbis decoding |
| `ffe::core` types | `engine/core/types.h` | `u8`, `u32`, `u64` primitive types |
| `ffe::core` logging | `engine/core/logging.h` | `FFE_LOG_ERROR`, `FFE_LOG_WARN`, `FFE_LOG_INFO` macros |

Neither `miniaudio` nor `stb_vorbis` is in `vcpkg.json`. They are embedded single-header libraries following the same pattern as `stb_image` and `glad`.

The audio subsystem has no dependency on: renderer, ECS, physics, input, or scripting. The renderer dependency is one-directional: `Application::tick()` calls `audio::updateListenerFromCamera()` with camera data, but the audio subsystem itself does not include any renderer headers.
