# website/ -- System Context

## System Purpose

The website directory contains the FastFreeEngine documentation and training site, built with MkDocs and the Material theme. It serves as the primary learning resource for developers using FFE -- from getting-started guides to full API reference. The site is designed to be deployed as a static site (e.g., GitHub Pages).

A key feature of this directory is the **API extraction pipeline**: a Python script that reads `.context.md` files from the engine source tree and transforms them into MkDocs-compatible API reference pages. This keeps the website API docs in sync with the engine code automatically.

---

## Building the Site Locally

### Prerequisites

- Python 3.9+
- pip

### Setup and Serve

```bash
# From the repo root:
cd website
pip install -r requirements.txt
mkdocs serve
```

This starts a local dev server at `http://127.0.0.1:8000` with live reload.

### Build Static Site

```bash
cd website
mkdocs build
```

Output goes to `website/site/` (git-ignored).

## API Extraction Pipeline

### How It Works

1. The engine ships `.context.md` files in each subsystem directory (e.g., `engine/core/.context.md`, `engine/renderer/.context.md`).
2. The script `website/scripts/generate_api_docs.py` reads each `.context.md`.
3. For each file, it prepends MkDocs YAML front matter (title, description) and an auto-generation notice.
4. It writes the result to `website/docs/api/<subsystem>.md`.

### Running the Pipeline

```bash
# From repo root:
python website/scripts/generate_api_docs.py

# Or with a custom output directory:
python website/scripts/generate_api_docs.py --output-dir /tmp/api-preview
```

The script prints which files were processed. It returns exit code 1 if any expected `.context.md` file is missing or empty.

### Source-to-Output Mapping

| Source `.context.md` | Output Page | Title |
|----------------------|-------------|-------|
| `engine/core/.context.md` | `api/core.md` | Core |
| `engine/renderer/.context.md` | `api/renderer.md` | Renderer |
| `engine/audio/.context.md` | `api/audio.md` | Audio |
| `engine/physics/.context.md` | `api/physics.md` | Physics |
| `engine/scripting/.context.md` | `api/scripting.md` | Scripting |
| `engine/networking/.context.md` | `api/networking.md` | Networking |
| `engine/scene/.context.md` | `api/scene.md` | Scene |
| `editor/.context.md` | `api/editor.md` | Editor |

### Adding a New Subsystem

1. Create a `.context.md` file in the new subsystem directory (follow the format in CLAUDE.md Section 9).
2. Add an entry to the `SUBSYSTEM_MAP` dict in `website/scripts/generate_api_docs.py`.
3. Add the new page to the `nav` section in `website/mkdocs.yml` under API Reference.
4. Run the extraction script.

## Deployment

### GitHub Pages Workflow

The site is automatically deployed to GitHub Pages via `.github/workflows/deploy-website.yml`. The workflow triggers on:

- **Push to `main`** when files in `website/**` or any `engine/**/.context.md` change (so API docs stay in sync).
- **Manual dispatch** via the GitHub Actions UI (`workflow_dispatch`).

The workflow steps are:

1. Checkout the repo.
2. Set up Python 3.12 with pip caching.
3. Install MkDocs dependencies from `website/requirements.txt`.
4. Run the API extraction script to generate `website/docs/api/*.md` from engine `.context.md` files.
5. Build the site with `mkdocs build --strict` (strict mode fails on warnings -- broken links, missing nav entries, etc.).
6. Upload and deploy to GitHub Pages via the official `actions/upload-pages-artifact` and `actions/deploy-pages` actions.

A concurrency group (`deploy-website`) prevents parallel deployments. Unlike the CI workflow, in-progress deployments are **not** cancelled -- the queued deployment waits for the current one to finish, ensuring every push to `main` gets deployed.

### Triggering a Manual Deploy

1. Go to the repository's **Actions** tab on GitHub.
2. Select the **Deploy Website** workflow in the left sidebar.
3. Click **Run workflow** and select the `main` branch.
4. The site will rebuild and deploy within a few minutes.

This is useful when you need to redeploy without pushing a commit (e.g., after enabling GitHub Pages in repo settings for the first time).

### Testing Locally Before Deploying

Always verify the site builds cleanly before pushing:

```bash
# From the repo root:
python website/scripts/generate_api_docs.py
cd website
mkdocs build --strict
```

If `--strict` passes locally, it will pass in CI. To preview interactively:

```bash
cd website
mkdocs serve
# Open http://127.0.0.1:8000
```

## Adding New Pages

### Documentation Pages (tutorials, guides)

1. Create a `.md` file under the appropriate `website/docs/` subdirectory (e.g., `tutorials/`, `internals/`).
2. Add the page to the `nav` section in `website/mkdocs.yml`.
3. MkDocs Material features available: admonitions (`!!! note`), code blocks with copy button, tabbed content, syntax highlighting, permalink anchors.

### API Reference Pages

Do **not** create API pages by hand. Use the extraction pipeline (see above). The `.context.md` files in the engine are the source of truth.

### Learn Pages (Build Your Own Engine Track)

The `docs/learn/` directory contains the "Build Your Own Engine" learning track -- self-contained tutorials where readers build simplified versions of engine subsystems from scratch in plain C++. Each installment is independent of FFE code (no FFE headers required). All C++ code in learn pages must be **complete and compilable** -- readers should be able to copy it into a single `.cpp` file and build it with any C++20 compiler.

- `learn/index.md` -- landing page listing all installments and prerequisites
- `learn/ecs-from-scratch.md` -- first installment: building a minimal ECS

When adding new installments:

1. Create the `.md` file in `website/docs/learn/`.
2. Add a nav entry under `Learn` in `website/mkdocs.yml`.
3. Update the installment table in `learn/index.md`.
4. Ensure all code compiles standalone (test with `g++ -std=c++20`).

### Community Pages

The `docs/community/` directory contains community-facing content beyond the main `community.md` page.

- `community/showcase.md` -- game showcase listing official demos and community-submitted games

When adding new showcase entries, include: game name, brief description, key FFE features demonstrated, how to run it, and controls.

## Content Authoring Guidelines

- **Audience:** Students, indie developers, hobbyists, young people learning engineering. Write clearly. Avoid jargon without explanation.
- **Code examples:** Every concept should have a working code example. Prefer complete, copy-pasteable snippets over fragments.
- **Tone:** Friendly but technical. Not condescending. Assume the reader is smart but may be new to game engines.
- **Structure:** Each page should have a clear purpose stated in the first paragraph. Use headings, tables, and code blocks liberally.
- **Links:** Cross-reference related pages. The API reference pages link to each other via the subsystem index.
- **No hand-editing API pages:** The `website/docs/api/*.md` files (except `index.md`) are auto-generated. Edit the engine's `.context.md` files instead.

---

## What Not To Do

### Do not edit auto-generated API pages directly

```
# BAD -- this will be overwritten
Edit website/docs/api/core.md by hand

# GOOD -- edit the source and regenerate
Edit engine/core/.context.md
python website/scripts/generate_api_docs.py
```

### Do not add heavy JavaScript or custom themes

The site must load fast on old hardware and slow connections, consistent with FFE's mission. Stick to MkDocs Material's built-in features.

### Do not use external image hosting

Store images in `website/docs/assets/` (create the directory as needed). External image links break when services go down.

### Do not skip the nav entry

Every new page needs an entry in `mkdocs.yml` `nav`. Pages without nav entries are still reachable by URL but invisible to readers browsing the site.

```
website/
  mkdocs.yml              # Site configuration and navigation
  requirements.txt         # Python dependencies (mkdocs, mkdocs-material)
  .gitignore               # Ignores site/ build output

  docs/
    index.md               # Homepage
    getting-started.md      # Setup and first project guide
    community.md            # Community links and contributing
    api/
      index.md             # API reference overview (hand-authored)
      core.md              # Auto-generated from engine/core/.context.md
      renderer.md          # Auto-generated from engine/renderer/.context.md
      audio.md             # Auto-generated from engine/audio/.context.md
      physics.md           # Auto-generated from engine/physics/.context.md
      scripting.md         # Auto-generated from engine/scripting/.context.md
      networking.md        # Auto-generated from engine/networking/.context.md
      editor.md            # Auto-generated from editor/.context.md
      scene.md             # Auto-generated from engine/scene/.context.md
    tutorials/
      index.md             # Tutorial listing
      first-2d-game.md     # Your First 2D Game tutorial
      first-3d-game.md     # Your First 3D Game tutorial
      multiplayer-basics.md # Multiplayer networking tutorial
    internals/
      index.md             # Engine internals overview
    learn/
      index.md             # "Build Your Own Engine" learning track landing page
      ecs-from-scratch.md  # Tutorial: Build an ECS from Scratch (~150 lines of self-contained C++)
    community/
      showcase.md          # Game showcase: official demos + community submissions
```

## Dependencies
