# Design Note: 2D Physics / Collision System

**Status:** Proposed
**Author:** architect
**Date:** 2026-03-06
**Related:** ADR-001 (core skeleton), ADR-003 (input system), ADR-004 (Lua scripting)

---

## 1. Scope

This is **arcade physics** — collision detection with overlap queries and basic collision response. It is sufficient for platformers, top-down games, shmups, and similar 2D genres.

MVP scope:

- 2D only. No 3D physics.
- Collision detection: overlap queries between shapes.
- Collision response: generate per-frame event lists. Game scripts handle the response.
- No rigid body dynamics in MVP (no velocity integration, no angular momentum, no friction/restitution). That is a future extension.

---

## 2. Collision Shapes

Two shapes for MVP:

| Shape | Parameters | Use Case |
|-------|-----------|----------|
| **AABB** | halfWidth, halfHeight | Rectangular entities, tiles, platforms |
| **Circle** | radius (stored in halfWidth) | Projectiles, pickups, radial triggers |

Three overlap tests required:

- AABB vs AABB: min/max overlap on both axes.
- Circle vs Circle: `distance_squared < (r1 + r2)^2`. No sqrt.
- AABB vs Circle: closest point on AABB to circle center, then distance check.

All shapes are defined relative to the entity's `Transform` position (center origin). No local offset in MVP — the collider is centered on the entity. A future revision may add `offsetX`/`offsetY` if needed.

---

## 3. Data Model

New component in `engine/physics/collider2d.h`:

```cpp
namespace ffe {

enum class ColliderShape : u8 { AABB, CIRCLE };

struct Collider2D {
    ColliderShape shape;     // 1 byte
    // 1 byte padding
    u16 layer;               // which layer this entity belongs to (bitmask)
    u16 mask;                // which layers this entity collides with (bitmask)
    f32 halfWidth;           // AABB half-extent X, or circle radius
    f32 halfHeight;          // AABB half-extent Y (ignored for CIRCLE)
    bool isTrigger;          // true = overlap event only, no position correction
    // 3 bytes padding
};
// sizeof(Collider2D) == 16 bytes. Fits in one cache line alongside Transform.

} // namespace ffe
```

Layer/mask filtering rule: two entities A and B collide only if `(A.layer & B.mask) && (B.layer & A.mask)`. This is the standard bidirectional check. 16-bit masks give 16 collision layers, which is sufficient for typical 2D games (player, enemy, projectile, terrain, trigger, pickup, etc.).

## 4. Broad Phase: Spatial Hash Grid

A spatial hash grid maps each collider to one or more grid cells based on its world-space AABB. Pairs are generated by checking entities within the same cell.

**Why spatial hash over quad-tree:**

- O(1) cell lookup vs O(log n) tree traversal.
- Simpler implementation with fewer branches.
- Excellent for uniform-size entities, which is the common case in 2D games.
- Quad-trees win when entity sizes vary wildly (e.g., a tiny bullet and a huge boss). For that case, the spatial hash cell size can be tuned, or we revisit the decision later. MVP targets the common case.

**Configuration:**

- Cell size default: 128 pixels (2x a typical 64px sprite). Configurable via `PhysicsConfig`.
- Hash function: integer cell coordinates `(cx, cy)` hashed to a flat array index. Use a simple spatial hash: `(cx * 73856093) ^ (cy * 19349663)` mod bucket count. Prime multipliers reduce clustering.
- Bucket count: next power of two >= expected entity count. Default 1024 buckets for up to ~1000 entities.

**Memory:**

- The spatial hash is rebuilt every frame. All storage comes from the per-frame arena allocator. Zero heap allocation in the hot path.
- Each bucket is a small array (arena-allocated) of `EntityId` values. Overflow spills into the next contiguous arena region.

## 5. Narrow Phase

Narrow phase runs on each candidate pair from the broad phase. Layer/mask filtering is checked first (single bitwise AND, effectively free) to skip pairs early.

**AABB vs AABB:**

```
overlap = (a.minX < b.maxX) && (a.maxX > b.minX) &&
          (a.minY < b.maxY) && (a.maxY > b.minY);
```

Four comparisons per pair. Minimal branching — the compiler will typically emit conditional moves or short-circuit.

**Circle vs Circle:**

```
dx = a.x - b.x;
dy = a.y - b.y;
distSq = dx*dx + dy*dy;
radiusSum = a.r + b.r;
overlap = distSq < radiusSum * radiusSum;
```

No sqrt. Five arithmetic ops and one comparison.

**AABB vs Circle:**

```
closestX = clamp(circle.x, aabb.minX, aabb.maxX);
closestY = clamp(circle.y, aabb.minY, aabb.maxY);
dx = circle.x - closestX;
dy = circle.y - closestY;
overlap = (dx*dx + dy*dy) < circle.r * circle.r;
```

Two clamps, five arithmetic ops, one comparison. No branching in the clamp if implemented with min/max intrinsics.

## 6. Collision Events

The narrow phase produces a per-frame collision event list:

```cpp
struct CollisionEvent {
    EntityId entityA;
    EntityId entityB;
};
```

8 bytes per event. Stored in a contiguous arena-allocated array. The array is valid for the duration of the frame and reset when the arena resets.

**Deduplication:** Each pair is ordered so that `entityA < entityB`. The broad phase skips self-pairs. Duplicate pairs (from entities spanning multiple grid cells) are eliminated by sorting the pair list and removing adjacent duplicates, or by using a small bitset if entity count is bounded.

**Delivery to Lua:** After the collision system runs, the engine iterates the event list and calls the registered Lua callback for each event. This happens once per frame, not per-tick — collision detection runs at the fixed timestep but Lua delivery batches the results.

## 7. System Integration

New ECS system: `collisionSystem`, registered at priority 200 (physics band, per `system.h` conventions).

**Per-frame execution order:**

```
Priority   0:  InputUpdateSystem
Priority   5:  CopyTransformSystem
Priority 100:  Gameplay / Scripting systems (move entities)
Priority 200:  collisionSystem              <-- NEW
Priority 500:  RenderPrepareSystem
```

**collisionSystem internals (single function, no virtual calls):**

1. Get arena allocator from ECS context.
2. Build spatial hash grid from all entities with `Transform` + `Collider2D`.
3. Generate candidate pairs from overlapping grid cells.
4. Filter by layer/mask.
5. Run narrow phase on surviving pairs.
6. Write `CollisionEvent` array to ECS context (arena-allocated).
7. Deliver events to Lua callback (if registered).

The entire system is a single `void(*)(World&, float)` function pointer. No virtual dispatch. No heap allocation.

## 8. Lua API

```lua
-- Add a collider to an entity
ffe.addCollider(entityId, "aabb", halfW, halfH, layer, mask, isTrigger)
ffe.addCollider(entityId, "circle", radius, 0, layer, mask, isTrigger)

-- Register collision callback (called once per overlapping pair per frame)
ffe.setCollisionCallback(function(entityA, entityB)
    -- game logic here
end)

-- Remove collider
ffe.removeCollider(entityId)
```

Shape is passed as a string (`"aabb"` or `"circle"`) for Lua ergonomics. The binding converts to `ColliderShape` enum on the C++ side. The string comparison happens once at registration time, not per frame.

`halfHeight` is ignored for circle shapes (passed as 0 by convention).

Layer and mask default to `0xFFFF` (collide with everything) if omitted.

## 9. Tier Support

| Tier | Supported | Notes |
|------|-----------|-------|
| RETRO | Yes | Pure CPU. No GPU dependency. |
| LEGACY | Yes (default) | Budget: 1000 colliders at 60fps. |
| STANDARD | Yes | Same implementation. Could add multithreaded broad phase later. |
| MODERN | Yes | Same implementation. |

The collision system is entirely CPU-bound. It uses no GPU resources, no OpenGL calls, and no tier-specific code paths. It runs identically on all tiers.

---

## 10. Performance Budget

**Target:** 1000 active colliders, 60fps, LEGACY tier hardware (~2012 CPU).

**Spatial hash build:** 1000 entities, each inserted into 1-4 cells. ~4000 hash lookups + arena allocations. At ~10ns per operation on a 2012 CPU, this is ~40us.

**Broad phase pair generation:** With 128px cells and typical 2D game density (5-20 entities per cell), pair count is manageable. Worst case with uniform random distribution in a 1280x720 world: ~50 cells occupied, ~200 candidate pairs.

**Narrow phase:** 200 pairs x ~10ns per overlap test = ~2us.

**Total estimated cost:** < 100us per frame. Well within the 16.67ms frame budget.

**Memory:** Spatial hash + pair list + event list, all arena-allocated. Upper bound ~64KB for 1000 entities. Fits comfortably in the 2MB LEGACY arena.

**Key constraints from the constitution:**

- No heap allocation in the hot path. Arena only.
- No virtual function calls. Single function pointer system.
- No `std::function`. Lua callback stored as registry reference (integer key).
- `Collider2D` is 16-byte POD. Cache-friendly iteration via ECS view.
- `CollisionEvent` is 8-byte POD. Contiguous array, linear scan.

## 11. Future Extensions (Out of Scope for MVP)

- **Rigid body dynamics:** velocity integration, angular momentum, restitution, friction. Requires `RigidBody2D` component.
- **Position correction:** push overlapping entities apart (MTV — minimum translation vector). Requires penetration depth from narrow phase.
- **Raycasts / shape casts:** query the spatial hash along a ray. Useful for line-of-sight, hitscan weapons.
- **Collision layers as named groups** in Lua (e.g., `"player"`, `"enemy"`) mapped to bitmask positions.
- **Multithreaded broad phase** for STANDARD/MODERN tiers.
- **Collider offset** (`offsetX`, `offsetY`) for non-centered collision shapes.
