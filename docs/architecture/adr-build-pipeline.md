# ADR: Editor Build/Export Pipeline

**Status:** Proposed
**Date:** 2025-03-07
**Author:** architect
**Tier:** LEGACY minimum

## Context

The editor needs to export a standalone game that players can run without the editor installed. The engine already has the pieces: a Lua scripting runtime, JSON scene serialisation, asset loading (textures, meshes, audio), and demo executables that follow a "generic Lua game runner" pattern (e.g., `ffe_lua_demo`). The question is how to package these into a distributable game.

## Decision: Copy-and-Package Model (No Recompilation)

The export pipeline copies a pre-built runtime binary and bundles it with the project's assets, scenes, and scripts. No C++ compilation occurs at export time. This is the simplest viable approach and matches how the demo games already work.

### Export Model

1. The engine build produces a generic **game runner** binary (e.g., `ffe_runtime`) that:
   - Looks for `main.lua` adjacent to itself (or in a known relative path)
   - Executes that script, which calls `ffe.loadScene()` to start the game
2. The editor's export step:
   - Copies `ffe_runtime` into the output directory, renamed to the project name
   - Copies all referenced assets, scenes, and scripts into structured subdirectories
   - Generates a `main.lua` entry point that loads the user's chosen entry scene
3. The exported game is fully self-contained. No editor, no SDK, no toolchain required.

### Build Configuration

The editor exposes these export settings:

| Setting | Type | Required | Notes |
|---------|------|----------|-------|
| Project name | string | yes | Used as binary name and output folder name |
| Entry scene | path | yes | Which `.json` scene file to load first |
| Asset directories | list of paths | yes | Directories to include (textures, meshes, audio) |
| Script directories | list of paths | yes | Lua script directories to bundle |
| Target platform | enum | yes | `linux-x86_64` initially; `windows-x86_64`, `macos-arm64` future |

For cross-platform export, the editor needs access to a pre-built `ffe_runtime` for each target platform. Initially only the host platform is supported. Cross-platform binaries can be supplied via CI artifacts in future phases.

### Exported Directory Structure

```
MyGame/
  game                 # runtime binary (renamed from ffe_runtime)
  assets/
    textures/          # .png, .jpg
    meshes/            # .glb
    audio/             # .wav, .ogg
  scenes/
    level1.json        # serialised scene files
    level2.json
  scripts/
    player.lua         # game logic scripts
    enemy.lua
  main.lua             # generated entry point
```

The generated `main.lua` is minimal:

```lua
-- Auto-generated by FFE Editor
ffe.loadScene("scenes/level1.json")
```

### Runtime Loader

**What exists today:**
- `ffe.loadScene(name)` in Lua triggers scene transitions via the scripting layer
- Demo games use a C++ `main()` that initialises the engine, registers a Lua file, and runs the app loop
- The Lua sandbox already supports loading scripts and calling scene lifecycle callbacks (`onInit`, `onUpdate`, `onDraw`)

**What needs to be added:**
- A **generic runtime main** (`ffe_runtime`) that does not hardcode a specific game. It initialises the engine, finds `main.lua` relative to the binary, and runs it.
- A **scene-from-file loader** that deserialises a JSON scene file into ECS entities. The existing scene serialisation format (JSON, entity components) provides the schema. The loader must reconstruct entities with their components (Transform, Sprite, Mesh, Collider2D, etc.) from the serialised data.
- The Lua `ffe.loadScene()` binding must support loading serialised scene files (JSON), not just switching to a named Lua callback table.

These are incremental additions, not architectural changes.

### Asset Path Resolution

The runtime resolves all asset paths relative to the binary location. This keeps exported games relocatable — the user can move the folder anywhere and it still works. The engine's existing `canonicalizePath()` in `engine/core/platform.h` handles path normalisation. The runtime must determine its own binary directory at startup and use that as the asset root.

### Security

Exported games load scene files and Lua scripts, both of which are external input. Existing protections apply:

- **Scene deserialisation:** Entity count limits, NaN/Inf rejection on numeric fields, string length limits (already in scene serialiser)
- **Lua sandbox:** 1M instruction budget, no filesystem/network/OS access (already enforced)
- **Path traversal:** `isPathSafe()` prevents asset loads from escaping the game directory (already in scripting layer)
- **Asset loading:** Texture/mesh/audio loaders already validate file headers and reject malformed data

No new attack surface is introduced. The export pipeline copies files; it does not generate code or compile anything.

### Tier Support

LEGACY (OpenGL 3.3, 1GB VRAM) minimum. The export pipeline is a file-copying tool with no GPU requirements. The exported game inherits whatever tier the runtime binary was built for, which defaults to LEGACY.

## Alternatives Considered

**Recompilation at export time** — Compile a custom binary per game with baked-in assets. Rejected: requires a C++ toolchain on the user's machine, massively increases export time, and gains nothing for Lua-scripted games.

**Single-file packaging** (zip/archive with embedded binary) — Bundle everything into one executable. Deferred: adds complexity (virtual filesystem, self-extraction) with minimal user benefit at this stage. Can be added later as an optional export format.

## Implementation Sequence

1. Build the generic `ffe_runtime` binary (new CMake target, reuses existing engine)
2. Implement JSON scene file loader (deserialise entities + components)
3. Extend `ffe.loadScene()` to accept scene file paths
4. Build the editor export UI and file-copy pipeline
5. Test with an exported version of an existing demo game
